HTML-First Migration Plan

Migrating KubeDojo orchestrator artifacts from Markdown to HTML, per Thariq @trq212's "The Unreasonable Effectiveness of HTML" (2026-05-08, Anthropic Claude Code team). The goal is artifacts you actually read in 30 seconds, not 200-line markdown walls you skim.
Adopted: 2026-05-09 Updated: 2026-05-09 — consumer-direction lens revision Status: Phases 1+5(rev) done, Phase 2 next, Phases 3-4(rev) on-demand Design: _design-system.html Reference: archived article
✓
Decision locked 2026-05-09, updated 2026-05-09 post ab-discuss: HTML is the default output format for orchestrator-generated artifacts that benefit from visualization, interactivity, or shareability. Markdown stays only for files the briefing API parses or that are grepped daily by both humans and tooling. Cross-family review now uses lightpanda-rendered markdown, so .notes.md sidecars are optional polish, not load-bearing.

Consumer-direction lens (refinement, 2026-05-09)

Format is picked by who reads the artifact, not just by artifact class. Three directions, three defaults:

Direction Primary consumer Default format Examples
AI → Human User reads it HTML Batch reports, PR explainers, bug autopsies, summary audits
Human → AI User types it (agent reads) Markdown CLAUDE.md, rules, memory files, decision cards
AI → AI Next agent reads first Markdown Dispatch briefs (paste-to-agent), session handoffs (Read by next session)

This flips Dispatch briefs and Session handoffs to Markdown by default. HTML is reserved for explicit user-facing variants of those artifact classes. Source of truth: memory/feedback_html_over_markdown_for_artifacts.md.

"I find it difficult to read a markdown file of more than a hundred lines. I want richer visualizations, color and diagrams and I want to be able to share them easily. I've started preferring HTML as an output format instead of Markdown and increasingly see this being used by others on the Claude Code team, this is why."
Thariq @trq212, Anthropic Claude Code team, 2026-05-08

Phase detail

PHASE 1 Batch reports DONE

First proof of concept. Ship a real artifact, validate the workflow on a single instance, then scale.
What shipped audit/388-cka-part1-batch/index.html with a 5-module status grid, KPI cards, color-coded verdicts, vertical timeline, and two-column lessons block.
What it replaced A flat markdown table that nobody would have opened. The user reaction, "now I have much better overview what is happening", validated the direction.

PHASE 2 PR review explainers NEXT

Convert non-APPROVE review output into an HTML diff explainer: color-coded findings, spotlighted files, reviewer concern summary, and exact fix-up checklist.
Where it slots in Generated after cross-family review for PRs that receive NEEDS CHANGES or APPROVE_WITH_NITS. The artifact can be linked from the PR comment or attached as a CI/local artifact.
Why promoted The CKA 1.3 fix-up cycle proved the value. The real bug was a wrong Helm repository URL buried inside a long markdown review. HTML makes the actionable defect pop.
Trigger: Next #388 sweep PR or migration PR with a non-APPROVE verdict.

PHASE 3 Bug autopsies + summary audits QUEUED

Lower-frequency, higher-stake artifacts. Autopsies need cause/effect flow, prevention panels, and proof links. Summary audits need side-by-side convergence tables and click-to-expand evidence.
Currently docs/bug-autopsies/INDEX.md plus per-category markdown files. audit/*/REPORT.md for periodic audits.
After migration Per-incident .html alongside the markdown index, which stays as the searchable list. Audit summaries become navigable HTML.
Trigger: Next incident-class bug fix or scheduled summary audit.

PHASE 4 Dispatch briefs REVISED

Reclassified AI→AI (Markdown by default). Dispatch briefs are pasted directly into agent prompts — the primary consumer is the dispatched agent, not the user. Markdown is the correct format. HTML is reserved for explicit user-facing pre-batch dashboards where the user wants to review gate status visually before kicking off a sweep.
When HTML still applies Only when the brief is genuinely user-facing: a dashboard a human reviews and approves before triggering a large sweep. In that case, the AI→Human rule applies and HTML wins.
Default practice Write briefs as docs/briefs/*.md. The native markdown /artifacts renderer handles display. No lightpanda conversion needed.
Trigger: Phase 4 reclassified Markdown 2026-05-09 per consumer-direction lens; HTML version reserved for explicit user-facing brief dashboards.

PHASE 5 Session handoffs REVISED

Reclassified AI→AI (Markdown by default). The primary consumer of a session handoff is the next-session agent, which reads it via the Read tool before starting work. That is an AI→AI flow — Markdown is the correct format. HTML is reserved for genuinely user-facing handoffs such as rare demo-ready end-of-week summaries the user will share externally.
Prerequisite DONE in PR #1021, commit 70130274: session-state directory accepts both .md and .html handoffs. Default path is now docs/session-state/*.md.
When HTML still applies Only for explicit user-facing handoffs: end-of-week summaries, external demos, or any handoff the user intends to share beyond the next agent session. Otherwise Markdown.
Trigger: Phase 5 reclassified Markdown 2026-05-09 per consumer-direction lens; HTML reserved for explicit user-facing handoffs (rare). Prereq still DONE in PR #1021.

Cross-family HTML review via lightpanda

ab-discuss outcome

Gemini's round-1 objection to HTML PRs was diff noise: raw HTML can be too noisy and token-heavy for cross-family review. That objection is resolved because /opt/homebrew/bin/lightpanda renders HTML to clean markdown for agent review.

All three agents have a working path: Claude can inline-read the HTML or rendered markdown, Codex can use the Browser plugin for visual inspection, and Gemini can use lightpanda plus web_fetch. Reviewers should read the semantic markdown, not the raw HTML, unless they are auditing markup itself.

Canonical command: lightpanda fetch --dump markdown http://127.0.0.1:8989/<path>. Reference memory: ~/.claude/projects/-Users-krisztiankoos-projects-kubedojo/memory/reference_lightpanda_html_rendering.md.

The .notes.md sidecar invariant from the discussion becomes optional polish, not load-bearing. Use sidecars only when grep/provenance requires them.

Anti-patterns to avoid

From the article itself:
  • Don't build a /html skill yet. Build muscle by prompting for HTML naturally; codify only after 5-10 real artifacts reveal patterns worth a skill.
  • Don't migrate everything at once. Phase 1 through 5 are sequenced by frequency, value, and review readiness.
  • Don't migrate things that are grepped or parsed. The briefing API parses STATUS.md heading-by-heading; rules and memory files are text-grepped continuously by cold-start agents.
From our own constraints:
  • Don't add a .notes.md sidecar by default. Use only when grep/provenance demands it, such as handoff search across N sessions. Cross-family review uses lightpanda, not the sidecar.
  • HTML diffs are noisy, but review should not read raw HTML first. Serve locally and render through lightpanda markdown before asking Gemini or another cross-family reviewer to judge semantics.
  • 2-4x generation time. Budget for it. The CKA Part 1 batch report took longer than a markdown table but was actually consumed.
  • Don't inline-write content modules as HTML. Module content under src/content/docs/**/*.md is curriculum rendered by Astro. This migration is about orchestrator artifacts only.

How to apply (in conversations)

Default move DO

  • Use _design-system.html as the canonical source for CSS variables and components.
  • Make an HTML file when generating any artifact in the migration matrix marked HTML and the content warrants visual structure.
  • Link to it from STATUS.md, cross-thread notes, PRs, or Slack. HTML is shareable; that is the point.
  • Open it in the browser after generating with open path/to/file.html.
  • Render it with lightpanda before asking for cross-family review.

Default avoid DON'T

  • Don't skill-ify yet. No /html-report, no /html-handoff. Build patterns by hand first.
  • Don't break grep paths. If a tool or human greps for it, keep it Markdown or add a deliberate sidecar.
  • Don't generate HTML for one-line answers. HTML overhead pays off only above roughly 50 lines or multiple visualizable dimensions.
  • Don't commit HTML diffs blindly for high-frequency artifacts. Decide per class: commit the latest, or generate and link.
