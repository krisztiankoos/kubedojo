## Lessons added this session

1. **Concurrent codex CLI dispatches fail silently** — running 3 `dispatch.py codex` jobs in parallel via background bash: 1 succeeded, 2 silently died with empty stdout/stderr. Sequential dispatches all succeeded. Probably codex CLI doesn't tolerate concurrent OAuth sessions. **Rule: dispatch codex sequentially, not in parallel.**
2. **`quality_post_review.py` silent-crash bug (#396)** — backgrounding the script via `... &` or `nohup` reliably killed it after the first iteration with no traceback. Caused recurring batch outages because the first module's banner-clear was uncommitted, then the next batch run hit `dirty_pre`. Fixed in 483a963c with line-buffered stdout/stderr + SIGHUP ignore + try/except wrapper. Bug filed at #396.
3. **`cd` inside a bash heredoc shifts the session cwd** — and that breaks subsequent relative-path commands (`scripts/ai_agent_bridge/...` started failing with "No such file or directory"). Always use absolute paths, even inside heredocs.
4. **Cross-family review catches real bugs that build-clean misses** — gemini flagged 3 must-fixes on the codex-authored route changes (security cert order inverted, 3 redundant legacy sections, 25 relative links violating the convention). Build was clean, but route quality was not. Reinforced lesson #2 from v3.
5. **My parallel file edits race the batch's merge step** — every file write while batch is in the merge phase can dirty the tree and trigger `merge: primary has uncommitted changes inside merge lock — foreign edit`. The recurring gitops 3.1 failures (270s wall, 4 retries) were caused by my bridge-page writes happening during batch's gitops merge attempts. **Rule: when actively editing curriculum or scripts, batch the writes into one short window then commit, rather than editing while batch is mid-flight.**
6. **Codex link-guessing is unreliable** — codex authored the bridge pages with several wrong toolkit paths (`/platform/toolkits/gitops/argocd/` should be `/platform/toolkits/cicd-delivery/gitops-deployments/module-2.1-argocd/`). Always grep for actual `slug:` values when verifying codex link output. Build doesn't fail (Starlight markdown links aren't validated), but users 404.
7. **Drain output buffer can survive a 30-minute task even when status looked dead** — a backgrounded drain that appeared dead after 1/13 actually completed all 13 modules over 36 minutes. Don't write off background tasks too quickly; check the output file even if `pgrep` is empty. (This is a workaround for #396; once the fix is verified in production, the silent-progress concern goes away.)
