TL;DR. openclaw/clawpatch is a 4-day-old (created 2026-05-15, v0.2.0 shipped 2026-05-17, 377 stars / 168 commits at time of read) CLI that slices a repo into semantic feature units, runs an agent-driven review per slice, persists findings to .clawpatch/, and exposes a separate fix --finding <id> command that produces patches without committing or pushing. Provider-agnostic via codex / acpx (Claude/Gemini bridge) / grok / opencode. The interesting kubedojo angle is not markdown-curriculum review (clawpatch's slicers don't know Astro/Starlight content) — it's reviewing our pipeline + verifier code (~150 Python files in scripts/) and converting findings to suggested regression tests, exactly the deterministic gate the grok audit just recommended. Eval target proposed below: have clawpatch review scripts/quality/ and check if it independently surfaces the image-binary preflight gap that the grok audit identified. If yes → strong adopt signal. If no → drop it; we don't get value from another LLM-reviewer overlay on what gemini-pro / codex / grok already cover.

What a finding looks like (per docs/findings.md)

Each finding record contains: feature ID, title, category, severity, confidence, triage status, evidence, reasoning, reproduction notes, recommendation, test coverage analysis, suggested regression test, minimum fix scope, status, linked patch attempts, triage/revalidation history. Status values: open / false-positive / fixed / wont-fix / uncertain.

The suggested regression test field is the standout — that's exactly the artifact the grok-audit action item 2 (deterministic image-binary preflight) needs. Most LLM reviewers produce prose findings without a paired test scaffolding; clawpatch's schema treats the test as a first-class output.

3. Where clawpatch could plug into kubedojo's workflow (5 candidate slots)

  1. Pipeline-code review (HIGHEST leverage). Run clawpatch review --provider codex against scripts/quality/ after any change to dispatch_388_pilot.py, verify_module.py, or dispatch_smart.py. Persistent findings DB means cross-session continuity — we currently re-discover the same code smells session after session because code-review skill output evaporates with the conversation.
  2. Verifier-spec generation. The grok-audit action item 2 asks for a deterministic image-binary preflight. Use clawpatch fix --finding X --provider acpx --model claude-opus-4-7 to draft the implementation, taking advantage of the suggested regression test field as the contract spec. This is exactly the lab-hardening focus the user signaled.
  3. Cross-family reviewer rotation per feature. clawpatch review --provider codex for one slice, --provider acpx --model claude-opus-4-7 for another, --provider grok for a third. Matches our cross-family rule per docs/review-protocol.md. Materially better than today: today the cross-family choice is per-PR, not per-feature-slice within a PR.
  4. Codex-budget-friendlier triage. Today, mid-session reviews burn codex weekly because codex is the default reviewer for non-trivial Python. clawpatch's persistent finding state means a codex review's output survives the conversation, so we don't need to re-run when context drops. Indirect budget win.
  5. Regression-test scaffolding. Every finding ships a suggested regression test field. If the field reliably produces runnable pytest fixtures, that's a meaningful uplift over what our LLM reviewers produce today (prose-only). This is the kind of structural contract advantage that's hard to replicate via prompt-engineering alone.

Where it does NOT fit

  • Curriculum content review — clawpatch's slicer is code-only. The module-quality-reviewer skill, review-module / review-part commands, and the gemini/grok cross-family reviewer cascade cover this surface and clawpatch adds nothing.
  • The #388 batch pipeline gate — that's already a fully-instrumented codex-writes / grok-reviews / verifier-gates flow with 27 reviews under its belt. Inserting clawpatch in the middle doesn't make it faster or cheaper.
  • Auto-merge — clawpatch deliberately does not commit/push/open PRs. So it doesn't replace any existing automation.

Acid-test scorecard (apply to clawpatch's findings, regardless of how many it produces):

  • Does it surface "exec binary not in pod image" as a missing check? Strong adopt
  • Does it surface stale-PR check, density-gate edge cases, or other real-but-known issues? Weak adopt
  • Does it produce only nit-class findings (naming, docstring style, type hint coverage)? Drop — we don't need another nit linter
  • Does it hallucinate findings against code that's clearly correct? Drop — false positives are worse than no review

5. Risks specific to kubedojo (different from learn-ukrainian)

  1. Existing hooks interaction. Kubedojo has 6 PreToolUse hooks at .claude/settings.json, including block-orchestrator-content-edits.sh, block-branch-create-in-primary.sh, block-direct-commit-on-main.sh. clawpatch runs as its own subprocess outside Claude Code, so these hooks don't gate it. That's actually good for safety (clawpatch can't accidentally trigger them) but means clawpatch's fix output is a worktree we must inspect manually before committing. Same gate posture as today.
  2. Codex budget collision. Codex has been on a 5× weekly cut since 2026-05-17 (per memory/project_budget_and_delegation.md). Adding clawpatch reviews as another codex consumer needs the budget tree: each clawpatch review --jobs N spawns N concurrent codex calls. Cap at --jobs 1 for the eval; respect the existing 2-Codex-in-flight ceiling for any production use.
  3. pnpm dependency. Kubedojo uses npm (not pnpm) for the Astro frontend. pnpm 10.28 is installed system-wide, so this isn't blocking, but it adds a tool surface. Mitigation: npx clawpatch@<version> per run avoids any global install.
  4. .clawpatch/ directory churn. Default location is repo root. Findings DB is regenerable, but the config + project.json baseline should be committed. Add .clawpatch/findings/, .clawpatch/patches/, .clawpatch/runs/ to .gitignore and commit .clawpatch/config.json + .clawpatch/project.json only — if we adopt.
  5. API churn risk. 4 days old, 168 commits, just shipped v0.2.0. Expect breakage every few weeks. Pin exact version in any wrapper script.
  6. No Monitor API integration. kubedojo's local API at :8768 wouldn't surface clawpatch findings. If clawpatch becomes important, we'd need a /api/clawpatch/findings bridge that reads .clawpatch/findings/*.json. Cost: small (file-system poll). Defer until adoption decision.
