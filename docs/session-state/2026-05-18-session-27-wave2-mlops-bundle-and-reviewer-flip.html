#1323 — hook audit detail

Bug 1: CWD class (false-negative-allow direction, same as #1321)

Both merge hooks fell back to cd "$CWD" && gh pr view when no explicit PR number was on the command (auto-detect from current branch). $CWD is the harness-reported cwd — the primary tree on main. When the orchestrator runs cd .worktrees/X && gh pr merge --squash from primary, the fallback queried the wrong tree, got empty PR_JSON, and exited 0 (fail open). The quality gate was silently bypassed for any worktree-side merge without an explicit PR ref. PR #1321 fixed the false-positive-deny direction of the same bug class on the commit hook; this PR fixes the false-negative-allow direction on the merge hooks.

Fix: new helper .claude/hooks/_lib_resolve_cwd.py walks cd <path> segments and returns the effective dir; both hooks use it before invoking gh pr view.

Bug 2: latent PR_REF parser bug (hooks failed open on EVERY explicit-PR-ref merge)

This one was discovered during the audit. The existing inline parser set found = True on the gh token, then on the NEXT iteration consumed the literal pr token and printed it as PR_REF. Verified by direct test:

$ python3 -c '<old parser>' "gh pr merge 1234 --squash"
pr
$ python3 -c '<new parser>' "gh pr merge 1234 --squash"
1234

In production gh pr view pr 404'd → PR_JSON empty → hook exited 0. Both the content-merge (Learner check) and bugfix-merge (regression test) gates have been silently bypassed for every explicit-PR-ref merge since the hooks were originally written. The existing test suite missed this because all tests use KUBEDOJO_HOOK_GH_JSON to bypass gh entirely.

Fix: new helper .claude/hooks/_lib_parse_pr_ref.py skips past the gh pr merge triple before consuming the first non-flag token as PR_REF. Edge cases verified: positional ref, flag-then-ref (gh pr merge --squash 1234), URL ref, cd X && prefix, sudo gh ..., GH_TOKEN= gh ..., empty ref.
