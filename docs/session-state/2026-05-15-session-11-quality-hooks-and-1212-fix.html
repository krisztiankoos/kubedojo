Empirical findings â€” Pattern B root cause established

Pattern B = unauthorized_prose_change in inject diff_issues. Earlier session theory was "codex misbehaves randomly." Diagnostic on 1.7-hubble (with the _verify_diff_is_additive truncation widened from 140 to 800 chars) showed the actual mechanism:

  1. Module 1.7-hubble was already backfilled in a prior run â€” its current body has 22 [phrase](https://...) citation wraps.
  2. The session-10 watchdog ran research again, generating a FRESH seed (research is stateless about existing module body citations).
  3. The fresh seed proposed a new claim citing the phrase "focused metrics set enabled" â€” but that phrase is INSIDE the existing wrap [install Cilium with Hubble, Relay, UI, and a focused metrics set enabled](https://.../values.yaml).
  4. Inject's apply step produced a body where the new [focused metrics set enabled](url) wrap clobbered the outer existing wrap.
  5. The verifier's _INLINE_LINK_RE unwrap doesn't handle nested/overlapping brackets correctly. After unwrap, the modified text has different prose than the original â†’ flagged as unauthorized_prose_change.
  6. On the second retry, codex produced a different inject plan that happened to avoid the overlapping phrase, and the module backfilled cleanly.

Mechanism is therefore "research/inject blind to existing citations + non-deterministic LLM phrase selection." Retry can succeed by luck. The deterministic fix would be either:

  • Pass the current module body (with its wraps) to research so it avoids proposing claims that overlap with existing wraps; OR
  • Have inject's apply_inject_plan detect that a proposed wrap is INSIDE an existing wrap and skip it (defer the claim).

3.5-cluster-api is an outlier â€” 0 existing wraps means the overlap mechanism doesn't apply, yet it consistently fails with the same error class.
