Decisions

Grok writer permanently DQ'd from #388 — by deterministic verifier gate

Closes the H6/H4/H5b hypothesis ladder. Both N=2 cells (reasoning=ON xhigh and reasoning=OFF) produced < 5,000 body words on the full production prompt. The cap source is the model's natural convergence ceiling on long structured prompts (~3,800–5,500w on this class), not the 8,192 max_output_tokens budget being shared with thinking. Reasoning=xhigh produced cleaner structure and avoided the deterministic-fact hallucination that non-reasoning produced (which fabricated verifier metadata "Word count: 5,847..." while actual was 4,942) — so xhigh is the right kubedojo standard if grok is ever reconsidered, but no current production lane needs it. Test A (source-patch max_output_tokens=30000) deprioritized: even unlocking the cap wouldn't push grok past the 5,000 gate.

1-line orchestrator-inline fixes are appropriate

Per .claude/rules/decision-card.md: "One-line fixes (typos, dead URLs, redundant code lines — orchestrator fixes inline)" is an explicit exception to feedback_dispatch_codex_for_code_changes's "claude orchestrates ONLY". Hygiene

Stale local_api.py PID origin SOLVED (session-14 carryover)

User reported the dashboard UI was returning empty data. Root cause: two local_api.py processes bound to port 8768. PID 1822 was the legitimate one (started by ./start-claude.sh, cwd = /Users/krisztiankoos/projects/kubedojo, bound to *:8768). PID 47962 was an orphan with cwd = /Users/krisztiankoos/projects/kubedojo/.worktrees/fix-reviewer-reassignment (a worktree removed 10 min earlier in the same session). Its repo_root resolved to the dead directory, docs_root.exists() returned False, so build_quality_scores() returned count: 0, exists: false. Because PID 47962 was bound to 127.0.0.1:8768 specifically (more specific than wildcard), all browser localhost requests went to it. Killed both per user direction; user restarted via ./services start. Worth adding a startup guard that refuses to bind 8768 if another process already binds it.
