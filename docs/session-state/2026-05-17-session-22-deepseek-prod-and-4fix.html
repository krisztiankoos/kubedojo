Grok false-positive autopsy

New failure mode: grok-4 over-extrapolates from truncated diff hunks.

PR #1287 grok-4 review returned NEEDS_CHANGES with this finding (verbatim):

[high] scripts/quality/pipeline.py:1025 — cmd_cleanup_banners iterates modules
and calls _cleanup_banner_for_module with no state.state_lease acquisition or
release visible in the diff. The lease must still be taken then released
before the helper call.

Actual code (line 1020 of the branch's pipeline.py, just OUTSIDE grok's hunk window @@ -1000,20 +1025,14 @@):

with state.state_lease(slug, timeout=5) as lease:
    st = lease.load()
    if st is None or st["stage"] != "COMMITTED":
        continue
    q = st.get("queue")
    if not q or q.get("completed_at") is not None:
        continue
    module_state = dict(st)           # ← last line inside `with` block

print(f"Cleaning stranded banner for {slug}...")
if _cleanup_banner_for_module(primary, slug, module_state):   # ← OUTSIDE `with`

The lease IS acquired and IS released before the helper call. The signal is the indentation drop from 3 levels (inside with) to 2 levels (outside). Grok-4 missed it because the with line was NOT in the diff hunk's context window — the hunk header starts at line 1000 of the original file, but the with statement was at line 994 (unchanged, outside the +3 context lines git diff shows). The test test_cmd_cleanup_banners_releases_state_lease_before_cleanup explicitly asserts in_lease["open"] is False when the cleanup helper is called — which would fail if grok's reading were correct.

Tiebreak: claude-sonnet-4-6 dispatched with the actual function body (lines 986–1041) pasted inline → VERDICT: GROK_WRONG, confidence high, 29s elapsed. Same reasoning.

Generalization: git diff -U3 (gh's default) shows 3 context lines around each hunk. Anchor statements that establish scope (with-blocks, if-blocks, function defs) can fall OUTSIDE that window when the actual edit is several lines below the anchor. Reviewers reading only the diff can mis-attribute "no anchor visible in the hunk" → "anchor doesn't exist." Affects grok-4; likely affects all LLMs reviewing diffs without file-fetch fallback.

Memory note added: feedback_grok_truncated_diff_context.md (this session). Memory + handoff are the durable record; no code change needed — orchestrators should ALWAYS do a quick "is the cited anchor actually missing or just outside the hunk?" sanity check on grok NEEDS_CHANGES before bouncing back a fixup.
