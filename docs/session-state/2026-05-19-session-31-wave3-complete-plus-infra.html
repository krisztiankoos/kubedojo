Pattern observations from this run

Stale-base merge handling for parallel codex chains

Three of the four content PRs (T2-15, T2-16, T2-22) were dispatched while a prior PR was still under review or pre-merge. Two of them (T2-16, T2-22) needed a git fetch origin main && git merge origin/main in the worktree before the GitHub PR was mergeable. T2-16 had a true conflict in index.md (both rows needed); T2-22 was a clean merge (different directory). Pattern: always merge main into the feat branch immediately before requesting the PR merge if any other content PR landed since the branch was spawned. Adding this to the standard workflow saves one round-trip per PR.

Squash-merge vs merge-commit on stale-base PRs

When a feat branch contains a Merge remote-tracking branch 'origin/main' commit (to resolve stale base), gh pr merge --squash can fail with "Pull Request is not mergeable" because GitHub's squash strategy doesn't reliably handle the merge commit. Workaround: use gh pr merge --merge instead. Cost: history shows a merge commit; benefit: it lands. Confirmed on PRs #1345 and #1347.

Dispatch output truncation pitfall

I piped dispatch_smart.py review stdout through tail -N in 3 background calls. The N-line cap discarded the full review body — only the trailing verdict survived. The full response (9018 chars in one case) sat in logs/smart_dispatch.jsonl as response_chars but the actual text was not preserved anywhere. For background review dispatches: do not pipe to tail — let the full stdout land in the task output file. The verdict on its own line is sufficient signal; trying to "minimize captured output" loses the actionable detail.

Rate-limit-bug ate one review (the irony)

The very first review dispatch (claude-sonnet-4-6 on PR #1340) classified its own output as RateLimitedError because the review text discussed the cascade order "claude → gemini → qwen" and dispatch_smart's adapter regex matched the phrase "rate limit" elsewhere in the response body. Recovered the verdict (APPROVE_WITH_NITS) from the stderr_excerpt but lost the line-by-line findings. Dispatched a gemini review as substitute; gemini cleared with APPROVE. PR #1341 fixed the bug; subsequent claude reviews on PRs #1344-#1347 worked correctly.

Auto-merge scheduling for CI gated PRs

The repo's CI checks (Analyze x3 + Incident dedup gate) take ~2-5 min per PR. gh pr merge --admin --merge failed with GraphQL: Pull Request is not mergeable when checks were pending. gh pr merge --merge --auto schedules the merge once checks pass and returns immediately. Combined with a Monitor watching the PR state, the orchestrator never blocks on CI. Used this pattern for #1346 and #1347.
