#1348 — dispatch log truncation fix

The hole

Session 31 lost the line-by-line findings of 3 claude-sonnet review dispatches because logs/smart_dispatch.jsonl only stored response_chars (count), and the body lived solely on stdout via print(response). Callers that piped through tail -N kept the trailing verdict but discarded the actionable findings. The original code path:

append_log({..., "response_chars": len(response), ...})
print("=" * 70)
print(f"OK: {ok}  |  resp_chars: {len(response)}")
print("=" * 70)
if response:
    print(response)   # ← only place the body lands; tail truncates this
The fix (2 commits)

Commit 1 (initial): add persist_response() that writes logs/dispatch_responses/<task_id>.txt (and <task_id>.stderr.txt if stderr_excerpt is non-empty). Add response_path to the JSONL row. Print the path before the body on stdout so even truncated callers see it. Also init previous_search/previous_dispatched to None before the try block (cheap proactive fix for a pyright warning while touching the file).

Commit 2 (fixup per gemini review):

  • Anchor LOG_PATH and RESPONSE_DIR to PRIMARY_REPO. The original placement resolved both against REPO = Path(__file__).resolve().parents[1], which becomes the .worktrees/X path when dispatch_smart runs inside a worktree. That silently fragmented audit trails per-worktree (the JSONL row was landing in .worktrees/X/logs/ for every worktree-cwd dispatch). Both now anchor to PRIMARY_REPO regardless of dispatch cwd. rel_response_path uses relative_to(PRIMARY_REPO) consistently to avoid a ValueError after the anchoring change.
  • Explicit utf-8 encoding on both write_text calls. Path.write_text() default picks up locale encoding (cp1252 on Windows); not the platform we ship on, but cheap defensive code.
  • Scope creep accepted: LOG_PATH was pre-existing-fragile (same root cause); fixing both in one PR is cleaner than splitting since the gemini review covered the same area. Documented in the commit message.
Verification + the meta-irony

Three live smoke tests on the worktree binary:

  1. Basic PONG dispatch: response file written, JSONL row has response_path, contents match response_chars.
  2. Pipe-truncation recovery: dispatch piped through head -3 → stdout body discarded → recovered via jq -r 'select(.task_id == "smoke-truncate-test") | .response_path' logs/smart_dispatch.jsonl | xargs cat.
  3. From worktree: dispatch ran inside the worktree → response landed in PRIMARY's logs/dispatch_responses/ as designed (post-fixup).

Meta-irony: the gemini round-1 review of this very PR was tail-truncated by the background dispatch wrapper (2>&1 | tail -5 captured ~750 of 1437 response chars). I had to re-dispatch the same review from inside the worktree (which had the fix applied) to recover the full body — including the verdict and findings A–C. So the bug bit me in real time, and the persistence file recovered the situation. Confirms the bug was real and the fix works.

8 occurrences fixed (reviewer flagged 4, I caught 4 more via grep):
