## Чому цей модуль важливий

У 2018 році велика роздрібна мережа зі списку Fortune 500 поставила 2,3 мільйона доларів на Apache Mesos як свою платформу для контейнерів. Вони найняли команду з 8 спеціалістів з Mesos, побудували власні фреймворки та розгорнули понад 200 сервісів. До 2020 року Mesos фактично вмер — фонд Apache перевів його в архів («attic»). Усю платформу довелося перебудовувати на Kubernetes. Два роки роботи — на смітник. А досвід інженерів у Mesos? Знецінився на ринку праці за одну ніч.

У технологіях знати, чого **НЕ** варто вчитися, так само важливо, як і знати, що вчити. Цей модуль — це екскурсія цвинтарем технологій. Не для того, щоб посміятися з померлих, а щоб допомогти вам розпізнати попереджувальні знаки й ніколи не ставити свою кар'єру на наступний «Mesos».

## Цвинтар оркестрації контейнерів

### Docker Swarm

**Чим він був**: Власне рішення для оркестрації від компанії Docker.

**Статус**: Фактично застарів. Docker Desktop видалив режим Swarm у 2022 році.

```
Хронологія:
2015: Запуск Docker Swarm як конкурента K8s
2017: Docker додає підтримку K8s (визнання поразки)
2019: Docker Enterprise продано компанії Mirantis
2020: Mirantis оголошує графік виведення Swarm з експлуатації
2022: Swarm видалено з Docker Desktop
```

**Чому він помер**:
- Обмежений набір функцій порівняно з K8s
- Залежність від одного вендора (Docker Inc.) проти K8s, керованого спільнотою
- Не зміг наздогнати темпи зростання екосистеми K8s
- Корпоративні клієнти вимагали саме K8s

**Не вчіть**: Концепції Swarm, мережі Swarm, визначення сервісів Swarm.

**Виняток**: Docker Compose все ще корисний для локальної розробки (але не для оркестрації в продакшні).

---

### Apache Mesos + Marathon

**Чим він був**: Дворівневий планувальник ресурсів (Mesos) з оркестрацією контейнерів (Marathon).

**Статус**: Marathon покинутий. Mesos у режимі підтримки.

```
Хронологія:
2009: Створення Mesos в Каліфорнійському університеті в Берклі
2013: Запуск Marathon для контейнерів
2016: Пік впровадження (Twitter, Airbnb, Apple)
2020: Twitter оголошує про міграцію на K8s
2021: Marathon оголошено таким, що не підтримується
2022: Використання Mesos для нових проєктів фактично нульове
```

**Чому він помер**:
- Складність дворівневого планування
- Marathon ніколи не зрівнявся за функціями з K8s
- Екосистема так і не сформувалася
- Ключові користувачі (Twitter) перейшли на K8s

**Не вчіть**: Архітектуру Mesos, конфігурації Marathon, фреймворки Mesosphere/D2iQ.

---

### Cloud Foundry (для оркестрації)

**Чим він був**: Platform-as-a-Service (PaaS), що з'явився раніше за K8s.

**Статус**: Змінив курс на роботу поверх K8s. Оригінальний підхід застарів.

```
Хронологія:
2011: Запуск Cloud Foundry компанією VMware
2015: Створення CF Foundation
2019: CF починає працювати на K8s (KubeCF)
2022: CF повністю переходить на архітектуру на базі K8s
```

**Чому вони змінили курс**:
- K8s став стандартом інфраструктури
- Не зміг конкурувати з екосистемою K8s
- Принцип "якщо не можеш перемогти — очоль" (або працюй поверх них)

**Не вчіть**: BOSH (інструмент розгортання CF), Diego (оркестратор CF), специфічні концепції CF.

## Цвинтар середовищ виконання (Runtimes)

### Docker як середовище виконання Kubernetes

**Чим він був**: Оригінальне середовище виконання контейнерів для Kubernetes.

**Статус**: Видалено з Kubernetes у версії 1.24 (травень 2022).

```
Хронологія:
2014-2016: Docker — ЄДИНИЙ спосіб запуску контейнерів у K8s
2016: Запроваджено CRI (Container Runtime Interface)
2017: containerd стає проєктом CNCF
2020: K8s оголошує про відмову від dockershim
2022: K8s 1.24 повністю видаляє dockershim
```

**Чому його видалили**:
- Docker — це ціла платформа, а K8s потрібен лише рантайм
- containerd (двигун Docker) працює з K8s напряму
- Стандартизація CRI зробила надбудову Docker непотрібною
- Покращення безпеки та продуктивності

**Не вчіть**: Специфічні для Docker налаштування K8s, усунення несправностей dockershim.

**Все ще актуально**: Docker для створення образів. `docker build` та Dockerfiles — це чудово.

```
┌─────────────────────────────────────────────────────────────┐
│              DOCKER ПРОТИ CONTAINERD В K8S                  │

│                                                             │
│  До K8s 1.24:                                              │
│  ┌───────────┐    ┌───────────┐    ┌───────────┐          │
│  │ Kubernetes│───►│  Docker   │───►│containerd │───► 🐳   │
│  └───────────┘    └───────────┘    └───────────┘          │
│                      Зайвий шар                             │
│                                                             │
│  Після K8s 1.24:                                           │
│  ┌───────────┐    ┌───────────┐                           │
│  │ Kubernetes│───►│containerd │───► 🐳                    │
│  └───────────┘    └───────────┘                           │
│                      Прямо, ефективно                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Цвинтар керування конфігураціями

### Chef/Puppet/Ansible для Kubernetes

**Чим вони були**: Інструменти керування конфігураціями для серверів.

**Статус**: Неправильна парадигма для Kubernetes.

**Чому вони не підходять**:

| Традиційне CM | Kubernetes Native |
|---------------|-------------------|
| Мутабельні сервери | Іммутабельні контейнери |
| SSH на сервери | Зміни через API |
| Наближення до стану | Декларування бажаного стану |
| Агент на кожному сервері | Агенти не потрібні |
| Імперативні скрипти | Декларативний YAML |

```
Традиційно (Chef/Puppet/Ansible):
1. SSH на сервер
2. Перевірка поточного стану
3. Застосування змін для досягнення бажаного стану
4. Надія, що стан не відхилиться згодом

Kubernetes:
1. Визначення бажаного стану в YAML
2. kubectl apply
3. K8s постійно узгоджує стан
4. Самолікування, жодного SSH
```

**Не вчіть**: Використання Ansible для керування ресурсами K8s, «кукбуки» Chef для K8s, маніфести Puppet для контейнерів.

**Виняток**: Ansible/Terraform для створення самого кластера K8s (інфраструктури), але не для керування навантаженнями всередині нього.

---

### Docker Compose для продакшну

**Чим він є**: Оркестрація декількох контейнерів для Docker.

**Статус**: Чудово для розробки, неправильно для продакшну.

```
Docker Compose ідеальний для:
- Локальної розробки
- Тестових середовищ CI/CD
- Демо-стендів

Docker Compose НЕ підходить для:
- Оркестрації в продакшні
- Розгортання на багатьох вузлах
- Вимог високої доступності (HA)
```

**Чому він не масштабується**:
- Працює лише на одному вузлі
- Немає самолікування
- Немає оновлень без простоїв (rolling updates)
- Немає вбудованого виявлення сервісів (service discovery)
- Немає керування секретами

**Не вчіть**: Розгортання Compose у продакшні, Compose у режимі Swarm.

## Чому ці технології померли

Типові патерни технологічних глухих кутів:

### Патерн 1: Один вендор проти спільноти

```
Swarm:    Контролюється Docker Inc. → Обмежене впровадження
K8s:      Нейтральна CNCF → Прийняття всією індустрією
Урок:     Управління спільнотою перемагає в інфраструктурі
```

### Патерн 2: Складність без вигоди

```
Mesos:    Потужний, але складний → Обмежена екосистема
K8s:      Складний, але цінний → Величезна екосистема
Урок:     Складність прийнятна лише за пропорційної вигоди
```

### Патерн 3: Неправильний рівень абстракції

```
Chef/Puppet: Рівень сервера → Не підходить контейнерам
K8s:         Рівень контейнера → Ідеально підходить
Урок:        Зміна парадигми потребує нових інструментів
```

### Патерн 4: Ефект екосистеми

```
Як тільки K8s досяг критичної маси:
- Хмари створили керовані сервіси
- Вендори інструментів орієнтувалися на K8s
- Фахівці вчили K8s
- Альтернативи стали нежиттєздатними

Урок: Мережеві ефекти потужні. Іноді найкраща технологія програє.
```

## Чи знали ви?

- **Mesos забезпечував роботу всієї інфраструктури Twitter** на піку. У команді Mesos було понад 100 людей. Вони все одно мігрували на Kubernetes.

- **Docker Inc. ледь не збанкрутувала.** Компанія, що почала революцію контейнерів, не змогла підтримувати бізнес-модель. Mirantis придбала Docker Enterprise.

- **Kubernetes ледь не назвали «Seven of Nine»** (посилання на Star Trek, як і Borg). Натомість обрали назву Kubernetes, бо вона звучала професійніше.

- **Chef та Puppet все ще прибуткові** для керування традиційними серверами. Вони просто не є правильним інструментом для хмарно-нативних навантажень.

## Як оцінювати нові технології

Коли з'являються нові технології, запитуйте:

1. **Чи розв'язує вона реальну проблему?** Чи створює складність заради складності?

2. **Вона керується спільнотою чи одним вендором?** Управління спільнотою краще масштабується.

3. **Чи відповідає вона принципам cloud-native?** Декларативність, керування через API, іммутабельність?

4. **Чи є імпульс у екосистеми?** Інструменти, інтеграції, фахівці?

5. **Чи впроваджують її великі хмарні провайдери?** Вони мають сильні сигнали щодо життєздатності технологій.

## Тест

1. **Чому Docker було видалено як рантайм Kubernetes у версії 1.24?**
   <details>
   <summary>Відповідь</summary>
   Docker — це повноцінна платформа, тоді як K8s потребував лише компонента середовища виконання (runtime). containerd (внутрішній рантайм Docker) працює з K8s напряму через CRI, усуваючи непотрібні накладні витрати Docker. Це покращило продуктивність та безпеку.
   </details>

2. **Чому Chef/Puppet/Ansible не підходять для моделі Kubernetes?**
   <details>
   <summary>Відповідь</summary>
   Вони розроблені для мутабельних серверів (підключення по SSH, застосування змін). Kubernetes використовує іммутабельні контейнери та декларативну конфігурацію через API. Парадигма принципово інша: K8s постійно узгоджує бажаний стан, а не просто застосовує зміни один раз.
   </details>

3. **Чому Mesos/Marathon програв Kubernetes, попри використання в Twitter?**
   <details>
   <summary>Відповідь</summary>
   Mesos був складним (дворівневе планування), Marathon не встигав за функціями K8s, а екосистема так і не виросла. K8s мав нейтральне управління CNCF, підтримку хмарних провайдерів та вибухове зростання екосистеми. Навіть Twitter мігрував на K8s.
   </details>

4. **Чи актуальний Docker зараз? У якому сенсі?**
   <details>
   <summary>Відповідь</summary>
   Так, для створення образів контейнерів. `docker build` та Dockerfiles залишаються основним способом створення образів. Docker як рантайм для Kubernetes застарів, але Docker для розробки та збірки образів залишається стандартом.
   </details>

## Вправа для роздумів

Цей модуль вчить того, чого НЕ варто вчити — це так само цінно, як і знання того, що вчити:

**1. Оцінка незворотних витрат (Sunk cost):**
- Чи витрачали ви час на технологію, яка згодом застаріла?
- Як ви зрозуміли, що час рухатися далі?
- Що б ви зробили інакше?

**2. Розпізнавання патернів:**
- Модуль описує патерни невдалих технологій (один вендор, неправильна абстракція тощо).
- Чи можете ви назвати сучасні технології, що мають схожі ознаки?
- Як би ви оцінили «гарячий новий інструмент»?

**3. Питання Docker:**
- Чому люди досі кажуть «Docker», коли мають на увазі «контейнери»?
- Чи варто все ще знати Docker? (Так, для збірки образів).
- Який урок у розрізненні інструментів та концепцій?

**4. Вплив на кар'єру:**
- Якби ви були експертом з Mesos у 2018 році, що б ви робили зараз?
- Як ви будуєте навички, що залишаються цінними навіть при зміні інструментів?
- Чи є «експерт із Kubernetes» надійнішою ставкою, ніж «експерт із Docker Swarm»? Чому?

**5. Захист від майбутнього:**
- Що мало б змінитися, щоб Kubernetes став «глухим кутом»?
- Чи є нові технології, що можуть кинути виклик K8s?
- Як би ви помітили їх на ранній стадії?

Здатність оцінювати технологічні ставки — це кар'єрна навичка, що виходить за межі будь-якого конкретного інструмента.

## Підсумок

Знання того, чого уникати, економить час:

**Мертва оркестрація**:
- Docker Swarm (застарів)
- Mesos/Marathon (покинутий)
- Cloud Foundry оригінальна модель (перейшов на K8s)

**Мертвий рантайм**:
- Docker як рантайм K8s (видалено в K8s 1.24)

**Неправильна парадигма**:
- Chef/Puppet/Ansible для навантажень K8s
- Docker Compose для продакшну

**Патерни невдач**:
- Контроль одного вендора
- Складність без пропорційної вигоди
- Неправильний рівень абстракції
- Поразка в битві екосистем

Витрачайте свій час на актуальні, підтримувані, широко впроваджені технології з сильним управлінням спільнотою.
