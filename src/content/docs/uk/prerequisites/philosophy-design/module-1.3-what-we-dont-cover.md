## Чому цей модуль важливий

Ось секрет, про який вам не розкажуть на платних курсах підготовки до сертифікації: **ви можете витратити понад 100 годин на вивчення тем, які ніколи не з'являться на іспиті.** Ми бачили, як це стається — інженери тижнями занурюються у внутрішню будову etcd або створюють власні CNI-плагіни, щоб потім виявити, що CKA взагалі цього не перевіряє.

KubeDojo підходить хірургічно до того, що ми викладаємо. Кожен модуль існує тому, що він або з'являється на іспиті, або є критично важливим для розуміння того, що там є. Цей модуль розповість вам, що саме ми пропускаємо і чому — щоб ви не витратили жодної години на непотрібні речі.

## Наша філософія: Фокус на іспиті, а не вичерпність

KubeDojo існує для того, щоб допомогти вам ефективно скласти іспити на сертифікацію. Ми застосовуємо правило 80/20:

```
┌─────────────────────────────────────────────────────────────┐
│              ПРАВИЛО 80/20 ДЛЯ СЕРТИФІКАЦІЙ                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ████████████████████████████████████████ 80% питань       │
│  █ Основні теми, які ми вивчаємо глибоко█ іспиту          │
│  ████████████████████████████████████████                   │
│                                                             │
│  ████████████ 20% питань іспиту                            │
│  █ Рідкісні випадки, складні теми,      █                  │
│  █ що рідко перевіряються               █                  │
│  ████████████                                               │
│                                                             │
│  Результат: Максимальна ймовірність успіху при ефективному  │
│             навчанні                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

Ми не намагаємося охопити все. Ми охоплюємо те, що важливо для успішного складання.

## Що ми свідомо пропускаємо

### 1. Специфіка хмарних провайдерів

| Тема | Чому ми пропускаємо | Де вивчити |
|------|--------------------|------------|
| Деталі AWS EKS | Іспит використовує стандартний K8s, а не хмарні реалізації | Документація AWS, документація eksctl |
| Функції GKE | З тієї ж причини | Документація Google Cloud |
| Специфіка AKS | З тієї ж причини | Документація Azure |
| Інтеграція з хмарним IAM | Залежить від провайдера | Документація провайдера |

**Наш підхід**: Ми викладаємо Kubernetes. Хмарні провайдери додають свої власні рівні. Вивчіть платформу, а потім вивчіть реалізацію вашого провайдера.

### 2. Експлуатація в продакшні у великих масштабах

| Тема | Чому ми пропускаємо | Де вивчити |
|------|--------------------|------------|
| Федерація мультикластерів | Поза межами сертифікації | Документація K8s, проєкт KubeFed |
| Автомасштабування кластера | Хмарні реалізації | Документація провайдера |
| Аварійне відновлення (DR) | Залежить від організації | Книги SRE, внутрішні регламенти компанії |
| Оптимізація витрат | Залежить від хмари | Ресурси FinOps, хмарні калькулятори |

**Наш підхід**: Сертифікації перевіряють фундамент. Експлуатація в продакшні потребує досвіду та контексту конкретної компанії.

### 3. Глибокі мережеві технології

| Тема | Чому ми пропускаємо | Де вивчити |
|------|--------------------|------------|
| Конфігурація BGP | CKA торкається лише основ | Ресурси з мережевої інженерії |
| Внутрішня будова Service Mesh | Istio/Linkerd — це окремі домени | Документація відповідних проєктів |
| Внутрішня будова eBPF/Cilium | Просунута мережева тема | Документація Cilium |
| Розробка власних CNI | Тема для розробників, не для адмінів | Специфікація CNI |

**Наш підхід**: Ми охоплюємо мережеві концепції, які перевіряються на іспиті. Глибокі мережі — це окрема спеціалізація.

### 4. Конкретні інструменти/проєкти

| Інструмент | Чому ми пропускаємо | Де вивчити |
|------------|--------------------|------------|
| ArgoCD | Інструмент GitOps, немає на іспиті | Документація ArgoCD |
| Istio | Service mesh, окремий трек сертифікації | Документація Istio |
| Terraform | Інструмент IaC, не специфічний для K8s | Навчальні матеріали HashiCorp |
| Prometheus/Grafana | Інструменти спостережуваності, лише коротко | Документація проєктів |

**Наш підхід**: Ми згадуємо ці інструменти для контексту, але не вивчаємо їх глибоко. Кожен із них заслуговує на власну програму навчання.

## Чому ми робимо такий вибір

### Причина 1: Економія часу

```
Вивчення всього про Kubernetes: 6-12 місяців
Підготовка до CKA з фокусованим навчанням: 4-8 тижнів

Різниця: Фокус
```

Сертифікації — це ворота, а не кінцева точка. Пройдіть їх ефективно, а потім заглиблюйтесь у те, чого вимагає ваша робота.

### Причина 2: Актуальність для іспиту

CKA, CKAD та CKS мають чітко визначені програми. Ми дотримуємося цих програм. Додавання контенту поза іспитом:
- Збільшує час навчання
- Додає плутанини
- Не покращує показники успішності

### Причина 3: Уникнення застарілого контенту

Kubernetes розвивається швидко. Чим більше тем ми охоплюємо, тим більше нам доводиться підтримувати. Фокусуючись на актуальному для іспиту:
- Менше контенту для оновлення
- Вища якість того, що ми висвітлюємо
- Більша сталість проєкту в довгостроковій перспективі

## Де ми радимо заглиблюватися

Після отримання сертифікатів ви захочете піти далі. Ось наш рекомендований шлях:

### Для платформних інженерів (Platform Engineers)
- Cluster API
- GitOps (ArgoCD/Flux)
- Патерни мультиорендності (Multi-tenancy)
- Ресурси з платформної інженерії

### Для розробників (Developers)
- Патерни Kubernetes (sidecar, ambassador, adapter)
- Розробка операторів та CRD
- Програмування Kubernetes API
- Плагіни для kubectl

### Для фокусу на безпеці (Security)
- Глибоке занурення в OPA/Gatekeeper
- Falco та безпека середовища виконання (runtime)
- Безпека ланцюжка постачання (Sigstore)
- Просунуті патерни мережевих політик

### Для SRE/експлуатації
- Алертинг на основі SLO
- Chaos engineering
- Планування потужностей (Capacity planning)
- Реагування на інциденти

## Чи знали ви?

- **Ландшафт CNCF налічує понад 1000 проєктів.** Жодна людина не може опанувати їх усі. Спеціалізація необхідна.

- **Більшість користувачів K8s у продакшні глибоко знають лише ~20% системи.** Вони знають те, чого вимагає їхня робота. Сертифікації доводять широту знань, робота — глибину.

- **Програми сертифікацій змінюються.** Теми додаються та видаляються. Ми відстежуємо зміни та оновлюємо матеріали відповідно.

- **Поняття "експерт" залежить від контексту.** Експерт із мереж знає інші речі, ніж експерт із безпеки. Не існує універсального чекліста "експерта з Kubernetes".

## Поширені запитання

### "Але мені може знадобитися [тема] на роботі!"

Цілком можливо. Наша порада:
1. Спочатку складіть іспит (це підтвердить фундамент)
2. Вивчайте специфічні для роботи теми безпосередньо на практиці
3. Використовуйте документацію проєктів для конкретних інструментів (ArgoCD, Istio тощо)

### "Чому б не включити все, щоб курс був повним?"

Повнота неможлива і контрпродуктивна:
- K8s постійно змінюється
- Більше контенту = складніша підтримка = застарілий контент
- Фокусований контент = вища якість

### "Що, якщо на іспиті запитають про те, що ви не розглянули?"

Наша програма відповідає офіційним цілям іспиту. Якщо щось є в екзаменаційних вимогах — воно є в нашому курсі. Рідкісні випадки, що трапляються раз на сто іспитів, не варті часу на вивчення.

## Тест

1. **Чому KubeDojo пропускає специфіку хмарних провайдерів (EKS, GKE, AKS)?**
   <details>
   <summary>Відповідь</summary>
   Екзамени на сертифікацію перевіряють стандартний Kubernetes, а не специфічні хмарні реалізації. Вивчення особливостей провайдера після розуміння ядра K8s є більш ефективним. Документація провайдерів покриває їхні специфічні функції.
   </details>

2. **Що таке правило 80/20 у контексті підготовки до сертифікації?**
   <details>
   <summary>Відповідь</summary>
   80% питань іспиту стосуються основних тем. Фокусування на цих темах максимізує ймовірність успіху при мінімальних витратах часу. Рідкісні випадки та теми, що майже не тестуються, не варті глибокого вивчення на початковому етапі.
   </details>

3. **Чому KubeDojo не викладає ArgoCD, Istio або Terraform?**
   <details>
   <summary>Відповідь</summary>
   Це окремі інструменти зі своєю кривою навчання. Вони не є частиною основних сертифікацій Kubernetes. Кожен із них заслуговує на окремий навчальний курс. Ми згадуємо їх для контексту, але не викладаємо глибоко.
   </details>

4. **Що варто робити після отримання сертифіката?**
   <details>
   <summary>Відповідь</summary>
   Заглиблюватися в теми, яких вимагає ваша посада. Сертифікація підтверджує широту поглядів; робота будує глибину знань. Використовуйте документацію проєктів для конкретних інструментів і спеціалізуйтеся відповідно до вашої ролі (платформна інженерія, безпека, SRE тощо).
   </details>

## Вправа для роздумів

Цей модуль встановлює очікування — подумайте про свій шлях навчання:

**1. Ваші цілі навчання:**
- Чому ви вивчаєте Kubernetes?
- Чи є сертифікація кінцевою метою чи лише сходинкою?
- Яка конкретна роль або проєкт мотивує вас?

**2. Принцип 80/20:**
- У вашій сфері діяльності, які 20% знань покривають 80% ситуацій?
- Чи завжди надмірне заглиблення допомагає, чи воно може відволікати?

**3. Дисципліна обсягу знань:**
- Чи витрачали ви коли-небудь час на вивчення того, що виявилося неактуальним?
- Як ви вирішуєте, що варто вивчати глибоко, а про що достатньо просто знати?

**4. Після сертифікації:**
- Яка спеціалізація вас цікавить? (Платформна інженерія, безпека, SRE, розробка?)
- Як би виглядала ваша ідеальна робота з Kubernetes?

**5. Бюджет часу:**
- Скільки часу ви можете приділити навчанню?
- З огляду на фокус KubeDojo, чи потрібні вам додаткові ресурси для ваших специфічних цілей?

Розуміння меж курсу допомагає вчитися ефективно та планувати розвиток поза межами сертифікації.

## Підсумок

KubeDojo робить свідомий вибір щодо обсягу матеріалу:

- **Ми охоплюємо**: Програми сертифікацій у повному обсязі
- **Ми пропускаємо**: Специфіку хмар, великі масштаби продакшну, глибокі мережі, сторонні інструменти
- **Наша філософія**: Ефективність фокусу на іспиті замість вичерпного охоплення всього на світі
- **Після сертифікації**: Спеціалізуйтеся відповідно до вашої ролі та робочих потреб

Розуміння цих меж допомагає встановити правильні очікування та спланувати вашу загальну стратегію навчання.
