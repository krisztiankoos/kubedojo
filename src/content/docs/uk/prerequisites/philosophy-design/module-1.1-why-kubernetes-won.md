## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Пояснити**, чому Kubernetes переміг Docker Swarm, Mesos та Nomad з конкретними технічними та екосистемними причинами
- **Визначити** патерни, що роблять технологію «переможцем» (спільнота, розширюваність, підтримка хмарних провайдерів)
- **Оцінити** нові технологічні заяви, запитуючи «чи має це екосистемні маркери, які мав Kubernetes?»
- **Описати** хронологію від Borg до Kubernetes і чому Google зробив його відкритим

## Чому цей модуль важливий

Була 3-тя година ночі «Чорної п'ятниці» 2014 року. Розробник стартапу з електронної комерції, що швидко ріс, дивився на 47 вікон терміналу, кожне з яких було підключене через SSH до окремого сервера. Їхній розпродаж став вірусним — трафік у 10 разів перевищив очікуваний. Контейнери Docker падали швидше, ніж він встигав їх перезапускати. «Збільшити сервер 23... ні, чекайте, 23 переповнений, спробуйте 31... 31 впав, коли це сталося?» До 4-ї ранку сайт був офлайн. До понеділка генеральний директор запитував, чому вони втратили 340 000 доларів продажів.

Цей кошмар розробника — саме те, чому існує Kubernetes. Не як академічна вправа, не як марнославний проєкт Google — а тому, що **ручне керування контейнерами у великому масштабі — це проблема, яка з'їсть вас живцем.**

Kubernetes переміг не випадково. Розуміння «війн оркестрації» допомагає зрозуміти, чому існують певні патерни й чому альтернативи зазнали невдачі.

---

## Проблема: Оркестрація контейнерів

До 2014 року контейнери довели свою цінність. Docker зробив їх доступними. Але виникла нова проблема:

**Як запускати тисячі контейнерів на сотнях машин?**

Ручне керування не масштабується. Вам потрібно:
- Автоматичне планування (де має працювати цей контейнер?)
- Самолікування (що відбувається, коли контейнер гине?)
- Масштабування (як впоратися з більшим трафіком?)
- Мережа (як контейнери знаходять один одного?)
- Оновлення (як розгортати без простоїв?)

Це і є **оркестрація контейнерів**.

## Претенденти

### Docker Swarm

**Простий вибір**

Відповідь Docker Inc. на потребу оркестрації. Вбудований у сам Docker.

```
Плюси:
- Простий у налаштуванні
- Нативна інтеграція з Docker
- Знайомий синтаксис Docker Compose
- "Просто працює" для невеликих розгортань

Мінуси:
- Обмежений набір функцій
- Слабка підтримка мультихмарності
- Прив'язка до вендора (Docker Inc.)
- Обмеження масштабування
```

**Що сталося**: Docker Inc. поставила все на Swarm. Коли переміг Kubernetes, Docker Inc. врешті змінила курс (була придбана Mirantis у 2019 році). Зараз Swarm фактично застарів.

### Apache Mesos + Marathon

**Вибір корпорацій**

Народився в Каліфорнійському університеті в Берклі, використовувався Twitter та Airbnb. Marathon забезпечував шар оркестрації контейнерів поверх Mesos.

```
Плюси:
- Перевірений у величезних масштабах (Twitter)
- Гнучкий (запускає контейнери ТА інші типи навантажень)
- Дворівнева архітектура планування
- Доведений досвід у продакшні

Мінуси:
- Складний в експлуатації
- Крута крива навчання
- Менша екосистема
- Потребував окремих компонентів (Marathon, Chronos)
```

**Що сталося**: Twitter відмовився від Mesos у 2020 році, перейшовши на Kubernetes. Mesosphere (компанія) змінила назву на D2iQ і тепер продає... Kubernetes. Marathon покинутий.

### Kubernetes

**Вибір Google**

Внутрішня система Google «Borg» оркеструвала контейнери понад десятиліття. Kubernetes став наступником Borg із відкритим кодом, переданим новоствореній CNCF.

```
Плюси:
- Десятирічний досвід Google
- Декларативна модель (бажаний стан)
- Величезна екосистема
- Cloud-native фундамент
- Сильне управління спільнотою

Мінуси:
- Складний
- Крута крива навчання
- "Занадто багато" для простих розгортань
```

**Що сталося**: Він переміг. Рішуче.

## Чому Kubernetes переміг

### 1. Декларативна модель

Kubernetes запровадив принципово інший підхід:

```
Імперативний (Swarm/Традиційний):
"Запусти 3 контейнери nginx на server-1"
"Якщо один помре, запусти інший"
"Якщо трафік зросте, запусти ще 2"

Декларативний (Kubernetes):
"Я хочу, щоб працювало 3 репліки nginx. Завжди."
(Kubernetes сам вирішує, як цього досягти)
```

Ця зміна глибока. Ви описуєте *чого ви хочете*, а не *як цього досягти*. Kubernetes постійно узгоджує реальність із вашим бажаним станом.

### 2. Досвід Google

Google запускав контейнери у величезних масштабах понад десять років за допомогою Borg. Kubernetes втілив уроки, засвоєні з:
- Мільярдів запусків контейнерів на тиждень
- Збоїв на всіх можливих рівнях
- Того, що справді працює в масштабі

Це не була перша спроба стартапу — це була система третього покоління від Google, випущена як відкритий код.

### 3. Ефект екосистеми

Kubernetes прийняв розумні архітектурні рішення:
- **Розширюваність**: Custom Resource Definitions (CRDs) дозволяють будь-кому розширювати K8s
- **Модульність**: Середовища виконання контейнерів, мережа, сховища — все підключається через плагіни
- **API-first**: Все є API, що дозволяє створювати безліч інструментів

Це спричинило вибух проєктів екосистеми. Сьогодні ландшафт CNCF налічує понад 1000 проєктів, побудованих на базі або навколо Kubernetes.

### 4. Прийняття хмарними провайдерами

До 2017 року всі основні хмарні провайдери пропонували керований Kubernetes:
- Google Kubernetes Engine (GKE)
- Amazon Elastic Kubernetes Service (EKS)
- Azure Kubernetes Service (AKS)

Це було безпрецедентно. Конкуренти зазвичай не впроваджують технології один одного. Але Kubernetes був:
- З відкритим кодом (жоден вендор ним не володіє)
- Під управлінням CNCF (нейтральний фонд)
- Ставав стандартом (його неможливо було ігнорувати)

### 5. Спільнота та управління

Фонд CNCF (Cloud Native Computing Foundation) забезпечив:
- Нейтральне управління (не контролюється Google)
- Вендор-нейтральну сертифікацію
- Чіткі правила внесків
- Захист торгової марки

Це означало, що компанії могли інвестувати в Kubernetes, не боячись прив'язки до одного постачальника.

## Хронологія

```
2013: Запуск Docker, контейнери стають мейнстрімом
2014: Google анонсує Kubernetes (червень)
      Анонсовано Docker Swarm
      Mesos/Marathon набирає обертів

2015: Реліз Kubernetes 1.0 (липень)
      Створення CNCF, передача Kubernetes
      Початок "війн оркестрації"

2016: Pokemon Go працює на Kubernetes (масштабне підтвердження)
      Усі великі хмари анонсують підтримку K8s

2017: Docker Inc. додає підтримку Kubernetes (капітуляція)
      Kubernetes стає стандартом де-факто

2018-2019: Swarm застаріває, Mesos покинутий
           Домінування Kubernetes стає повним

2020+: Фокус зміщується з "чи варто використовувати K8s?" на
       "як використовувати K8s краще?"
```

## Чи знали ви?

- **Kubernetes означає «керманич» грецькою.** Логотип — це штурвал корабля. Сім спиць представляють оригінальних сімох розробників.

- **Borg був названий на честь Star Trek.** Внутрішній попередник Kubernetes у Google. Інші проєкти Google наслідували це: Omega (ще одне посилання на Star Trek).

- **Запуск Pokemon Go став віхою для K8s.** Коли гра вийшла у 2016 році, трафік у 50 разів перевищив очікуваний. Kubernetes автоматично масштабував бекенд. Це стало тим доказом, який переконав багато корпорацій.

- **Docker намагався купити Kubernetes.** Перш ніж Kubernetes став відкритим проєктом, Docker звернувся до Google з пропозицією про покупку. Google відмовив і замість цього передав його CNCF.

## Поширені помилки

| Помилка | Реальність |
|---------|------------|
| "K8s переміг, бо це Google" | Google допоміг, але ключовим було нейтральне управління. AWS не прийняв би продукт, контрольований Google. |
| "Swarm програв через Docker" | Swarm програв, бо не зміг зрівнятися з функціями або екосистемою K8s. Проблеми компанії лише прискорили це. |
| "K8s — це завжди правильний вибір" | Для простих розгортань K8s може бути надмірним. Docker Compose все ще має свої сценарії використання. |
| "Mesos був гіршою технологією" | Mesos був потужним, але занадто складним. Технологія сама по собі не перемагає — важать екосистема й простота. |

## Чому ця історія важлива для навчання

Розуміння того, чому Kubernetes переміг, допомагає вам:

1. **Оцінити декларативний дизайн**: Петля узгодження (reconciliation loop) не є випадковою — це ключова інновація.

2. **Зрозуміти екосистему**: Знання того, чому існують CRD, допомагає використовувати їх ефективно.

3. **Уникати глухих кутів**: Ви не витрачатимете час на підходи, що відходять у минуле.

4. **Говорити професійною мовою**: На співбесідах і в роботі цей контекст демонструє глибше розуміння.

## Тест

1. **Яка фундаментальна різниця між підходом Kubernetes та Docker Swarm?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes використовує декларативну модель (ви визначаєте бажаний стан, K8s його забезпечує), тоді як Swarm був більш імперативним (ви даєте команди для внесення змін). Декларативна модель із постійним узгодженням — це ключова інновація Kubernetes.
   </details>

2. **Чому всі великі хмарні провайдери впровадили Kubernetes, попри те, що його розпочав Google?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes було передано CNCF, нейтральному фонду. Це означало, що жоден вендор його не контролював. Хмарні провайдери могли інвестувати в Kubernetes, не надаючи конкурентної переваги супернику.
   </details>

3. **Який був внутрішній попередник Kubernetes у Google?**
   <details>
   <summary>Відповідь</summary>
   Borg. Він забезпечував роботу навантажень Google понад десять років. Kubernetes увібрав у себе уроки, засвоєні під час експлуатації Borg у величезних масштабах.
   </details>

4. **Чому декларативна модель вважається кращою для інфраструктури?**
   <details>
   <summary>Відповідь</summary>
   Декларативні системи здатні до самолікування. Ви визначаєте, чого хочете, і система постійно працює над підтримкою цього стану. Якщо щось ламається, вона автоматично відновлюється. Імперативні системи потребують зовнішнього моніторингу та втручання.
   </details>

## Вправа для роздумів

Цей модуль концептуальний — CLI не потрібен. Подумайте над цими питаннями:

**1. Патерни впровадження технологій:**
- Як ви думаєте, чому хмарні провайдери прийняли Kubernetes, хоча його почав Google?
- Яку роль відіграла нейтральність CNCF?

**2. Декларативне бачення:**
- Чому «Я хочу 3 репліки» потужніше за «запусти 3 контейнери»?
- Що відбувається в кожній моделі, коли щось йде не так?

**3. Динаміка екосистеми:**
- Чому розширюваність (CRD) важила більше, ніж наявність усіх функцій «з коробки»?
- Чи можете ви згадати інші технології, що перемогли завдяки екосистемі, а не функціям?

**4. Уроки для майбутнього:**
- Що знадобиться, щоб щось замінило Kubernetes?
- Які сигнали вказували б на те, що K8s стає «глухим кутом»?

Ці питання допомагають критично оцінювати вибір технологій — навичка, що живе довше за будь-який конкретний інструмент.

## Підсумок

Kubernetes переміг у війнах оркестрації завдяки:
- **Декларативній моделі**: Визначайте бажаний стан, дозвольте K8s подбати про решту
- **Досвіду Google**: Десятиріччя уроків продакшну з Borg
- **Екосистемі**: Архітектура, що розширюється, дозволила створити величезну екосистему інструментів
- **Управлінню**: Нейтральність CNCF дозволила впровадження усій індустрії
- **Прийняттю хмарами**: Усі великі провайдери пропонують керований K8s

Альтернативи не програли тому, що були поганими — вони програли тому, що Kubernetes був краще позиційований для прийняття всією індустрією.
