## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Пояснити** різницю між декларативним та імперативним підходами з реальними прикладами K8s
- **Передбачити**, що зробить Kubernetes, коли ви вручну змінюєте щось, чим він керує (і чому)
- **Писати** декларативні маніфести замість імперативних команд для продакшен навантажень
- **Діагностувати** проблеми, спричинені імперативним мисленням у декларативній системі

## Чому цей модуль важливий

Інженер чергової зміни в SaaS-компанії отримала сповіщення о 2-й годині ночі — критично важливий под (pod) перебував у стані циклічного збою (crash-looping). Напівсонна, вона підключилася по SSH до вузла і вручну перезапустила контейнер. Проблему вирішено, час спати. Але Kubernetes негайно «вбив» її вручну запущений контейнер і замінив його на новий — який також пішов у циклічний збій. Вона перезапустила його знову. Kubernetes знову його «вбив». Протягом 45 хвилин вона боролася з системою, не усвідомлюючи, що **Kubernetes не був зламаний — вона намагалася думати імперативно в декларативному світі.**

Щойно вона оновила YAML-файл Деплойменту (Deployment), щоб виправити справжню помилку конфігурації, і виконала `kubectl apply`, система сама вилікувала себе за 12 секунд. У цьому полягає різниця між імперативним та декларативним мисленням — і це найважливіша концепція в Kubernetes.

Якщо ви розумієте декларативне мислення, Kubernetes стає логічним. Якщо ні — ви будете боротися з системою замість того, щоб використовувати її переваги.

## Два підходи

### Імперативний: Кажіть системі, ЩО РОБИТИ

```bash
# Імперативний підхід
ssh server1
docker run -d nginx
docker run -d nginx
docker run -d nginx
# Перевірка, чи працюють вони
docker ps
# Якщо один вмирає, запустіть інший
# Якщо трафік зростає, запустіть більше
# Якщо сервер виходить з ладу, підключіться по SSH інде і повторіть
```

Ви самі є «петлею керування». Ви спостерігаєте, приймаєте рішення і дієте.

### Декларативний: Кажіть системі, ЧОГО ВИ ХОЧЕТЕ

```yaml
# Декларативний підхід
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx
```

```bash
kubectl apply -f nginx-deployment.yaml
# Готово. Kubernetes бере на себе решту.
```

Kubernetes є «петлею керування». Він спостерігає, приймає рішення і діє — безперервно.

## Петля узгодження (Reconciliation Loop)

Це основний механізм Kubernetes:

```
┌─────────────────────────────────────────────────────────────┐
│              ПЕТЛЯ УЗГОДЖЕННЯ (RECONCILIATION LOOP)         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│    ┌──────────────┐                                        │
│    │              │                                        │
│    │  БАЖАНИЙ     │◄───── Ви визначаєте це (YAML)          │
│    │  СТАН        │                                        │
│    │              │                                        │
│    └──────┬───────┘                                        │
│           │                                                 │
│           │  Порівняння                                     │
│           ▼                                                 │
│    ┌──────────────┐                                        │
│    │              │                                        │
│    │  ПОТОЧНИЙ    │◄───── K8s спостерігає за цим           │
│    │  СТАН        │                                        │
│    │              │                                        │
│    └──────┬───────┘                                        │
│           │                                                 │
│           │  Якщо відрізняється...                          │
│           ▼                                                 │
│    ┌──────────────┐                                        │
│    │              │                                        │
│    │  ВЖИТИ       │◄───── K8s діє для узгодження           │
│    │  ЗАХОДІВ     │                                        │
│    │              │                                        │
│    └──────┬───────┘                                        │
│           │                                                 │
│           └─────────────────────────────────────────┐      │
│                                                     │      │
│           Повторювати вічно ◄───────────────────────┘      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**Ця петля працює постійно.** Не один раз, коли ви запускаєте `kubectl apply`, а завжди.

## Реальна аналогія: Термостат

### Імперативний (без термостата)

```
Ви: "Холодно. Увімкни обігрівач."
[Минає час]
Ви: "Тепер жарко. Вимкни обігрівач."
[Минає час]
Ви: "Знову холодно. Увімкни обігрівач."
[Повторювати вічно або здатися й страждати]
```

Ви самі є петлею керування.

### Декларативний (термостат)

```
Ви: "Я хочу, щоб було 22°C."
Термостат: [Постійно контролює та регулює]
```

Термостат є петлею керування. Ви задекларували свій бажаний стан (22°C), а система підтримує його.

## Чому декларативність перемагає

### 1. Самолікування

```yaml
# Ви кажете: "Я хочу 3 репліки"
spec:
  replicas: 3
```

Що відбувається, коли:
- Контейнер падає? → K8s запускає інший
- Вузол вмирає? → K8s переплановує поди на інші вузли
- Хтось вручну видаляє под? → K8s створює його знову

Ви не писали жодної логіки для цього. Ви просто задекларували, чого хочете.

### 2. Ідемпотентність

```bash
# Запустіть це 100 разів
kubectl apply -f deployment.yaml

# Результат: Однаковий стан щоразу
# Жодних дублікатів, жодних конфліктів, жодних помилок "вже існує"
```

Декларативні операції є **ідемпотентними** — повторне застосування тієї самої конфігурації дає той самий результат.

### 3. Контроль версій та GitOps

```bash
# Ваша інфраструктура — це код
git log --oneline
a1b2c3d feat: масштабування вебвузла до 5 реплік
d4e5f6g fix: збільшення ліміту пам'яті
g7h8i9j feat: додавання перевірок стану (health checks)

# Відкочування інфраструктури за допомогою git
git revert a1b2c3d
kubectl apply -f .
```

Декларативні конфігурації можна версіонувати, рецензувати та перевіряти.

### 4. Виявлення відхилень (Drift Detection)

```
Імперативний світ:
- Хтось підключився по SSH і вніс зміни
- Документація не відповідає реальності
- "У мене на машині працює"
- Страх торкатися продакшну

Декларативний світ:
- Git — єдине джерело істини
- K8s постійно забезпечує дотримання цієї істини
- Зміни потребують PR-рев'ю
- Впевненість у розгортаннях
```

## Імперативна пастка

Kubernetes має імперативні команди:

```bash
# Вони працюють, але...
kubectl run nginx --image=nginx
kubectl scale deployment nginx --replicas=5
kubectl set image deployment/nginx nginx=nginx:1.19
```

**Чому вони небезпечні:**

1. **Немає журналу аудиту**: Хто запустив цю команду? Коли?
2. **Немає перевірки (review)**: Зміни обходять процес PR/рев'ю
3. **Відхилення (Drift)**: Стан системи не відповідає жодному файлу
4. **Невідтворюваність**: "Які команди ми запускали, щоб це налаштувати?"

**Коли імперативність — це ОК:**
- Навчання та експерименти
- Налагодження (тимчасові зміни)
- Надзвичайні ситуації (з негайним подальшим оновленням декларативної конфігурації)

**Найкраща практика:**
```bash
# Створіть YAML, не застосовуйте його одразу
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deployment.yaml

# Перевірте, зафіксуйте в git (commit), а потім застосуйте
kubectl apply -f deployment.yaml
```

## Чи знали ви?

- **Петля узгодження працює кожні 10 секунд** за замовчуванням. Контролери постійно перевіряють, чи відповідає реальність бажаному стану.

- **Кожен ресурс Kubernetes є декларативним.** Поди (Pods), Сервіси (Services), ConfigMaps — все це лише декларації бажаного стану, які узгоджують контролери.

- **GitOps народився з цієї моделі.** ArgoCD та Flux розширюють декларативне мислення: Git стає джерелом істини, а K8s узгоджує стан із Git.

- **Terraform використовує ту саму модель.** Декларативна революція виходить за межі K8s. Сучасні інструменти інфраструктури майже повсюдно декларативні.

## Типові помилки

| Помилка | Чому це шкодить | Рішення |
|---------|-----------------|---------|
| Використання імперативних команд у продакшні | Немає журналу аудиту, ризик відхилень | Завжди використовуйте `kubectl apply -f` |
| Редагування живих ресурсів через `kubectl edit` | Зміни не зберігаються в Git | Оновлюйте YAML-файли, застосовуйте з Git |
| Нерозуміння того, що зміни відбуваються безперервно | Сюрприз, коли K8s «скасовує» ручні зміни | Прийміть модель — змінюйте декларацію |
| Боротьба з петлею узгодження | Розчарування, милиці (workarounds) | Працюйте із системою, а не проти неї |

## Зміна мислення

### Старе мислення (Імперативне)
```
"Мені потрібно розгорнути мій застосунок"
→ Підключитися до сервера по SSH
→ Стягнути код
→ Зібрати
→ Запустити процес
→ Налаштувати nginx
→ Оновити файрвол
→ Протестувати
→ Задокументувати те, що я зробив
```

### Нове мислення (Декларативне)
```
"Мені потрібно розгорнути мій застосунок"
→ Визначити бажаний стан у YAML
→ Зафіксувати в Git
→ kubectl apply (або дозволити GitOps зробити це)
→ K8s бере на себе решту
→ Git І Є документацією
```

## Тест

1. **Що таке петля узгодження (reconciliation loop)?**
   <details>
   <summary>Відповідь</summary>
   Це безперервний процес, у якому Kubernetes порівнює бажаний стан (ваш YAML) із поточним станом (тим, що насправді працює) і вживає заходів, щоб вони збігалися. Ця петля працює вічно, а не лише тоді, коли ви застосовуєте зміни.
   </details>

2. **Чому імперативні команди небезпечні в продакшні?**
   <details>
   <summary>Відповідь</summary>
   Вони створюють відхилення (drift) між тим, що в Git, і тим, що працює. Немає журналу аудиту, немає процесу рецензування, і зміни не є відтворюваними. Стан системи не відповідає жодному файлу.
   </details>

3. **Що означає "ідемпотентність" і чому вона важлива?**
   <details>
   <summary>Відповідь</summary>
   Операція є ідемпотентною, якщо її запуск кілька разів дає той самий результат, що й запуск один раз. `kubectl apply` ідемпотентний — застосування одного й того самого YAML 100 разів дає однаковий стан. Це робить автоматизацію безпечною та передбачуваною.
   </details>

4. **Якщо ви вручну видалите под із Деплойменту, що станеться?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes негайно створить новий под на заміну. Контролер Деплойменту побачить, що поточний стан (2 поди) не відповідає бажаному стану (3 поди), і виконає узгодження, створивши ще один под.
   </details>

## Вправа для роздумів

Цей модуль концептуальний — подумайте над цими питаннями:

**1. Аналогія з термостатом:**
- Які ще системи у вашому житті працюють декларативно? (Круїз-контроль? Автояскравість?)
- Що робить їх простішими у використанні, ніж ручні альтернативи?

**2. Ваш поточний робочий процес:**
- Як ви розгортаєте ПЗ сьогодні?
- Це більше імперативно чи декларативно?
- Що б змінилося, якби ви перейшли на іншу модель?

**3. Наслідки самолікування:**
- Якщо Kubernetes «скасовує» ручні зміни — це функція чи баг?
- Як це змінює ваш підхід до усунення несправностей?

**4. Git та інфраструктура:**
- Чому зберігання інфраструктури як YAML у Git є потужним інструментом?
- На які запитання ви можете відповісти за допомогою `git log`, на які не могли відповісти раніше?

**5. Зміна ментальної моделі:**
- Оператор запитує: «Які команди мені запустити?». Декларативний мислитель запитує: «Якого стану я хочу досягти?».
- Яке запитання веде до створення більш придатних для підтримки систем? Чому?

Ця зміна в мисленні — від «як» до «що» — є найважливішою концепцією у всьому цьому навчальному курсі.

## Підсумок

Декларативна модель — це фундамент Kubernetes:

- **Бажаний стан**: Ви визначаєте, чого хочете (YAML)
- **Узгодження**: K8s постійно приводить реальність у відповідність до бажання
- **Самолікування**: Збої виправляються автоматично
- **Ідемпотентність**: Застосовуйте той самий конфіг багаторазово, отримуйте той самий результат
- **Контроль версій**: Ваша інфраструктура — це код у Git

Це не просто технічний вибір — це філософія, яка забезпечує надійну, масштабовану та перевірювану інфраструктуру.
