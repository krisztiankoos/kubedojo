# Модуль 0.6: Основи Git — Відстежуйте свою роботу

**Складність**: [ПОЧАТКІВЕЦЬ]
**Час на проходження**: 45 хвилин
**Передумови**: Модуль 0.5 (Редагування файлів)

## Що ви зможете зробити

Після завершення цього модуля ви зможете:
- **Ініціалізувати** локальні репозиторії Git та **налаштувати** ідентифікацію користувача для точної атрибуції комітів.
- **Створити** логічну історію комітів шляхом вибіркового індексування змін у файлах за допомогою `git add` та `git commit`.
- **Діагностувати** неочікувані стани репозиторію, аналізуючи результати виконання команд `git status`, `git log` та `git diff`.
- **Синхронізувати** локальні репозиторії з віддаленими серверами за допомогою `git push` та `git pull`.
- **Сформувати** файл `.gitignore` для запобігання відстеженню конфіденційних даних або згенерованих артефактів.

## Чому це важливо

Фінансова технологічна компанія середнього розміру використовувала спільний мережевий диск для зберігання файлів конфігурації своєї інфраструктури. Інженери підключали диск, відкривали файли у своїх локальних редакторах, вносили зміни та зберігали їх назад у мережу. Однієї п'ятниці вдень молодший інженер отримав завдання оновити тестове середовище. Помилково він відкрив production-файл `payment-processor-deployment.yaml` замість тестового варіанту. Він змінив тег образу контейнера на експериментальну збірку, зберіг файл і застосував конфігурацію до кластера.

За кілька хвилин система обробки платежів почала відхиляти транзакції. Коли старші інженери поспішили розібратися в ситуації, вони переглянули конфігураційний файл на мережевому диску, але він показував лише поточний, зламаний стан. Оскільки системи контролю версій не було, не існувало жодного історичного запису про те, як виглядав файл десять хвилин тому, хто його змінив або чому. Команді довелося вручну відновлювати production-конфігурацію з пам'яті та застарілої документації. Подальша затримка у відновленні роботи сервісу зупинила всі клієнтські транзакції на шість годин, що призвело до мільйонних збитків і серйозної шкоди для репутації.

Контроль версій запобігає саме таким сценаріям. Git, галузевий стандарт контролю версій, діє як надійна машина часу для вашого коду та конфігурації. Він фіксує кожну зміну, точно визначає, хто її зробив, і дозволяє миттєво повернутися до будь-якого попереднього стану. У сучасному cloud-native світі інфраструктура визначається як код. Якщо ви не можете з абсолютною впевненістю відстежувати, перевіряти та повертати код вашої інфраструктури, ви керуєте системою, яка є бомбою уповільненої дії. Опанування Git не є чимось необов'язковим для платформних інженерів; це базова навичка, на якій будується вся надійна автоматизація та співпраця.

Без Git сучасні конвеєри безперервної інтеграції та безперервного розгортання (CI/CD) не можуть існувати. Вся ідея автоматизованих розгортань ґрунтується на єдиному, надійному джерелі істини (source of truth), яке запускає автоматизацію щоразу, коли виявляється нова, затверджена зміна. Якщо ви не розумієте Git, ви не зможете зрозуміти сучасну доставку програмного забезпечення (software delivery).

## Розділ 1: Концепція контролю версій та архітектура Git

Перш ніж вводити будь-які команди, ви повинні зрозуміти, як Git сприймає ваші файли. Багато новачків мають труднощі з Git, оскільки ставляться до нього як до простого інструменту резервного копіювання (на зразок Dropbox або Google Drive). Git працює зовсім інакше. Він не просто безперервно синхронізує файли у фоновому режимі; він робить свідомі, незмінні знімки (snapshots) вашого проєкту в певні моменти часу, але лише тоді, коли ви явно даєте йому таку команду.

Уявіть Git як відеогру з ручними точками збереження. Ви можете грати, робити помилки, отримувати шкоду і заходити в глухий кут. Поки ви явно створюєте «точку збереження» перед спробою небезпечного бою з босом, ви завжди можете завантажити саме цей стан, якщо щось піде не так. Git працює так само, але для ваших текстових файлів.

### Централізований та розподілений контроль версій

Історично склалося так, що системи контролю версій, такі як Subversion (SVN) або Team Foundation Server (TFS), були централізованими. Існував один головний сервер, який зберігав історію. Якщо ви хотіли переглянути історію або зробити коміт (commit), вам потрібно було підключитися до інтернету, щоб зв'язатися з цим сервером. Якщо сервер виходив з ладу, ніхто не міг працювати.

Git — це **розподілена** система контролю версій. Коли ви використовуєте Git, ви не просто завантажуєте останні файли; ви завантажуєте всю історію проєкту. Ваш локальний ноутбук стає повноцінно функціонуючим репозиторієм. Ви можете переглядати історію, порівнювати версії та робити коміти, перебуваючи повністю в офлайні в літаку. Інтернет вам потрібен лише тоді, коли ви хочете синхронізувати свою історію з кимось іншим.

### Три дерева Git

Git керує вашими файлами, переміщуючи їх через три окремі логічні зони, які часто називають «деревами». Розуміння цього конвеєра є ключем до діагностики майже будь-якої проблеми з Git.

```text
+---------------------+       +---------------------+       +---------------------+
|                     |       |                     |       |                     |
|  Working Directory  | ----> |    Staging Area     | ----> |     Repository      |
|  (Your local files) |       | (The loading dock)  |       | (The saved history) |
|                     |       |                     |       |                     |
+---------------------+       +---------------------+       +---------------------+
           |                             |                             |
           |   1. Modify files           |                             |
           |---------------------------->|                             |
           |                             |   2. Group changes          |
           |                             |---------------------------->|
           |                             |                             |
           |                                                           |
           |<----------------------------------------------------------|
                               3. Restore old versions
```

1. **Робоча директорія (Working Directory)**: Це ваш поточний робочий простір на комп'ютері. Вона містить фактичні файли, які ви редагуєте, видаляєте або створюєте. Коли ви відкриваєте файл у Vim або VS Code, ви змінюєте робочу директорію. Git бачить ці зміни, але ще не зберіг їх.
2. **Зона підготовки (Staging Area / Index)**: Це надзвичайно важливий проміжний крок, унікальний для Git. Уявіть це як вантажну платформу або майданчик для підготовки до фотосесії. Перш ніж зробити остаточний знімок, ви точно обираєте, які змінені файли помістити на сцену. Ви можете додати до цієї зони одні файли, залишивши інші поза нею. Це дозволяє створювати вузьконаправлені, логічні коміти, навіть якщо ви змінили півсотні файлів одночасно.
3. **Репозиторій (Repository / Commit History)**: Це постійна база даних, де Git зберігає ваші знімки (які називаються комітами). Щойно файли зафіксовано тут, вони надійно записуються в історію з іменем автора, міткою часу та описовим повідомленням. Коміт математично запечатаний; його неможливо таємно змінити без зміни його унікального ідентифікатора.

### Активне навчання: Зупиніться та подумайте
> **Зупиніться та подумайте**: Ви щойно завершили тривалий сеанс дебагінгу. Ви виправили помилку підключення до бази даних у `db.py`, але під час пошуку помилки ви також додали тимчасові команди `print` у файли `auth.py` та `api.py`, які не хочете зберігати назавжди. Як архітектура Трьох дерев Git дозволяє вам створити чисту історію в цьому сценарії, не втрачаючи тимчасовий налагоджувальний код у робочій директорії?
>
> *Перевірка передбачення: Архітектура Трьох дерев відокремлює ваші робочі файли від того, що зберігається. Ви можете використовувати зону підготовки (Staging Area), щоб вибірково додати лише `db.py` для наступного коміту. Тимчасові команди `print` в `auth.py` та `api.py` безпечно залишаються у вашій робочій директорії (Working Directory), щоб ви могли продовжувати їх використовувати або видалити пізніше, не засмічуючи при цьому постійну історію репозиторію (Repository).*

## Розділ 2: Підготовка: Встановлення та конфігурація

За своєю суттю Git є інструментом командного рядка. Хоча графічні інтерфейси існують, вивчення команд термінала є обов'язковим для платформної інженерії, оскільки ви часто використовуватимете Git на віддалених серверах без графічних можливостей. Більшість сучасних дистрибутивів Linux і macOS постачаються з попередньо встановленим Git, або його можна легко додати за допомогою стандартного менеджера пакетів, такого як `apt`, `yum` або `brew`.

Щоб перевірити встановлення, відкрийте термінал і перевірте версію:

```bash
git --version
```

Очікуваний результат:
```text
git version 2.39.2
```

### Налаштування ідентифікації

Оскільки Git створений для співпраці потенційно тисяч розробників, він відмовляється створювати знімки, доки не знатиме точно, хто їх робить. Кожен коміт вимагає ім'я автора та адресу електронної пошти. Це критично важливо для підзвітності — якщо зміна інфраструктури призводить до падіння production, команді потрібно знати, кого запитати про обґрунтування цієї зміни.

Ви налаштовуєте це за допомогою команди `git config`. Прапорець `--global` застосовує ці налаштування до всіх репозиторіїв на вашому поточному комп'ютері, записуючи їх у прихований файл у вашій домашній директорії (`~/.gitconfig`).

```bash
# Set your name (use your real name, this appears in the history)
git config --global user.name "Alex Chen"

# Set your email address
git config --global user.email "alex.chen@example.com"
```

Ви можете перевірити свою конфігурацію в будь-який час, попросивши Git вивести всі його поточні налаштування:

```bash
git config --list
```

### Ініціалізація репозиторію: git init

Щоб наказати Git почати відстежувати проєкт, ви повинні ініціалізувати репозиторій у кореневій директорії вашого проєкту. Давайте створимо нову директорію для гіпотетичного проєкту Kubernetes та ініціалізуємо в ній Git.

```bash
# Create a new directory
mkdir k8s-webapp
cd k8s-webapp

# Tell Git to start tracking this directory
git init
```

Очікуваний результат:
```text
Initialized empty Git repository in /home/alex/k8s-webapp/.git/
```

Що саме зробила команда `git init`? Вона не сканувала ваш жорсткий диск магічним чином. Вона просто створила приховану директорію з назвою `.git` всередині вашої папки `k8s-webapp`. Ця прихована директорія `.git` і є тим самим «Репозиторієм» з нашої моделі Трьох дерев. Саме тут Git зберігає всі внутрішні об'єкти бази даних, стиснутий вміст файлів, граф історії комітів та локальну конфігурацію для цього конкретного проєкту.

Якщо ви видалите директорію `.git`, ви видалите всю історію версій проєкту, хоча ваші поточні робочі файли залишаться недоторканими на диску.

```bash
# List all files, including hidden ones
ls -la
```

Очікуваний результат:
```text
total 12
drwxr-xr-x 3 alex alex 4096 Oct 12 10:00 .
drwxr-xr-x 5 alex alex 4096 Oct 12 09:59 ..
drwxr-xr-x 7 alex alex 4096 Oct 12 10:00 .git
```

Після успішної ініціалізації репозиторію та налаштування вашої ідентифікації, ви заклали необхідний фундамент. Далі ми розглянемо, як наповнити цей репозиторій, свідомо переміщуючи файли через життєвий цикл Git, щоб створити ваші перші зафіксовані знімки (коміти).

## Розділ 3: Життєвий цикл знімків: Add, Commit та Status

Тепер, коли у нас є активний репозиторій, давайте пройдемося щоденним робочим процесом відстеження файлів. Команда, яку ви використовуватимете частіше за будь-яку іншу — сотні разів на день — це `git status`. Вона діє як ваш компас, повідомляючи вам точно, де знаходяться ваші файли в межах Трьох дерев.

### Крок 1: Зміна робочої директорії

Давайте створимо наш перший файл — базову конфігурацію простору імен Kubernetes.

```bash
cat << 'EOF' > namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: webapp-prod
EOF
```

Запустіть `git status`, щоб побачити, що Git думає про цей щойно створений файл:

```bash
git status
```

Результат:
```text
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	namespace.yaml

nothing added to commit but untracked files present (use "git add" to track)
```

Git розпізнає, що файл існує в робочій директорії, але позначає його як «Untracked» (невідстежуваний). Це означає, що Git ніколи раніше не бачив цього файлу. Він не моніторить його на наявність змін і не буде автоматично створювати його резервні копії. Git відстежує лише те, що ви явно наказуєте йому відстежувати.

### Крок 2: Переміщення до зони підготовки (git add)

Щоб повідомити Git, що ми хочемо додати цей файл до нашого найпершого знімка, ми повинні перемістити його до зони підготовки (на вантажну платформу) за допомогою команди `git add`.

```bash
git add namespace.yaml
```

Знову запустіть `git status`, щоб побачити зміну стану:

```bash
git status
```

Результат:
```text
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   namespace.yaml
```

Файл перемістився. Тепер він класифікується як «Changes to be committed» (Зміни, що будуть зафіксовані). Він знаходиться на вантажній платформі, очікуючи, поки фотограф зробить знімок.

### Крок 3: Створення знімка (git commit)

Щоб назавжди зберегти підготовлені зміни в репозиторії, ми використовуємо `git commit`. Кожен коміт вимагає повідомлення коміту (commit message), яке пояснює, *чому* була зроблена зміна. Написання хороших повідомлень комітів є ключовою професійною навичкою. Хороше повідомлення пояснює намір і контекст, а не лише те, які рядки змінилися.

```bash
git commit -m "feat: add production namespace definition"
```

Результат:
```text
[main (root-commit) a1b2c3d] feat: add production namespace definition
 1 file changed, 4 insertions(+)
 create mode 100644 namespace.yaml
```

Давайте ще раз перевіримо `git status`, щоб побачити кінцевий результат нашого робочого процесу:

```bash
git status
```

Результат:
```text
On branch main
nothing to commit, working tree clean
```

Тепер ваша робоча директорія описується як «clean» (чиста). Це означає, що кожен файл, який наразі знаходиться на вашому диску, точно відповідає останньому знімку, збереженому в базі даних `.git`. Жодних очікуючих змін немає.

### Історія з досвіду: Випадковий коміт із секретними даними

Молодший розробник локально тестував застосунок, який вимагав ключа доступу до AWS. Для зручності він жорстко закодував ключ безпосередньо у своєму файлі `deployment.yaml`, щоб просто перевірити, чи запустяться Pods. Усе спрацювало. Зрадівши, він виконав `git add .` (команду, яка без розбору додає до зони підготовки кожен змінений файл у всій директорії), а потім виконав `git commit -m "fix deployment"`.

Потім він відправив код у публічний репозиторій на GitHub. Протягом 120 секунд автоматизовані боти безпеки, які сканують публічні репозиторії GitHub, знайшли відкритий ключ AWS. Використовуючи скомпрометовані облікові дані, боти миттєво розгорнули сотні потужних серверів для майнінгу криптовалюти в багатьох регіонах AWS. На момент, коли розробник прокинувся наступного ранку, компанія отримала рахунок за хмарні послуги на суму 65 000 доларів.

**Урок**: Ніколи сліпо не використовуйте `git add .`, якщо ви не абсолютно впевнені в тому, що саме ви змінили. Завжди запускайте `git status` та `git diff` перед додаванням файлів до зони підготовки, щоб переконатися, що ви випадково не комітите паролі, ключі API, приватні ssh-ключі або тимчасові файли для дебагінгу.

Тепер, коли ви знаєте, як безпечно створювати знімки, вам неминуче знадобиться переглядати минулі знімки або аналізувати точні зміни, внесені в окремі файли з часом. У наступному розділі ми заглибимося в команди, які дозволяють аналізувати історію комітів і вивчати точні відмінності у файлах.

## Розділ 4: Подорож у часі: Log та Diff

Коли ви зробили кілька комітів, вам потрібні надійні способи перегляду хронології історії та розуміння того, що саме змінилося між різними моментами в часі.

Внесемо ще одну зміну до нашого проєкту. Ми оновимо файл простір імен, щоб додати мітку — типове завдання в Kubernetes для організаційних цілей.

```bash
cat << 'EOF' > namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: webapp-prod
  labels:
    environment: production
EOF
```

### Перегляд змін: git diff

Перш ніж індексувати або комітити файл, ви завжди повинні перевіряти, які саме рядки ви змінили. Пам'ять може підвести; diff є об'єктивним. Команда `git diff` порівнює ваш поточний робочий каталог (Working Directory) з останнім зробленим знімком.

```bash
git diff
```

Вивід:
```text
diff --git a/namespace.yaml b/namespace.yaml
index e46b825..8394c41 100644
--- a/namespace.yaml
+++ b/namespace.yaml
@@ -2,3 +2,5 @@ apiVersion: v1
 kind: Namespace
 metadata:
   name: webapp-prod
+  labels:
+    environment: production
```

**Як розшифрувати вивід diff:**
- Заголовки `--- a/namespace.yaml` та `+++ b/namespace.yaml` показують дві версії файлів, що порівнюються (стара проти нової).
- `@@ -2,3 +2,5 @@` — це заголовок фрагмента (chunk header). Він надає системі контекст щодо того, де приблизно у файлі відбулися зміни.
- Рядки, що починаються з пробілу, є незміненими рядками контексту. Git показує їх, щоб допомогти вам зорієнтуватися.
- Рядки, що починаються з `+` (зазвичай виділені зеленим кольором), є повністю новими доповненнями.
- Рядки, що починаються з `-` (зазвичай виділені червоним кольором), є видаленнями. Якщо ви змінили рядок, Git відображає це як видалення старого рядка та додавання нового.

### Активне навчання: Читання diff
> **Зупиніться та подумайте**: Уявіть, що ви виконали `git diff` для файлу Deployment і побачили такий вивід:
>
> ```text
> @@ -10,3 +10,3 @@
>  spec:
>    replicas: 3
> -  image: nginx:1.14
> +  image: nginx:1.24
> ```
>
> Перш ніж читати далі, що саме інженер зробив у цьому файлі? Будьте конкретними.
>
> *Перевірка здогадки: Інженер не додав повністю новий структурний елемент. Він змінив наявний рядок. Він видалив рядок, який вказував версію контейнер Nginx 1.14, і замінив його на рядок із версією 1.24. Це є оновленням версії образу контейнера.*

Тепер, коли ми переконалися, що наші зміни правильні і не містять секретів, проіндексуємо (stage) і закомітимо додавання нашої мітки:

```bash
git add namespace.yaml
git commit -m "chore: add environment label to namespace"
```

### Перегляд історії: git log

Щоб побачити хронологію ваших знімків, використовуйте команду `git log`. Це відкриє пейджер (зазвичай `less`), який покаже вашу історію у зворотному хронологічному порядку (найновіші спочатку).

```bash
git log
```

Вивід:
```text
commit 9f8e7d6c5b4a39281716151413121110abcdef12 (HEAD -> main)
Author: Alex Chen <alex.chen@example.com>
Date:   Wed Oct 12 10:45:12 2023 -0400

    chore: add environment label to namespace

commit a1b2c3d4e5f60718293a4b5c6d7e8f9012345678
Author: Alex Chen <alex.chen@example.com>
Date:   Wed Oct 12 10:15:30 2023 -0400

    feat: add production namespace definition
```

Зверніть увагу на 40-символьні шістнадцяткові рядки, показані вище. У більшості репозиторіїв це хеш коміту, згенерований за допомогою SHA-1. Це математично згенерований унікальний ідентифікатор для конкретного знімка, який відображає вміст файлу, автора, дату та батьківський коміт. Ви можете використовувати цей хеш, щоб перевірити або відновити цей точний момент в історії. Новіші репозиторії також можна налаштувати на використання SHA-256, але в прикладах цього модуля використовується поширений формат SHA-1.

Для більш компактного вигляду, який особливо корисний, коли ви досліджуєте репозиторій із тисячами комітів за кілька років, використовуйте прапорець `--oneline`:

```bash
git log --oneline
```

Вивід:
```text
9f8e7d6 chore: add environment label to namespace
a1b2c3d feat: add production namespace definition
```

Ви також можете використовувати `git log -p`, щоб побачити фактичні зміни (diffs), внесені кожним окремим комітом в історії, що дозволяє вам бачити не лише *факт* створення коміту, але й *які саме рядки* були ним змінені.

Оволодіння навичкою навігації вашою локальною хронологією надає величезну впевненість під час експериментів з інфраструктурним кодом. Однак сучасна інженерія — це командна робота; наступний розділ розповість, як безпечно ділитися вашою локальною історією з віддаленими серверами для безперешкодної співпраці з іншими.


## Розділ 5: Співпраця зі світом: Remotes, Push та Pull

До цього моменту все, що ми робили, існує виключно на жорсткому диску вашого локального ноутбука. Якщо ваш комп'ютер зламається або ви проллєте на нього чашку кави, вся історія вашого проєкту зникне назавжди. Крім того, ніхто інший у вашій команді не може бачити вашу роботу.

Щоб співпрацювати з іншими та створювати резервні копії вашої роботи, ви повинні підключити свій локальний репозиторій до віддаленого сервера (remote), такого як GitHub, GitLab або Bitbucket.

### Що таке Remote?

"Remote" (віддалений репозиторій) — це просто ще один репозиторій Git, розміщений у мережі або інтернеті. Оскільки Git децентралізований, кожен має повну, автономну копію історії. "Pushing" (відправлення) та "Pulling" (отримання) — це явні дії, які ми виконуємо для синхронізації нашої локальної хронології з віддаленою хронологією.

```text
+-----------------------+                    +-----------------------+
|                       |                    |                       |
|   Local Repository    |                    |   Remote Repository   |
|   (Your Laptop)       |                    |   (GitHub/GitLab)     |
|                       |                    |                       |
|  commit C (HEAD)      | ==== git push ===> |  commit C             |
|  commit B             |                    |  commit B             |
|  commit A             | <=== git pull ==== |  commit A             |
|                       |                    |                       |
+-----------------------+                    +-----------------------+
```

### Підключення до віддаленого репозиторію

Коли ви створюєте порожній репозиторій на платформі на кшталт GitHub, платформа надає вам URL-адресу для підключення (зазвичай HTTPS або SSH). Ви вказуєте вашій локальній інсталяції Git підключитися до цієї URL-адреси за допомогою команди `git remote add`.

За галузевою конвенцією, основний віддалений сервер за замовчуванням майже завжди називається `origin`.

```bash
# Example command (do not run unless you have a real repository URL prepared)
git remote add origin https://github.com/yourusername/k8s-webapp.git
```

Якщо ви приєднуєтеся до компанії і хочете завантажити існуючий проєкт, ви не використовуєте `git init` та `git remote add`. Замість цього ви використовуєте `git clone <url>`. Клонування автоматично ініціалізує локальний репозиторій, додає віддалений сервер як `origin` і завантажує всю історію та файли за один крок.

### Відправлення змін (git push)

Щоб завантажити ваші локально створені коміти на віддалений сервер, ви їх "пушите" (відправляєте). Ви повинні вказати ім'я віддаленого репозиторію (зазвичай `origin`) та ім'я гілки, яку ви відправляєте (часто `main` або `master`).

```bash
# Push your main branch to the origin remote for the first time
git push -u origin main
```

Прапорець `-u` означає "upstream" (висхідний потік). Його потрібно використовувати лише під час найпершого відправлення нової гілки. Він створює постійний зв'язок відстеження між вашою локальною гілкою `main` та віддаленою гілкою `main`. У майбутньому ви зможете просто ввести `git push` без жодних аргументів, і Git точно знатиме, куди надсилати дані.

### Отримання змін (git pull)

Якщо член команди вносить зміни, комітить їх локально та відправляє на GitHub, ваш локальний репозиторій *не* оновиться автоматично. Git ніколи не змінить ваші локальні файли без вашого прямого дозволу. Ви повинні активно звернутися до сервера, завантажити їхні нові коміти та інтегрувати їх у вашу локальну історію за допомогою команди `git pull`.

```bash
# Fetch changes from the remote and merge them into your local branch
git pull origin main
```

Під капотом `git pull` насправді є макросом, який послідовно виконує дві окремі команди: `git fetch` (яка безпечно завантажує нові коміти з віддаленого репозиторію, не змінюючи ваші робочі файли) та `git merge` (яка намагається безшовно об'єднати завантажені зміни з вашим поточним робочим каталогом).

### Активне навчання: Синхронізація з віддаленим репозиторієм
> **Зупиніться та подумайте**: Ви провели ранок, працюючи офлайн, і зробили два локальні коміти. Тим часом ваш колега відправив три коміти в ту саму гілку на віддаленому сервері. Що станеться, якщо ви наосліп виконаєте `git push origin main`, коли знову підключитесь до інтернету?
> 
> *Перевірка здогадки: Віддалений сервер відхилить відправлення. Git розпізнає, що у віддаленій гілці є коміти, яких бракує вашій локальній гілці, що запобіжить ненавмисному перезапису роботи вашого колеги. Ви повинні виконати `git pull`, щоб отримати та інтегрувати їхні зміни у вашу локальну історію, перш ніж ви зможете успішно відправити вашу об'єднану хронологію.*


## Розділ 6: Ігнорування шуму: .gitignore

У будь-якому реальному програмному або інфраструктурному проєкті існує безліч файлів, які ви **ніколи** не захочете додавати до системи контролю версій. До них належать:
- Скомпільовані бінарні файли, виконувані файли або артефакти збірки (наприклад, `.exe`, `.jar`, каталоги `/dist/`).
- Файли журналів (log files), згенеровані вашим додатком під час тестування.
- Приховані файли операційної системи (наприклад, `.DS_Store`, згенерований macOS Finder).
- **Секрети, ключі API, паролі до баз даних та локальні змінні середовища** (наприклад, файли `.env`).

Якщо ви випадково закомітите їх, ви роздуєте розмір репозиторію або, що ще гірше, спричините серйозне порушення безпеки. Щоб сказати Git повністю ігнорувати ці файли і робити вигляд, ніби їх не існує, ви створюєте звичайний текстовий файл з назвою `.gitignore` у кореневій папці вашого проєкту.

Давайте створимо `.gitignore`, адаптований для нашого cloud-native середовища.

```bash
cat << 'EOF' > .gitignore
# Ignore operating system generated files
.DS_Store
Thumbs.db

# Ignore local secret and credential files
.env
secret-keys.yaml
kubeconfig-local

# Ignore terraform state files (if we add Infrastructure as Code later)
*.tfstate
*.tfstate.backup
.terraform/
EOF
```

Git читає цей файл зверху вниз. Будь-який файл, що відповідає шаблону, зазначеному в `.gitignore`, ніколи не з'явиться в `git status` як невідстежуваний (untracked). Це унеможливлює випадкову індексацію за допомогою команди з символом підстановки (wildcard), такої як `git add .`.

### Активне навчання: Зіставлення із шаблонами
> **Зупиніться та подумайте**: Враховуючи наведений вище файл `.gitignore`, який з наступних трьох щойно створених файлів все одно з'явиться як "Untracked" (невідстежуваний), коли ви виконаєте `git status`? 1) `main.tfstate`, 2) `secret-keys.yaml`, 3) `secret-keys.txt`.
>
> *Перевірка здогадки: Лише `secret-keys.txt` з'явиться як невідстежуваний. `main.tfstate` ігнорується за правилом підстановки `*.tfstate`, а `secret-keys.yaml` явно ігнорується за назвою. `secret-keys.txt` не відповідає жодному шаблону ігнорування, тому Git і далі позначатиме його як невідстежуваний файл.*

### Активне навчання: Запізніле ігнорування
> **Зупиніться та подумайте**: У вас є файл із назвою `database-creds.txt`, який ви створили минулого тижня. Ви закомітили його в Git кілька днів тому. Сьогодні ви усвідомлюєте свою помилку і додаєте `database-creds.txt` до вашого файлу `.gitignore`. Ви змінюєте файл з обліковими даними і виконуєте `git status`. Чи проігнорує Git ці зміни?
>
> *Перевірка здогадки: Ні, Git не проігнорує зміни. Файл `.gitignore` лише запобігає додаванню **невідстежуваних** файлів до бази даних. Як тільки файл починає відстежуватися (закомічений), Git продовжуватиме відстежувати його незалежно від того, що вказано в `.gitignore`. Ви повинні явно вилучити його з відстеження за допомогою `git rm --cached database-creds.txt`, перш ніж правило ігнорування почне діяти.*


## Чи знали ви?

1. **Git був створений за два тижні.** У 2005 році спільнота ядра Linux раптово втратила безкоштовну ліцензію на пропрієтарну систему контролю версій. Лінус Торвальдс, оригінальний творець Linux, потребував заміни негайно. Не вражений існуючими варіантами, він написав початкову версію Git всього за 14 днів, і масивна кодова база ядра Linux мігрувала на неї через два місяці.
2. **Назва — це самоіронічна образа.** Торвальдс, відомий своїм різким гумором, назвав систему "Git" (британський сленг, що означає вперту, неприємну або некомпетентну людину). На одній з конференцій він жартома сказав: "Я єгоїстичний виродок, і називаю всі свої проєкти на честь себе. Спочатку Linux, тепер Git".
3. **Git не відстежує порожні каталоги.** Через свою базову архітектуру бази даних, яка зіставляє шляхи безпосередньо з вмістом файлів, Git відстежує лише файли. Якщо ви створите порожній каталог і виконаєте `git status`, Git повністю його проігнорує. Розробники обходять це обмеження, розміщуючи прихований порожній файл (часто за домовленістю названий `.gitkeep`) всередині каталогу, щоб змусити Git відстежувати існування папки.
4. **Колосальна стійкість до колізій.** Хеші, що ідентифікують ваші коміти (рядки SHA-1), є 40-символьними шістнадцятковими числами. Математична ймовірність того, що два різні знімки згенерують абсолютно однаковий хеш (колізія хешів), є астрономічно низькою. Статистично набагато ймовірніше, що у вас влучить блискавка, поки ви одночасно виграватимете в лотерею, ніж те, що у вашому репозиторії станеться випадкова колізія хешу Git.

## Типові помилки

| Помилка | Чому це стається | Як це виправити |
| :--- | :--- | :--- |
| **Випадковий коміт пароля/секрету** | Використання нерозбірливої команди `git add .` без попередньої перевірки `git status`, що випадково затягує файл `.env` до зони підготовки (staging area). | Якщо коміт ще не відправлено на сервер (unpushed): `git reset HEAD~1`, щоб скасувати його локально. Якщо відправлено (pushed), **секрет скомпрометовано**. Ви маєте негайно відкликати/оновити (revoke/rotate) облікові дані в AWS/GCP. Не просто видаляйте їх у новому коміті; історія залишається назавжди. |
| **"fatal: refusing to merge unrelated histories"** | Ви ініціалізували репозиторій локально, а також ініціалізували окремий репозиторій на GitHub зі стандартним README, після чого спробували виконати pull. Git вважає, що це два абсолютно різні, непов'язані проєкти. | Виконайте `git pull origin main --allow-unrelated-histories`, щоб примусово вказати Git об'єднати дві різні історії в одну. |
| **"Updates were rejected because the remote contains work that you do not have locally"** | Колега відправив (pushed) нові коміти на сервер GitHub, поки ви працювали офлайн. Git не дозволяє вам виконати push і перезаписати його роботу. | Спочатку виконайте `git pull`, щоб завантажити та інтегрувати його зміни у вашу локальну гілку. Вирішіть усі можливі конфлікти злиття (merge conflicts), після чого виконайте `git push`. |
| **Порожні або беззмістовні повідомлення комітів** | Поспіх у роботі. Використання розмитих повідомлень на кшталт `git commit -m "update"` або `git commit -m "fixed stuff"`. | Використайте `git commit --amend -m "new better message"`, якщо ви ще не виконали push. Виробіть професійну звичку писати "Чому", а не лише "Що". |
| **Забули додати файли до зони підготовки перед комітом** | Виконання `git commit -m "message"`, коли ваші зміни все ще знаходяться в Робочій директорії (Working Directory), повністю оминаючи Зону підготовки (Staging Area). | Виконайте `git status`, щоб побачити, що не додано до індексу (unstaged). Виконайте `git add <file>`, щоб перемістити зміни до зони підготовки, а потім повторіть команду `git commit`. |
| **Коміт величезних бінарних файлів** | Випадкове додавання скомпільованих артефактів, дампів пам'яті бази даних або великих відео до репозиторію. Git створений для відстеження тексту, а не великих бінарних файлів. | Видаліть файл із бази даних відстеження за допомогою `git rm --cached <file>`, закомітьте видалення та негайно додайте цей тип файлу до вашого `.gitignore`. |


## Контрольні запитання

<details>
<summary>1. Ваша команда щойно розгорнула нову конфігурацію Kubernetes, і кластер одразу впав. Вам потрібно швидко побачити, хто зробив останню зміну та яким було повідомлення коміту. Яку команду ви виконаєте?</summary>
Вам слід виконати `git log` або `git log --oneline`. Ця команда відображає історію комітів у зворотному хронологічному порядку, показуючи автора, мітку часу та повідомлення коміту для кожного знімка стану (snapshot). Аналізуючи цей вивід, ви можете миттєво визначити, хто вніс останні зміни, і прочитати пояснення, чому ця зміна була необхідною. Саме така швидка історична видимість робить контроль версій критично важливим під час інцидентів у продакшені.
</details>

<details>
<summary>2. Ви змінили файл `service.yaml` на своєму ноутбуці, щоб відкрити новий порт. Ви вводите `git commit -m "expose port 8080"`, але Git повертає повідомлення "nothing added to commit but untracked files present". Що ви забули зробити?</summary>
Ви забули перемістити файл із Робочої директорії до Зони підготовки за допомогою команди `git add`. Архітектура Git вимагає від вас явного вибору файлів, які мають бути включені до наступного знімка стану. Оскільки ви оминули Зону підготовки, Git побачив ваш змінений файл, але відмовився автоматично включати його до репозиторію. Ви повинні виконати `git add service.yaml` перед наступною спробою зробити коміт.
</details>

<details>
<summary>3. Ви збираєтесь додати `configMap.yaml` до зони підготовки (stage), але не можете пригадати, чи встановили ви тестовий пароль бази даних, чи залишили там продакшен-пароль. Яку команду слід виконати, щоб перевірити точні рядки, які ви змінили, перед додаванням до індексу?</summary>
Вам слід виконати `git diff`, щоб переглянути точні покрокові (по рядках) зміни. Ця команда порівнює вашу поточну Робочу директорію з останнім збереженим знімком стану в репозиторії. Вона відобразить видалені рядки червоним кольором, а додані — зеленим, дозволяючи об'єктивно перевірити вміст файлу. Покладання на цю команду запобігає випадковому коміту продакшен-секретів до постійного графа історії.
</details>

<details>
<summary>4. Ви щойно приєдналися до нового проєкту. Ви клонуєте репозиторій і виявляєте, що Git не знає ваших `user.name` та `user.email`, тому просить вас виконати `git config` перед комітом. Чому це необхідно і чим це відрізняється від запитів на автентифікацію, які ви можете побачити під час `git push`?</summary>
Git вимагає ідентифікацію автора під час створення коміту, тому `user.name` та `user.email` мають бути налаштовані до того, як `git commit` зможе зафіксувати, хто є автором знімка стану. Ця ідентифікація зберігається в метаданих коміту для підзвітності та відстеження. Натомість запити під час `git push` — це зазвичай перевірки віддаленої автентифікації від сервісу хостингу, такі як помічник облікових даних HTTPS, особистий токен доступу або SSH-ключі. Коротше кажучи: ідентифікація коміту — це локальні метадані для `git commit`; автентифікація під час push доводить, що вам дозволено завантажувати дані на віддалений сервер.
</details>

<details>
<summary>5. Ви створили файл із назвою `aws-credentials.json` на своїй локальній машині для тестування скрипта. Ви хочете, щоб цей файл ніколи не потрапив до репозиторію компанії. Що саме вам слід зробити, щоб гарантувати його ігнорування назавжди?</summary>
Ви маєте створити звичайний текстовий файл із назвою `.gitignore` у корені вашого репозиторію, якщо його ще не існує. Усередині цього файлу потрібно додати точну назву `aws-credentials.json` з нового рядка. Git читає цей конфігураційний файл зверху вниз і автоматично відфільтровуватиме будь-які файли, що збігаються, зі свого радара відстеження. Це гарантує, що файл облікових даних ніколи не буде випадково доданий до індексу чи закомічений, запобігаючи серйозному порушенню безпеки.
</details>

<details>
<summary>6. Молодший розробник намагається "скинути" зламаний проєкт, щоб почати спочатку. Він відкриває свій файловий менеджер, видаляє приховану папку `.git`, а потім вводить `git status` у терміналі, очікуючи, що Git зрозуміє, що проєкт було скинуто, і покаже порожню історію. Що відбувається насправді і чому?</summary>
Коли розробник вводить `git status`, термінал поверне помилку "fatal: not a git repository", оскільки він повністю знищив систему контролю версій для цієї папки. Прихована директорія `.git` — це не просто конфігураційний файл; це фактично вся база даних репозиторію, що містить усі коміти, гілки та історичні знімки стану. Видаливши її, він не скинув історію — він назавжди стер її локально, перетворивши свій проєкт назад на звичайну, невідстежувану папку в операційній системі. Робочі файли на диску залишаються недоторканими, але Git ними більше не керує.
</details>

<details>
<summary>7. Ви та ваш колега одночасно змінюєте той самий файл `nginx-deployment.yaml`. Він виконує push зміни, яка встановлює `replicas: 5`, поки ви офлайн. Ви локально змінюєте цей же рядок на `replicas: 2` і намагаєтесь виконати push. Git відхиляє ваш push, тому ви виконуєте `git pull`. Що відбувається далі і як це вирішити?</summary>
Git спробує автоматично об'єднати (merge) зміни, але оскільки ви обидва змінили один і той самий рядок, він призупиниться та оголосить про конфлікт злиття (merge conflict). Він не може безпечно вгадати, чи повинен Deployment мати 5 реплік, чи 2, тому залишає рішення за людиною. Щоб вирішити конфлікт, ви повинні відкрити файл у своєму редакторі та знайти маркери конфлікту (які виглядають як `<<<<<<< HEAD` та `>>>>>>>`). Після ручного редагування файлу до бажаного стану та видалення маркерів, ви маєте використати `git add` та `git commit`, щоб завершити злиття перед виконанням push.
</details>


## Практична вправа

У цій вправі ви створите локальний репозиторій з нуля, зімітуєте стандартний робочий процес інженера, зробивши кілька логічних комітів, і проаналізуєте отриману історію. Ми використаємо фіктивні конфігураційні файли Kubernetes, щоб зімітувати реальний робочий процес розгортання інфраструктури.

Створіть нову директорію з назвою `dojo-k8s-project` і перейдіть до неї. Ініціалізуйте порожній репозиторій Git.
- [ ] Директорія створена, виконано перехід до неї.
- [ ] Репозиторій Git ініціалізовано.

<details>
<summary>Рішення: Завдання 1</summary>

```bash
mkdir dojo-k8s-project
cd dojo-k8s-project
git init
git status
```
</details>

Створіть файл `README.md` із текстом "# KubeDojo Project". Додайте файл до індексу (stage) та закомітьте його з повідомленням "docs: add initial readme".
- [ ] Файл створено з правильним вмістом.
- [ ] Файл додано до зони підготовки (staging area).
- [ ] Коміт успішно створено.

Створіть файл із назвою `deployment.yaml` і додайте наступний фіктивний вміст:

apiVersion: apps/v1

  name: web

Додайте до індексу та закомітьте цей файл із повідомленням "feat: add web deployment skeleton".
- [ ] Файл створено.
- [ ] Коміт успішно створено з правильним повідомленням.

<details>
<summary>Рішення: Завдання 3</summary>

```bash
cat << 'EOF' > deployment.yaml

git commit -m "feat: add web deployment skeleton"

### Завдання 4: Зміна існуючих файлів
Відкрийте `deployment.yaml` і додайте `replicas: 3` під блоком `spec:`, щоб він виглядав так:

apiVersion: apps/v1

  name: web

  replicas: 3
```
Використайте команду для перегляду точних відмінностей перед додаванням до індексу. Потім додайте зміни до індексу та закомітьте їх із повідомленням "fix: set deployment replicas to 3".
- [ ] Файл змінено.
- [ ] Різницю (diff) успішно переглянуто.
- [ ] Зміни додано до індексу та закомічено.

Виконайте команду для перегляду всієї вашої історії комітів у компактному, однорядковому форматі. Переконайтеся, що всі три ваші коміти присутні у хронологічному порядку.
- [ ] Команду історії виконано.

### Завдання 6: Бонусне випробування — Розширені правила ігнорування
Ви працюєте над новим застосунком, який генерує численні лог-файли з розширенням `.log` у різних директоріях. Ви хочете, щоб Git ігнорував їх усі задля економії місця, але ви повинні гарантувати, що один конкретний файл під назвою `audit-trail.log` у кореневій директорії завжди відстежується з міркувань комплаєнсу.
Створіть файл `.gitignore`, який реалізує саме таку конфігурацію. Перевірте ваше рішення, створивши фіктивні файли (наприклад, `app.log`, `database.log` та `audit-trail.log`) і перевіривши `git status`, щоб переконатися, що лише `audit-trail.log` готовий до відстеження.
- [ ] Файл `.gitignore` створено з відповідними символами підстановки та правилами виключення.
- [ ] Фіктивні файли створено для тестування правил.
- [ ] `git status` підтверджує, що лише потрібний файл є невідстежуваним (untracked).

<details>
<summary>Рішення: Завдання 6</summary>

```bash
# Create the .gitignore file
cat << 'EOF' > .gitignore
*.log
!audit-trail.log
EOF

# Create dummy files
touch app.log
touch database.log
touch audit-trail.log

# Check status
git status
```
Git покаже `.gitignore` та `audit-trail.log` як невідстежувані файли. Інші файли `.log` будуть успішно проігноровані.
</details>
