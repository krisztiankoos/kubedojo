# Модуль 0.6: Основи Git — Відстежуйте свою роботу

**Складність**: [BEGINNER]
**Час на виконання**: 45 хвилин
**Попередні вимоги**: Модуль 0.5 (Редагування файлів)

## Що ви зможете зробити

Після завершення цього модуля ви зможете:
- **Ініціалізувати** локальні Git репозиторії та налаштувати ідентифікаційні дані користувачів для коректного призначення commit.
- **Створити** логічну історію commit шляхом вибіркового stage змін у файлах за допомогою `git add` та `git commit`.
- **Діагностувати** неочікувані стани репозиторію шляхом аналізу виводу команд `git status`, `git log` та `git diff`.
- **Синхронізувати** локальні репозиторії з видаленими серверами за допомогою `git push` та `git pull`.
- **Сформулювати** файл `.gitignore` для запобігання відстеженню чутливих даних або згенерованих артефактів.

## Чому це важливо

Середня за розміром фінансова технологічна компанія використовувала спільний мережевий диск для зберігання файлів конфігурації інфраструктури. Інженери підключали диск, відкривали файли у локальних редакторах, вносили зміни та зберігали їх назад у мережу. Одного вечора в п'ятницю молодшому інженеру доручили оновити тестове середовище. Помилково він відкрив файл `payment-processor-deployment.yaml` для продуктивного середовища замість тестового варіанту. Він змінив тег образу контейнера на експериментальну збірку, зберіг файл і застосував конфігурацію до кластера.

Протягом кількох хвилин система обробки платежів почала відхиляти транзакції. Коли старші інженери поспішили розібратися в ситуації, вони подивилися на файл конфігурації на мережевому диску, але він показував лише поточний, зламаний стан. Оскільки системи контролю версій не було, не залишилося жодного історичного запису про те, як файл виглядав десять хвилин тому, хто його змінив або чому. Команді довелося вручну відновлювати конфігурацію продуктивного середовища по пам’яті та за застарілою документацією. Подальша затримка у відновленні сервісу зупинила всі клієнтські транзакції на шість годин, що призвело до мільйонних збитків і серйозної шкоди репутації.

Системи контролю версій запобігають саме такому сценарію. Git, галузевий стандарт контролю версій, діє як незнищенна машина часу для вашого коду та конфігурації. Він фіксує кожну зміну, ідентифікує, хто саме її вніс, і дозволяє миттєво повернутися до будь-якого попереднього стану. У сучасному Cloud-Native світі інфраструктура визначається як код. Якщо ви не можете відстежувати, перевіряти та скасовувати код вашої інфраструктури з абсолютною впевненістю, ви працюєте в очікуванні катастрофи. Оволодіння Git не є факультативним для Platform Engineer; це фундаментальна навичка, на якій базується вся надійна автоматизація та співпраця.

Без Git не можуть існувати сучасні конвеєри безперервної інтеграції та розгортання (CI/CD). Вся концепція автоматизованого розгортання покладається на єдине довірене джерело істини, яке запускає автоматизацію щоразу, коли виявляється нова, схвалена зміна. Якщо ви не розумієте Git, ви не зможете зрозуміти сучасну доставку програмного забезпечення.

## Розділ 1: Концепція контролю версій та архітектура Git

Перш ніж вводити будь-які команди, ви повинні зрозуміти, як Git сприймає ваші файли. Багато новачків мають труднощі з Git, бо ставляться до нього як до простого інструменту резервного копіювання (як Dropbox або Google Drive). Git зовсім інший. Git не просто безперервно синхронізує файли у фоновому режимі; він робить навмисні, незмінні знімки (snapshots) вашого проекту в певні моменти часу, але лише тоді, коли ви явно даєте йому таку команду.

Уявіть Git як відеогру з ручними точками збереження. Ви можете грати, робити помилки, отримувати пошкодження та заходити у глухий кут. Поки ви явно створюєте «точку збереження» перед небезпечним боєм з босом, ви завжди можете перезавантажити той самий стан, якщо щось піде не так. Git працює так само, але для ваших текстових файлів.

### Централізований проти розподіленого контролю версій

Історично системи контролю версій, такі як Subversion (SVN) або Team Foundation Server (TFS), були централізованими. Існував один головний сервер, який зберігав історію. Якщо ви хотіли переглянути історію або зробити commit, ви повинні були бути підключені до інтернету, щоб зв'язатися з цим сервером. Якщо сервер виходив з ладу, ніхто не міг працювати.

Git — це **розподілена** система контролю версій. Коли ви використовуєте Git, ви не просто завантажуєте останні файли; ви завантажуєте всю історію проекту. Ваш локальний ноутбук стає повноцінним репозиторієм. Ви можете переглядати історію, порівнювати версії та робити commit, перебуваючи повністю офлайн у літаку. Інтернет потрібен лише тоді, коли ви хочете синхронізувати свою історію з кимось іншим.

### Три дерева Git

Git керує вашими файлами, переміщуючи їх через три різні логічні області, які часто називають «деревами». Розуміння цього конвеєра є ключем до діагностики майже будь-якої проблеми з Git.

```text
+---------------------+       +---------------------+       +---------------------+
|                     |       |                     |       |                     |
|  Робоча директорія  | ----> |  Область підготовки | ----> |     Репозиторій     |
|  (Working Directory)|       |   (Staging Area)    |       | (Repository)        |
|                     |       |                     |       |                     |
+---------------------+       +---------------------+       +---------------------+
           |                             |                             |
           |   1. Змінити файли          |                             |
           |---------------------------->|                             |
           |                             |   2. Згрупувати зміни       |
           |                             |---------------------------->|
           |                             |                             |
           |                                                           |
           |<----------------------------------------------------------|
                               3. Відновити старі версії
```

1. **Робоча директорія (Working Directory)**: Це ваш поточний робочий простір на комп'ютері. Вона містить реальні файли, які ви редагуєте, видаляєте або створюєте. Коли ви відкриваєте файл у Vim або VS Code, ви змінюєте саме робочу директорію. Git бачить ці зміни, але ще не зберіг їх.
2. **Область підготовки (Staging Area / Index)**: Це критично важливий проміжний крок, унікальний для Git. Уявіть це як вантажну платформу або сцену для фотосесії. Перед тим як зробити фінальний знімок, ви обираєте, які саме змінені файли розмістити на сцені. Ви можете додати до stage одні файли, залишивши інші. Це дозволяє створювати дуже сфокусовані, логічні commit, навіть якщо ви змінили п'ятдесят файлів одночасно.
3. **Репозиторій (Repository / Commit History)**: Це постійна база даних, де Git зберігає ваші знімки (snapshots), які називаються commit. Після того, як файли зафіксовані (committed) тут, вони надійно записані в історії з іменем автора, часовою міткою та описовим повідомленням. Commit математично запечатаний; його неможливо таємно змінити без зміни його унікального ідентифікатора.

### Зупиніться та подумайте
> **Зупиніться та подумайте**: Ви щойно завершили тривалу сесію налагодження. Ви виправили помилку з'єднання з базою даних у `db.py`, але під час пошуку помилки ви також додали тимчасові інструкції виводу (print) у `auth.py` та `api.py`, які ви не хочете зберігати назавжди. Як архітектура «Трьох дерев» Git дозволяє створити чисту історію в цьому сценарії, не втрачаючи ваш тимчасовий код налагодження у робочій директорії?
>
> *Перевірка передбачення: Архітектура «Трьох дерев» відокремлює ваші робочі файли від того, що зберігається. Ви можете використовувати область підготовки (Staging Area), щоб вибірково додати до stage лише `db.py` для наступного commit. Тимчасові інструкції виводу в `auth.py` та `api.py` залишаються у вашій робочій директорії, щоб ви могли продовжувати їх використовувати або видалити пізніше, ніколи не засмічуючи постійну історію репозиторію.*

## Розділ 2: Підготовка сцени: Встановлення та налаштування

Git — це інструмент командного рядка. Хоча існують графічні інтерфейси, вивчення команд термінала є обов'язковим для Platform Engineering, оскільки ви часто будете використовувати Git на видалених серверах без графічних можливостей. Більшість сучасних дистрибутивів Linux і macOS постачаються з попередньо встановленим Git, або його можна легко додати за допомогою стандартного менеджера пакетів, такого як `apt`, `yum` або `brew`.

Щоб перевірити встановлення, відкрийте термінал і перевірте версію:

```bash
git --version
```

Очікуваний вивід:
```text
git version 2.39.2
```

### Налаштування ідентифікації

Оскільки Git розроблений для спільної роботи тисяч розробників, він відмовляється створювати знімки, доки не знає точно, хто їх робить. Кожен commit потребує імені автора та адреси електронної пошти. Це критично важливо для підзвітності — якщо зміна інфраструктури призведе до збою в продуктивному середовищі, команда повинна знати, у кого запитати про причини такої зміни.

Ви налаштовуєте це за допомогою команди `git config`. Прапор `--global` застосовує ці налаштування до всіх репозиторіїв на вашому поточному комп'ютері, записуючи їх у прихований файл у вашій домашній директорії (`~/.gitconfig`).

```bash
# Встановіть своє ім'я (використовуйте справжнє ім'я, воно з'явиться в історії)
git config --global user.name "Alex Chen"

# Встановіть свою адресу електронної пошти
git config --global user.email "alex.chen@example.com"
```

Ви можете перевірити свою конфігурацію в будь-який час, попросивши Git вивести всі поточні налаштування:

```bash
git config --list
```

### Ініціалізація репозиторію: git init

Щоб наказати Git почати відстежувати проект, ви повинні ініціалізувати репозиторій у кореневій директорії вашого проекту. Давайте створимо нову директорію для гіпотетичного Kubernetes проекту та ініціалізуємо в ній Git.

```bash
# Створіть нову директорію
mkdir k8s-webapp
cd k8s-webapp

# Накажіть Git почати відстежувати цю директорію
git init
```

Очікуваний вивід:
```text
Initialized empty Git repository in /home/alex/k8s-webapp/.git/
```

Що саме зробила команда `git init`? Вона не почала магічним чином сканувати ваш жорсткий диск. Вона просто створила приховану директорію з назвою `.git` всередині вашої папки `k8s-webapp`. Ця прихована директорія `.git` і є справжнім «Репозиторієм» з нашої моделі «Трьох дерев». Саме тут Git зберігає всі внутрішні об'єкти бази даних, стиснутий вміст файлів, граф історії commit та локальну конфігурацію для цього конкретного проекту.

Якщо ви видалите директорію `.git`, ви видалите всю історію версій проекту, хоча ваші поточні робочі файли залишаться недоторканими на диску.

```bash
# Перелічіть усі файли, включаючи приховані
ls -la
```

Очікуваний вивід:
```text
total 12
drwxr-xr-x 3 alex alex 4096 Oct 12 10:00 .
drwxr-xr-x 5 alex alex 4096 Oct 12 09:59 ..
drwxr-xr-x 7 alex alex 4096 Oct 12 10:00 .git
```

Успішно ініціалізувавши репозиторій і налаштувавши свою ідентичність, ви заклали необхідний фундамент. Далі ми дослідимо, як наповнити цей репозиторій, свідомо переміщуючи файли через життєвий цикл Git для створення ваших перших збережених знімків.

## Розділ 3: Життєвий цикл знімка: Add, Commit та Status

Тепер, коли у нас є активний репозиторій, давайте пройдемо через щоденний робочий процес відстеження файлів. Команда, яку ви будете використовувати частіше за будь-яку іншу — сотні разів на день — це `git status`. Вона діє як ваш компас, повідомляючи точно, де знаходяться ваші файли в межах «Трьох дерев».

### Крок 1: Зміна робочої директорії

Давайте створимо наш перший файл, базову конфігурацію Kubernetes Namespace.

```bash
cat << 'EOF' > namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: webapp-prod
EOF
```

Запустіть `git status`, щоб побачити, що Git думає про цей щойно створений файл:

```bash
git status
```

Вивід:
```text
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
	namespace.yaml

nothing added to commit but untracked files present (use "git add" to track)
```

Git розпізнає, що файл існує в робочій директорії, але позначає його як «Untracked» (невідстежуваний). Це означає, що Git ніколи раніше не бачив цей файл. Він не стежить за змінами в ньому і не буде автоматично створювати його резервну копію. Git відстежує лише те, що ви явно наказали йому відстежувати.

### Крок 2: Переміщення в область підготовки (git add)

Щоб повідомити Git, що ми хочемо включити цей файл у наш найперший знімок, ми повинні перемістити його в область підготовки (на вантажну платформу) за допомогою команди `git add`.

```bash
git add namespace.yaml
```

Знову запустіть `git status`, щоб спостерігати за зміною стану:

```bash
git status
```

Вивід:
```text
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
	new file:   namespace.yaml
```

Файл перемістився. Тепер він класифікується як «Changes to be committed» (зміни, які будуть зафіксовані). Він знаходиться на вантажній платформі, чекаючи, поки фотограф зробить знімок.

### Крок 3: Створення знімка (git commit)

Щоб назавжди зберегти підготовлені зміни в репозиторії, ми використовуємо `git commit`. Кожен commit потребує повідомлення, яке пояснює, *чому* була зроблена зміна. Написання якісних повідомлень commit — це ключова професійна навичка. Гарне повідомлення пояснює намір і контекст, а не просто те, які рядки змінилися.

```bash
git commit -m "feat: add production namespace definition"
```

Вивід:
```text
[main (root-commit) a1b2c3d] feat: add production namespace definition
 1 file changed, 4 insertions(+)
 create mode 100644 namespace.yaml
```

Давайте перевіримо `git status` ще раз, щоб побачити фінальний результат нашого робочого процесу:

```bash
git status
```

Вивід:
```text
On branch main
nothing to commit, working tree clean
```

Ваша робоча директорія тепер описується як «clean» (чиста). Це означає, що кожен файл, який зараз знаходиться на вашому диску, точно відповідає останньому знімку, збереженому в базі даних `.git`. Немає жодних очікуваних змін.

### Повчальна історія: Випадковий commit секретів

Молодший розробник тестував додаток локально, для якого був потрібен ключ доступу AWS. Для зручності він прописав ключ безпосередньо у своєму файлі `deployment.yaml`, просто щоб подивитися, чи запустяться Pod. Це спрацювало. Зрадівши, він запустив `git add .` (команду, яка без розбору додає до stage кожен змінений файл у всій директорії), а потім виконав `git commit -m "fix deployment"`.

Потім він надіслав (push) код у публічний репозиторій GitHub. Протягом 120 секунд автоматизовані боти безпеки, що сканують публічні репозиторії GitHub, знайшли відкритий ключ AWS. Боти миттєво запустили сотні потужних серверів для майнінгу криптовалюти в кількох регіонах AWS, використовуючи скомпрометовані дані. До того часу, як розробник прокинувся наступного ранку, компанія отримала рахунок від хмарного провайдера на суму 65 000 доларів.

**Урок**: Ніколи сліпо не використовуйте `git add .`, якщо ви не впевнені на сто відсотків, що саме ви змінили. Завжди запускайте `git status` та `git diff` перед додаванням до stage, щоб переконатися, що ви випадково не фіксуєте паролі, API-ключі, приватні SSH-ключі або тимчасові файли налагодження.

Тепер, коли ви знаєте, як безпечно створювати знімки, вам неминуче знадобиться переглядати минулі знімки або вивчати точні зміни, внесені до окремих файлів з часом. У наступному розділі ми заглибимося в команди, які дозволяють аналізувати історію commit та вивчати точні відмінності у файлах.

## Розділ 4: Подорож у часі: Log та Diff

Після того, як ви зробили кілька commit, вам знадобляться надійні способи перегляду часової шкали історії та розуміння того, що саме змінилося між різними моментами часу.

Давайте внесемо ще одну зміну в наш проект. Ми оновимо файл Namespace, щоб додати мітку (label) — поширене завдання в Kubernetes для цілей організації.

```bash
cat << 'EOF' > namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: webapp-prod
  labels:
    environment: production
EOF
```

### Перегляд змін: git diff

Перед тим як додати файл до stage або зафіксувати його, ви завжди повинні перевірити, які саме рядки ви змінили. Пам'ять може підвести; diff об'єктивний. Команда `git diff` порівнює вашу поточну робочу директорію з останнім знімком, який ви зробили.

```bash
git diff
```

Вивід:
```text
diff --git a/namespace.yaml b/namespace.yaml
index e46b825..8394c41 100644
--- a/namespace.yaml
+++ b/namespace.yaml
@@ -2,3 +2,5 @@ apiVersion: v1
 kind: Namespace
 metadata:
   name: webapp-prod
+  labels:
+    environment: production
```

**Як розшифрувати вивід diff:**
- Заголовки `--- a/namespace.yaml` та `+++ b/namespace.yaml` показують дві версії файлів, що порівнюються (стара проти нової).
- `@@ -2,3 +2,5 @@` — це заголовок фрагмента (chunk). Він надає системі контекст про те, приблизно в якому місці файлу відбулися зміни.
- Рядки, що починаються з пробілу, — це незмінені контекстні рядки. Git показує їх, щоб допомогти вам зорієнтуватися.
- Рядки, що починаються з `+` (зазвичай підсвічуються зеленим), — це абсолютно нові додавання.
- Рядки, що починаються з `-` (зазвичай підсвічуються червоним), — це видалення. Якщо ви змінили рядок, Git представляє це як видалення старого рядка та додавання нового.

### Зупиніться та подумайте
> **Зупиніться та подумайте**: Уявіть, що ви запустили `git diff` для файлу Deployment і побачили наступний вивід:
>
> ```text
> @@ -10,3 +10,3 @@
>  spec:
>    replicas: 3
> -  image: nginx:1.14
> +  image: nginx:1.24
> ```
>
> Перед тим як читати далі, що саме зробив інженер у цьому файлі? Будьте конкретними.
>
> *Перевірка передбачення: Інженер не додав абсолютно новий структурний елемент. Він змінив існуючий рядок. Він видалив рядок, що вказував версію контейнера Nginx 1.14, і замінив його рядком із версією 1.24. Це означає оновлення версії образу контейнера.*

Тепер, коли ми перевірили, що наші зміни правильні і не містять секретів, давайте додамо до stage та зафіксуємо наше додавання мітки:

```bash
git add namespace.yaml
git commit -m "chore: add environment label to namespace"
```

### Перегляд історії: git log

Щоб побачити часову шкалу ваших знімків, використовуйте команду `git log`. Вона відкриває програму перегляду (зазвичай `less`), показуючи вашу історію у зворотному хронологічному порядку (найновіші спочатку).

```bash
git log
```

Вивід:
```text
commit 8f9e0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f (HEAD -> main)
Author: Alex Chen <alex.chen@example.com>
Date:   Wed Oct 12 10:45:12 2023 -0400

    chore: add environment label to namespace

commit a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8g9h
Author: Alex Chen <alex.chen@example.com>
Date:   Wed Oct 12 10:15:30 2023 -0400

    feat: add production namespace definition
```

Зверніть увагу на довгий рядок літер і цифр (наприклад, `8f9e0a1b2...`). Це хеш commit (Commit Hash, конкретно хеш SHA-1). Це математично згенерований унікальний ідентифікатор для цього конкретного знімка. Він охоплює вміст файлів, автора, дату та хеш попереднього commit. Ви використовуєте цей хеш, якщо вам коли-небудь знадобиться повернути ваш проект саме до цього моменту часу.

Для більш компактного вигляду, що особливо корисно при дослідженні репозиторію з тисячами commit за кілька років, використовуйте прапор `--oneline`:

```bash
git log --oneline
```

Вивід:
```text
8f9e0a1 chore: add environment label to namespace
a1b2c3d feat: add production namespace definition
```

Ви також можете використовувати `git log -p`, щоб побачити реальні diff, внесені кожним окремим commit в історії. Це дозволяє побачити не лише те, *що* commit відбувся, але й те, які саме *рядки* були змінені.

Оволодіння здатністю орієнтуватися у вашій локальній часовій шкалі надає величезної впевнесті під час експериментів з кодом інфраструктури. Однак сучасна інженерія — це командна робота; наступний розділ розповість, як безпечно ділитися вашою локальною історією з видаленими серверами для безперешкодної співпраці з іншими.

## Розділ 5: Спільна робота зі світом: Remotes, Push та Pull

До цього моменту все, що ми робили, існувало лише на жорсткому диску вашого локального ноутбука. Якщо ваш комп'ютер зламається або ви проллєте на нього каву, вся історія вашого проекту зникне назавжди. Крім того, ніхто інший у вашій команді не може бачити вашу роботу.

Щоб співпрацювати з іншими та створювати резервні копії своєї роботи, ви повинні підключити свій локальний репозиторій до видаленого (remote) сервера, такого як GitHub, GitLab або Bitbucket.

### Що таке Remote?

«Remote» — це просто інший репозиторій Git, розміщений у мережі або в інтернеті. Оскільки Git розподілений, кожен має повну, автономну копію історії. «Pushing» (надсилання) та «Pulling» (отримання) — це явні дії, які ми виконуємо для синхронізації нашої локальної часової шкали з видаленою.

```text
+-----------------------+                    +-----------------------+
|                       |                    |                       |
|  Локальний репозиторій |                    | Видалений репозиторій |
|     (Ваш ноутбук)     |                    |    (GitHub/GitLab)    |
|                       |                    |                       |
|  commit C (HEAD)      | ==== git push ===> |  commit C             |
|  commit B             |                    |  commit B             |
|  commit A             | <=== git pull ==== |  commit A             |
|                       |                    |                       |
+-----------------------+                    +-----------------------+
```

### Підключення до Remote

Коли ви створюєте порожній репозиторій на такій платформі, як GitHub, платформа надає вам URL-адресу для підключення (зазвичай HTTPS або SSH). Ви наказуєте своєму локальному Git підключитися до цієї URL-адреси за допомогою команди `git remote add`.

За галузевою угодою, основний видалений сервер за замовчуванням майже завжди називається `origin`.

```bash
# Приклад команди (не запускайте, якщо у вас немає реальної URL-адреси репозиторію)
git remote add origin https://github.com/yourusername/k8s-webapp.git
```

Якщо ви приєднуєтеся до компанії і хочете завантажити існуючий проект, ви не використовуєте `git init` та `git remote add`. Замість цього ви використовуєте `git clone <url>`. Клонування автоматично ініціалізує локальний репозиторій, додає видалений сервер як `origin` і завантажує всю історію та файли за один крок.

### Надсилання змін (git push)

Щоб завантажити ваші локально створені commit на видалений сервер, ви їх надсилаєте (push). Ви повинні вказати назву видаленого сервера (зазвичай `origin`) та назву гілки (branch), яку ви надсилаєте (часто `main` або `master`).

```bash
# Надішліть вашу гілку main на сервер origin вперше
git push -u origin main
```

Прапор `-u` означає «upstream» (вгору по течії). Вам потрібно використовувати його лише під час першого надсилання нової гілки. Він створює постійний зв'язок між вашою локальною гілкою `main` та видаленою гілкою `main`. У майбутньому ви зможете просто вводити `git push` без жодних аргументів, і Git знатиме точно, куди відправляти дані.

### Отримання змін (git pull)

Якщо колега вносить зміни, фіксує їх локально і надсилає на GitHub, ваш локальний репозиторій *не* оновиться автоматично. Git ніколи не змінить ваші локальні файли без вашого явного дозволу. Ви повинні самі звернутися до сервера, завантажити їхні нові commit та інтегрувати їх у свою локальну історію за допомогою `git pull`.

```bash
# Отримайте зміни з сервера та об'єднайте їх зі своєю локальною гілкою
git pull origin main
```

Насправді `git pull` — це макрос, який послідовно запускає дві різні команди: `git fetch` (яка безпечно завантажує нові commit з сервера, не змінюючи ваші робочі файли) та `git merge` (яка намагається безперешкодно об'єднати завантажені зміни з вашою поточною робочою директорією).

### Зупиніться та подумайте
> **Зупиніться та подумайте**: Ви провели ранок працюючи офлайн і зробили два локальних commit. Тим часом ваш колега надіслав три commit у ту саму гілку на видаленому сервері. Що станеться, якщо ви сліпо запустите `git push origin main`, коли знову підключитеся до інтернету?
>
> *Перевірка передбачення: Надсилання буде відхилено видаленим сервером. Git розпізнає, що видалена гілка має commit, яких немає у вашій локальній гілці, що запобігає ненавмисному перезапису роботи вашого колеги. Ви повинні запустити `git pull`, щоб отримати та інтегрувати їхні зміни у свою локальну історію, перш ніж ви зможете успішно надіслати вашу об'єднану часову шкалу.*

## Розділ 6: Ігнорування шуму: .gitignore

У будь-якому реальному проекті розробки програмного забезпечення або інфраструктури існує безліч файлів, які ви **ніколи** не захочете фіксувати в системі контролю версій. До них належать:
- Скомпільовані бінарні файли, виконувані файли або артефакти збірки (наприклад, `.exe`, `.jar`, директорії `/dist/`).
- Лог-файли, згенеровані вашим додатком під час тестування.
- Приховані файли операційної системи (наприклад, `.DS_Store`, згенерований macOS Finder).
- **Секрети, API-ключі, паролі баз даних та локальні змінні оточення** (наприклад, файли `.env`).

Якщо ви випадково зафіксуєте їх, ви роздуєте розмір репозиторію або, що гірше, спричините серйозне порушення безпеки. Щоб наказати Git вдавати, що цих файлів взагалі не існує, ви створюєте звичайний текстовий файл з назвою `.gitignore` у кореневій папці вашого проекту.

Давайте створимо `.gitignore`, адаптований для нашого Cloud-Native середовища.

```bash
cat << 'EOF' > .gitignore
# Ігнорувати файли, згенеровані операційною системою
.DS_Store
Thumbs.db

# Ігнорувати локальні секрети та файли з обліковими даними
.env
secret-keys.yaml
kubeconfig-local

# Ігнорувати файли стану Terraform (якщо ми додамо IaC пізніше)
*.tfstate
*.tfstate.backup
.terraform/
EOF
```

Git читає цей файл зверху вниз. Будь-який файл, що відповідає шаблону, вказаному в `.gitignore`, ніколи не з'явиться в `git status` як невідстежуваний. Це унеможливлює його випадкове додавання до stage за допомогою команд із підстановочними знаками, як-от `git add .`.

### Зупиніться та подумайте
> **Зупиніться та подумайте**: Враховуючи наведений вище файл `.gitignore`, який із трьох наступних щойно створених файлів все одно відображатиметься як «Untracked», коли ви запустите `git status`? 1) `main.tfstate`, 2) `secret-keys.yaml`, 3) `secret-keys.txt`.
>
> *Перевірка передбачення: Лише `secret-keys.txt` відображатиметься як невідстежуваний. `main.tfstate` ігнорується правилом `*.tfstate`, а `secret-keys.yaml` явно ігнорується за назвою. `secret-keys.txt` не відповідає жодному шаблону ігнорування, тому Git продовжуватиме позначати його як невідстежуваний файл.*

### Зупиніться та подумайте
> **Зупиніться та подумайте**: У вас є файл з назвою `database-creds.txt`, який ви створили минулого тижня. Ви зафіксували його в Git кілька днів тому. Сьогодні ви усвідомлюєте свою помилку і додаєте `database-creds.txt` до свого файлу `.gitignore`. Ви змінюєте файл з обліковими даними і запускаєте `git status`. Чи проігнорує Git ці зміни?
>
> *Перевірка передбачення: Ні, Git не проігнорує зміни. Файл `.gitignore` запобігає додаванню до бази даних лише **невідстежуваних** файлів. Якщо файл уже відстежується (зафіксований), Git продовжуватиме відстежувати його незалежно від того, що написано в `.gitignore`. Ви повинні явно видалити його з відстеження за допомогою команди `git rm --cached database-creds.txt`, перш ніж правило ігнорування почне діяти.*

## Чи знали ви?

1. **Git був створений за два тижні.** У 2005 році спільнота розробників ядра Linux раптово втратила безкоштовну ліцензію на власницьку систему контролю версій. Лінус Торвальдс, творець Linux, потребував заміни негайно. Не задоволений існуючими варіантами, він написав початкову версію Git всього за 14 днів, а через два місяці на неї перейшов величезний код ядра Linux.
2. **Назва — це самопринизлива образа.** Торвальдс, відомий своїм різким гумором, назвав систему «Git» (британський сленг для впертої, неприємної або некомпетентної людини). Він відомо пожартував на конференції: «Я егоїстичний виродок, і я називаю всі свої проекти на свою честь. Спочатку Linux, тепер Git».
3. **Git не відстежує порожні директорії.** Через архітектуру своєї бази даних, яка пов'язує шляхи безпосередньо з вмістом файлів, Git відстежує лише файли. Якщо ви створите порожню директорію і запустите `git status`, Git повністю її проігнорує. Розробники обходять це обмеження, розміщуючи прихований порожній файл (часто за домовленістю з назвою `.gitkeep`) всередині директорії, щоб змусити Git відстежувати факт існування папки.
4. **Колосальна стійкість до колізій.** Хеші, що ідентифікують ваші commit (рядки SHA-1), — це 40-символьні шістнадцяткові числа. Математична ймовірність того, що два різні знімки згенерують однаковий хеш (колізія хешів), астрономічно низька. Статистично у вас набагато більше шансів отримати удар блискавкою, одночасно вигравши в лотерею, ніж зіткнутися з випадковою колізією хешів Git у вашому репозиторії.

## Типові помилки

| Помилка | Чому це трапляється | Як це виправити |
| :--- | :--- | :--- |
| **Випадкова фіксація пароля/секрету** | Використання нерозбірливої команди `git add .` без попередньої перевірки `git status`, випадкове додавання файлу `.env` в область підготовки. | Якщо зміни не були надіслані (push): `git reset HEAD~1`, щоб скасувати commit локально. Якщо надіслані — **секрет скомпрометовано**. Ви повинні негайно відкликати/змінити облікові дані. Не просто видаліть їх у новому commit; історія незмінна. |
| **"fatal: refusing to merge unrelated histories"** | Ви ініціалізували репозиторій локально і окремо ініціалізували репозиторій на GitHub із типовим README, а потім спробували виконати pull. Git вважає їх двома абсолютно різними проектами. | Запустіть `git pull origin main --allow-unrelated-histories`, щоб змусити Git об'єднати дві окремі часові шкали в одну. |
| **"Updates were rejected because the remote contains work that you do not have locally"** | Колега надіслав нові commit на сервер GitHub, поки ви працювали офлайн. Git відмовляється дозволити вам push, щоб не перезаписати їхню роботу. | Спочатку запустіть `git pull`, щоб завантажити та інтегрувати їхні зміни у свою локальну гілку. Вирішіть потенційні конфлікти злиття, а потім запустіть `git push`. |
| **Порожні або безглузді повідомлення commit** | Поспіх. Використання розпливчастих повідомлень на кшталт `git commit -m "update"` або `git commit -m "fixed stuff"`. | Використовуйте `git commit --amend -m "нове краще повідомлення"`, якщо ви ще не робили push. Виробіть професійну звичку писати «Чому», а не лише «Що». |
| **Забування додати файли до stage перед фіксацією** | Запуск `git commit -m "message"`, коли ваші зміни все ще знаходяться в робочій директорії, повністю минаючи область підготовки. | Запустіть `git status`, щоб побачити, що не додано до stage. Виконайте `git add <file>`, щоб перемістити зміни на сцену, а потім повторіть команду `git commit`. |
| **Фіксація величезних бінарних файлів** | Випадкове додавання скомпільованих артефактів, дампів пам'яті баз даних або великих відео в репозиторій. Git розроблений для відстеження тексту, а не великих бінарних файлів. | Видаліть файл із бази відстеження за допомогою `git rm --cached <file>`, зафіксуйте видалення і негайно додайте цей тип файлів у свій `.gitignore`. |

## Контрольні запитання

<details>
<summary>1. Ваша команда щойно розгорнула нову конфігурацію Kubernetes, і кластер негайно вийшов з ладу. Вам потрібно швидко подивитися, хто вніс останню зміну і яким було їхнє повідомлення commit. Яку команду ви запустите?</summary>
Ви повинні запустити `git log` або `git log --oneline`. Ця команда відображає історію commit у зворотному хронологічному порядку, показуючи автора, часову мітку та повідомлення commit для кожного знімка. Аналізуючи цей вивід, ви можете миттєво визначити, хто вніс останні зміни, і прочитати їхнє пояснення того, чому ця зміна була необхідною. Така швидка видимість історії — саме те, чому контроль версій є критично важливим під час інцидентів у продуктивному середовищі.
</details>

<details>
<summary>2. Ви змінили файл `service.yaml` на своєму ноутбуці, щоб відкрити новий порт. Ви вводите `git commit -m "expose port 8080"`, але Git повертає повідомлення: "nothing added to commit but untracked files present". Що ви забули зробити?</summary>
Ви забули перемістити файл із робочої директорії в область підготовки за допомогою команди `git add`. Архітектура Git вимагає, щоб ви явно вибирали, які файли мають бути включені в наступний знімок. Оскільки ви проігнорували область підготовки, Git побачив ваш змінений файл, але відмовився автоматично включати його в репозиторій. Ви повинні запустити `git add service.yaml`, перш ніж намагатися зробити commit знову.
</details>

<details>
<summary>3. Ви збираєтеся додати до stage файл `configMap.yaml`, але не можете згадати, чи встановили ви пароль для тестування, чи залишили там пароль для продуктивного середовища. Яку команду слід запустити, щоб перевірити точні рядки, які ви змінили перед додаванням до stage?</summary>
Ви повинні запустити `git diff`, щоб переглянути точні рядкові зміни. Ця команда порівнює вашу поточну робочу директорію з останнім збереженим знімком у репозиторії. Вона покаже видалені рядки червоним кольором, а додані — зеленим, що дозволить вам об'єктивно перевірити вміст файлу. Використання цієї команди запобігає випадковій фіксації секретів продуктивного середовища у постійному графі історії.
</details>

<details>
<summary>4. Ви щойно приєдналися до нового проекту. Ви клонуєте репозиторій, але коли ви запускаєте `git push`, термінал запитує ваше ім'я користувача та електронну пошту і просить запустити `git config`. Чому це необхідно?</summary>
Git вимагає ідентифікації автора для кожного commit, щоб забезпечити підзвітність і простежуваність у середовищах спільної розробки. Без налаштування ваших `user.name` та `user.email` Git відмовляється створювати знімки, оскільки не може приписати зміни конкретній людині. Ця інформація про особу назавжди вбудовується в криптографічний хеш commit. У разі збою інфраструктури ця атрибуція дозволяє команді негайно дізнатися, до кого звернутися за контекстом щодо зміни.
</details>

<details>
<summary>5. Ви створили файл з назвою `aws-credentials.json` на своїй локальній машині для тестування скрипта. Ви ніколи не хочете, щоб цей файл потрапив до репозиторію компанії. Що саме ви повинні зробити, щоб він ігнорувався назавжди?</summary>
Ви повинні створити звичайний текстовий файл з назвою `.gitignore` у корені вашого репозиторію, якщо він ще не існує. Всередині цього файлу потрібно додати точну назву файлу `aws-credentials.json` з нового рядка. Git читає цей файл конфігурації зверху вниз і автоматично відфільтровуватиме будь-які відповідні файли зі свого радара відстеження. Це гарантує, що файл із обліковими даними ніколи випадково не буде доданий до stage або зафіксований, запобігаючи серйозному порушенню безпеки.
</details>

<details>
<summary>6. Молодший розробник намагається «скинути» (reset) зламаний проект, щоб почати спочатку. Він відкриває провідник файлів, видаляє приховану папку `.git`, а потім вводить `git status` у терміналі, очікуючи, що Git зрозуміє, що проект був скинутий, і покаже порожню історію. Що насправді станеться і чому?</summary>
Коли розробник введе `git status`, термінал поверне помилку «fatal: not a git repository», оскільки він повністю знищив систему контролю версій для цієї папки. Прихована директорія `.git` — це не просто файл конфігурації; це вся фактична база даних репозиторію, що містить усі commit, гілки та історичні знімки. Видаливши її, він не скинув історію — він назавжди стер її локально, перетворивши свій проект назад на звичайну, невідстежувану папку в операційній системі. Робочі файли на диску залишаться недоторканими, але Git більше не керуватиме ними.
</details>

<details>
<summary>7. Ви та ваш колега одночасно змінюєте один і той самий файл `nginx-deployment.yaml`. Він надсилає (push) зміну, яка встановлює `replicas: 5`, поки ви перебуваєте офлайн. Ви локально змінюєте той самий рядок на `replicas: 2` і намагаєтеся зробити push. Git відхиляє ваш запит, тому ви запускаєте `git pull`. Що станеться далі і як це вирішити?</summary>
Git спробує автоматично об'єднати зміни, але оскільки ви обоє змінили один і той самий рядок, він зупиниться і оголосить конфлікт злиття (merge conflict). Він не може самостійно вгадати, чи має розгортання мати 5 реплік, чи 2, тому залишає рішення людині. Щоб вирішити конфлікт, ви повинні відкрити файл у редакторі та знайти маркери конфлікту (які виглядають як `<<<<<<< HEAD` та `>>>>>>>`). Після того, як ви вручну відредагуєте файл до бажаного стану та видалите маркери, ви повинні використати `git add` та `git commit`, щоб завершити злиття перед тим, як робити push.
</details>

## Практична вправа

У цій вправі ви створите локальний репозиторій з нуля, змоделюєте стандартний інженерний робочий процес, зробивши кілька логічних commit, і оглянете отриману історію. Ми будемо використовувати фіктивні файли конфігурації Kubernetes для імітації реального процесу роботи з інфраструктурою.

### Завдання 1: Ініціалізація
Створіть нову директорію з назвою `dojo-k8s-project` і перейдіть у неї. Ініціалізуйте порожній репозиторій Git.
- [ ] Директорія створена, виконано перехід у неї.
- [ ] Репозиторій Git ініціалізовано.

<details>
<summary>Рішення: Завдання 1</summary>

```bash
mkdir dojo-k8s-project
cd dojo-k8s-project
git init
```
</details>

### Завдання 2: Початковий commit
Створіть файл `README.md` із текстом "# KubeDojo Project". Додайте файл до stage та зафіксуйте його з повідомленням "docs: add initial readme".
- [ ] Файл створено з правильним вмістом.
- [ ] Файл додано до області підготовки.
- [ ] Commit успішно створено.

<details>
<summary>Рішення: Завдання 2</summary>

```bash
echo "# KubeDojo Project" > README.md
git add README.md
git commit -m "docs: add initial readme"
```
</details>

### Завдання 3: Моделювання розробки інфраструктури
Створіть файл з назвою `deployment.yaml` і додайте наступний фіктивний вміст:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
```
Додайте до stage та зафіксуйте цей файл із повідомленням "feat: add web deployment skeleton".
- [ ] Файл створено.
- [ ] Commit успішно створено з правильним повідомленням.

<details>
<summary>Рішення: Завдання 3</summary>

```bash
cat << 'EOF' > deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
EOF

git add deployment.yaml
git commit -m "feat: add web deployment skeleton"
```
</details>

### Завдання 4: Зміна існуючих файлів
Відкрийте `deployment.yaml` і додайте `replicas: 3` у блоці `spec:`, щоб він виглядав так:
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
```
Використовуйте команду для перегляду точних відмінностей перед додаванням до stage. Потім додайте до stage та зафіксуйте зміну з повідомленням "fix: set deployment replicas to 3".
- [ ] Файл змінено.
- [ ] Диференціал (diff) успішно переглянуто.
- [ ] Зміна додана до stage та зафіксована.

<details>
<summary>Рішення: Завдання 4</summary>

```bash
# Змініть файл за допомогою вашого улюбленого редактора (nano, vim або cat)
cat << 'EOF' > deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
EOF

# Перегляд diff
git diff

# Додавання до stage та commit
git add deployment.yaml
git commit -m "fix: set deployment replicas to 3"
```
</details>

### Завдання 5: Огляд часової шкали
Запустіть команду для перегляду повної історії commit у компактному однорядковому форматі. Переконайтеся, що всі три ваші commit присутні в хронологічному порядку.
- [ ] Команда перегляду історії виконана.
- [ ] Три окремі commit видимі у виводі.

<details>
<summary>Рішення: Завдання 5</summary>

```bash
git log --oneline
```
Вивід має виглядати приблизно так:
```text
3b2a1c4 fix: set deployment replicas to 3
9f8e7d6 feat: add web deployment skeleton
1a2b3c4 docs: add initial readme
```
</details>

### Завдання 6: Бонусне завдання — Розширені правила ігнорування
Ви працюєте над новим додатком, який генерує численні лог-файли з розширенням `.log` у різних директоріях. Ви хочете, щоб Git ігнорував їх усі для економії місця, але ви повинні переконатися, що один конкретний файл з назвою `audit-trail.log` у кореневій директорії завжди відстежується з міркувань відповідності вимогам (compliance).
Створіть файл `.gitignore`, який реалізує саме таку конфігурацію. Перевірте своє рішення, створивши фіктивні файли (наприклад, `app.log`, `database.log` та `audit-trail.log`) і перевіривши `git status`, щоб переконатися, що лише `audit-trail.log` готовий до відстеження.
- [ ] Файл `.gitignore` створено з відповідними шаблонами та правилами виключення.
- [ ] Створено фіктивні файли для тестування правил.
- [ ] `git status` підтверджує, що лише потрібний файл є невідстежуваним.

<details>
<summary>Рішення: Завдання 6</summary>

```bash
# Створіть файл .gitignore
cat << 'EOF' > .gitignore
*.log
!audit-trail.log
EOF

# Створіть фіктивні файли
touch app.log
touch database.log
touch audit-trail.log

# Перевірте статус
git status
```
Git покаже `.gitignore` та `audit-trail.log` як невідстежувані файли. Інші файли `.log` будуть успішно проігноровані.
</details>
