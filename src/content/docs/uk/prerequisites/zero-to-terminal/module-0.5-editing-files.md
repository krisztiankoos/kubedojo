## Що ви зможете зробити

Після цього модуля ви зможете:
- **Створювати та редагувати** файли за допомогою `nano` безпосередньо в терміналі без графічного редактора
- **Написати** простий bash-скрипт, який об'єднує кілька команд в одному файлі
- **Зробити** скрипт виконуваним за допомогою `chmod +x` та пояснити, чому цей крок є необхідним
- **Обирати** між `nano` та `vim` і пояснити, коли використовувати кожен із них

---

## Чому це важливо

У попередньому модулі ви створювали файли за допомогою `touch`, але вони були порожніми. Порожні файли — це як чисті бланки замовлень на кухні ресторану. Корисно, щоб забронювати місце, але від них мало користі, поки хтось у них не запише.

У реальному світі вам доведеться постійно редагувати файли:
- **Конфігураційні файли** вказують програмам, як поводитися (як правила внутрішнього розпорядку на кухні)
- **Скрипти** — це файли, що містять послідовність команд (як картка з рецептом)
- **Маніфести Kubernetes** — це файли, які кажуть Kubernetes, що саме запускати (ви до цього ще дійдете!)

Згадайте реальний випадок у великій технологічній компанії: молодшому інженеру потрібно було оновити просте налаштування в балансувальнику навантаження. Він відкрив файл `nginx.conf` у `nano`, випадково ввів кілька зайвих символів під час спроби збереження і зберіг файл. За лічені хвилини весь сайт «ліг», що коштувало компанії тисячі доларів втраченого доходу. Чому? Тому що він редагував файл безпосередньо на робочому (production) сервері, не усвідомлюючи цього, а одна друкарська помилка в конфігураційному файлі може вивести сервіс із ладу.

Ви повинні вміти відкрити файл, написати в ньому щось, зберегти та закрити — і все це з термінала. Без миші. Без графічного текстового редактора. Тільки ви та клавіатура. Вміння безпечно редагувати файли в терміналі — це не просто про написання тексту; це про точну роботу в «центрі керування» вашими серверами.

## Навіщо редагувати файли в терміналі?

«Хіба я не можу просто використовувати Блокнот (Notepad) або TextEdit?»

Можете — на власному комп’ютері. Але згадайте Модуль 0.1: багато серверів і хмарних середовищ працюють на Linux, а сервери зазвичай не мають графічного інтерфейсу. Коли ви підключаєтеся до віддаленого сервера (про що ви дізнаєтеся в Модулі 0.7), там немає миші, робочого столу чи Блокнота. Є тільки термінал.

Текстовий редактор у терміналі — це як кухарський ніж у світі комп’ютерів. Це не найвишуканіший інструмент, але ви будете використовувати його всюди.

---

## Знайомтеся з nano: вашим першим редактором у терміналі

Існує багато текстових редакторів для термінала. Два найвідоміші — це **vim** та **nano**.

Ми почнемо з **nano**, тому що:

| nano | vim |
|------|-----|
| Працює так, як ви очікуєте | Має круту криву навчання |
| Просто друкуйте — і текст з'являється | Потрібно натиснути `i`, перш ніж зможете друкувати |
| Меню знизу показує, як зберегти та вийти | Відомий тим, що люди «застрягають» і не знають, як вийти |
| Ідеально для новачків | Потужний, але спочатку приголомшує |

Використовувати nano зовсім не соромно. Багато досвідчених інженерів використовують його для швидких правок. Ви вивчите vim пізніше, коли будете готові до більшої потужності.

> **Цікавий факт**: «Як вийти з vim» стало бородатим жартом серед програмістів, тому що багато новачків відкривають vim, починають друкувати, а потім не знають команди для виходу. У nano комбінації клавіш для збереження та виходу відображаються на екрані з самого початку.

## Відкриття nano

Давайте створимо та відредагуємо файл. Спочатку переконайтеся, що ви перебуваєте у своїй домашній директорії:

```bash
cd ~
```

Тепер відкрийте nano з новим файлом:

```bash
nano hello.txt
```

Ваш екран повністю зміниться. Ви побачите щось на кшталт цього:

```
  GNU nano 9.0                    hello.txt







^G Help    ^O Write Out  ^W Where Is   ^K Cut       ^C Location
^X Exit    ^R Read File  ^\ Replace    ^U Paste     ^T Execute
```

Давайте розберемо це:

- **Верхній рядок**: Назва редактора та назва вашого файлу
- **Середня область**: Тут ви вводите текст (зараз вона порожня, бо файл новий)
- **Нижні два рядки**: Меню, що показує доступні команди

---

## Символ ^ означає Ctrl

Це те, що спочатку плутає всіх:

> **Символ `^` означає «утримуйте клавішу Ctrl».**

Тож коли ви бачите `^O Write Out`, це означає: «Натисніть Ctrl та O одночасно, щоб зберегти файл».

| Те, що ви бачите | Що потрібно натиснути | Що це робить |
|------------------|-----------------------|--------------|
| `^O` | Ctrl + O | Зберегти файл |
| `^X` | Ctrl + X | Вийти з nano |
| `^K` | Ctrl + K | Вирізати поточний рядок |
| `^U` | Ctrl + U | Вставити вирізаний рядок |
| `^W` | Ctrl + W | Пошук тексту |
| `^G` | Ctrl + G | Показати допомогу |

Ось і все. Шість комбінацій клавіш — і ви можете робити все необхідне.

---

## Введення тексту

Це найпростіша частина: **просто друкуйте.**

Відкривши nano, введіть наступне (натискайте Enter в кінці кожного рядка):

```
Welcome to the Kitchen!
Today's special: Learning to edit files.
Chef says: You're doing great.
```

Тепер на екрані має бути:

```
  GNU nano 9.0                    hello.txt                    Modified

Welcome to the Kitchen!
Today's special: Learning to edit files.
Chef says: You're doing great.


^G Help    ^O Write Out  ^W Where Is   ^K Cut       ^C Location
^X Exit    ^R Read File  ^\ Replace    ^U Paste     ^T Execute
```

Зверніть увагу на слово **Modified** у верхньому рядку. Це означає, що ви внесли зміни, які ще не були збережені.

---

## Збереження: Ctrl + O

Давайте збережемо файл. Натисніть:

```
Ctrl + O
```

nano попросить вас підтвердити назву файлу:

```
File Name to Write: hello.txt
```

Натисніть **Enter** для підтвердження. Індикатор «Modified» зникне. Ваш файл збережено.

---

## Вихід: Ctrl + X

Щоб вийти з nano та повернутися до термінала:

```
Ctrl + X
```

Якщо ви внесли зміни після останнього збереження, nano запитає:

```
Save modified buffer?  Y Yes  N No  ^C Cancel
```

Натисніть `Y`, щоб зберегти та вийти, `N`, щоб вийти без збереження, або `Ctrl + C`, щоб скасувати вихід і залишитися в nano.

---

## Перевірка файлу: `cat`

Тепер, коли ви повернулися в термінал, давайте підтвердимо, що файл було збережено. Команда `cat` виводить вміст файлу на екран:

```bash
cat hello.txt
```

Очікуваний результат:

```
Welcome to the Kitchen!
Today's special: Learning to edit files.
Chef says: You're doing great.
```

`cat` — це скорочення від «concatenate» (вона може об'єднувати кілька файлів), але більшість людей використовують її, щоб швидко глянути на вміст файлу. Сприймайте це як читання записки, не беручи її в руки — ви просто кидаєте на неї погляд.

---

## Редагування існуючого файлу

Щоб відредагувати файл, який уже існує, просто відкрийте його так само:

```bash
nano hello.txt
```

Ваш текст уже там. Використовуйте клавіші зі стрілками для переміщення. Друкуйте, щоб додати текст. Використовуйте Backspace/Delete, щоб видалити текст.

Додайте новий рядок в самому кінці:

```
PS: The pantry is fully stocked.
```

Збережіть за допомогою `Ctrl + O`, Enter. Вийдіть за допомогою `Ctrl + X`.

---

## Вирізання та вставлення рядків

У nano є прості функції вирізання та вставлення цілих рядків:

1. Наведіть курсор на рядок, який хочете вирізати
2. Натисніть **Ctrl + K**, щоб вирізати його (рядок зникне)
3. Наведіть курсор туди, куди хочете вставити рядок
4. Натисніть **Ctrl + U**, щоб вставити його

Ви можете вирізати кілька рядків, натиснувши `Ctrl + K` кілька разів — вони «накопичуються». Потім `Ctrl + U` вставить їх усі разом.

Це схоже на те, як шеф-кухар змінює порядок кроків у рецепті.

---

## Пошук тексту

Працюєте з довгим файлом і потрібно щось знайти? Натисніть **Ctrl + W**, введіть те, що шукаєте, і натисніть Enter.

```
Ctrl + W
```

nano запитає:

```
Search:
```

Введіть `special` і натисніть Enter. Курсор перескочить до першої згадки слова «special» у файлі.

Натисніть `Ctrl + W`, а потім знову Enter (нічого не вводячи), щоб знайти наступну згадку.

## Ваш перший скрипт

Тепер давайте зробимо щось потужне: напишемо скрипт. Скрипт — це просто текстовий файл, який містить команди, які комп'ютер може виконати. Це картка з рецептом для термінала.

### Крок 1: Створення скрипта

```bash
nano my-first-script.sh
```

> **Зупиніться та подумайте**: Цей скрипт починається з дивного рядка: `#!/bin/bash`. Як ви думаєте, що він робить? Це називається «шебанг» (shebang) — так скрипт повідомляє системі: «Коли цей файл запускається безпосередньо, використовуй bash для його інтерпретації». Якщо ви замість цього запустите файл за допомогою явної команди, наприклад `bash my-first-script.sh`, bash вже знатиме, як його інтерпретувати, навіть без шебангу. Ви побачите шебанги на початку багатьох shell-скриптів, оскільки вони чітко вказують на потрібний інтерпретатор.

Введіть наступне точно так, як написано:

```bash
#!/bin/bash

echo "Welcome to the kitchen!"
echo "Today's date is: $(date)"
echo "You are logged in as: $(whoami)"
echo "Your current directory is: $(pwd)"
echo ""
echo "Great job, chef! Your first script works!"
```

Давайте розберемо кожен рядок:

- `#!/bin/bash` — це той самий «шебанг». Він каже системі «використовуй програму bash для запуску цього файлу», коли ви виконуєте його безпосередньо через `./my-first-script.sh`. На практиці його варто додавати в усі shell-скрипти, які ви хочете запускати напряму, хоча скрипт може працювати і без нього, якщо ви явно викликаєте інтерпретатор, наприклад `bash my-first-script.sh`. (Так, «шебанг» — це реальний термін.)
- `echo` — команда, яка виводить текст на екран. Уявіть, що на кухні вигукують: «Замовлення готове!»
- `$(date)` — запускає команду `date` і вставляє результат. Синтаксис `$()` означає «виконай цю команду і дай мені її результат».
- `$(whoami)` — запускає `whoami`, яка повідомляє ваше ім'я користувача.
- `$(pwd)` — ви вже знаєте цю команду — вона виводить вашу поточну директорію.

### Крок 2: Збереження та вихід

Натисніть `Ctrl + O`, Enter, потім `Ctrl + X`.

### Крок 3: Перетворення на виконуваний файл

> **Зупиніться та подумайте**: Перед запуском `chmod` спробуйте запустити скрипт напряму: `./my-first-script.sh`. Що станеться? Ви маєте отримати повідомлення «Permission denied» (Доступ заборонено). Це зроблено навмисно — нові файли за замовчуванням вважаються просто даними. Комп'ютеру потрібен явний дозвів, щоб сприймати файл як програму. Це функція безпеки, а не помилка.

Наразі цей файл — просто текст. Комп'ютер не запускатиме його, бо він не має *дозволу* на виконання (execute). Давайте це виправимо:

```bash
chmod +x my-first-script.sh
```

`chmod` розшифровується як **ch**ange **mod**e (змінити режим). `+x` означає «додати дозвіл на виконання». Ви ніби кажете менеджеру ресторану: «Це не просто папірець, це рецепт, за яким треба готувати».

### Крок 4: Запуск

```bash
./my-first-script.sh
```

`./` означає «запустити цей файл із поточної директорії». Очікуваний результат:

```
Welcome to the kitchen!
Today's date is: Sun Mar 23 14:30:00 UTC 2026
You are logged in as: yourname
Your current directory is: /Users/yourname

Great job, chef! Your first script works!
```

Ви щойно написали та виконали свою першу програму. Це не дрібниця — це величезний крок.

## Примітка щодо vim

Рано чи пізно ви зустрінетеся з vim. Він неймовірно потужний, і його використовують багато досвідчених інженерів, хоча він не завжди встановлений за замовчуванням у мінімальних дистрибутивах Linux. Але vim відомий своєю крутою кривою навчання, оскільки він працює в «режимах» — ви не можете просто відкрити його і почати друкувати.

Ми не ігноруємо vim. Ми прибережемо його на потім, коли у вас буде більше досвіду. Зараз nano справляється зі своїми завданнями, а вчитися спочатку ходити, а потім бігати — це те, як тренуються і шеф-кухарі.

Коли ви будете готові, vim чекатиме на вас. А поки що nano — ваш друг.

## Чи знали ви?

- **nano — це клон клону.** Він був створений як вільна заміна редактору під назвою `pico`, який був вбудований у поштовий клієнт Pine у 1989 році. Назва `nano` походить від системи префіксів SI, відображаючи його роль як наступника pico (нано в 1000 разів більше за піко — хоча технічно нано менше за піко, тут мається на увазі порядок наступництва або жарт розробників).

- **Найстаріший текстовий редактор, який досі використовується, — це `ed`, створений у 1969 році.** Його написав Кен Томпсон, один із творців Unix. `ed` — це «рядковий редактор» (line editor), ви не можете бачити весь файл одразу. Ви редагуєте один рядок за раз. Інженери в 1970-х роках писали цілі операційні системи за допомогою `ed`. Ваш досвід роботи з nano у порівнянні з цим — справжня розкіш.

- **Конфігураційні файли керують світом.** Майже кожна програма під час запуску зчитує текстовий конфігураційний файл. Ваш вебсервер, ваша база даних, сам Kubernetes — усі вони читають текстові файли, щоб знати, як працювати. Вміння редагувати ці файли з термінала — одна з найкорисніших навичок у всьому ІТ.

## Типові помилки

| Помилка | Чому це проблема | Що робити натомість |
|---------|------------------|---------------------|
| Натискання Ctrl+Z замість Ctrl+X для виходу | Ctrl+Z призупиняє nano (ховає його) замість того, щоб закрити. Файл залишається відкритим у фоновому режимі | Якщо ви випадково призупинили програму, введіть `fg`, щоб повернути nano. Потім скористайтеся Ctrl+X для правильного виходу |
| Забули зберегти перед виходом | Ваші зміни будуть втрачені | Завжди натискайте Ctrl+O для збереження перед Ctrl+X для виходу. Або просто натисніть Ctrl+X і оберіть Y, коли з'явиться запит про збереження |
| Пропущено `#!/bin/bash` у скриптах, які ви плануєте запускати через `./script.sh` | Пряме виконання може не спрацювати, бо система не знатиме, який інтерпретатор використовувати | Якщо ви хочете запускати bash-скрипт напряму, починайте його з `#!/bin/bash`. Якщо ви запускаєте його як `bash script.sh`, bash уже обрано явно |
| Забули `chmod +x` перед запуском скрипта | Ви отримаєте помилку «Permission denied» | Виконайте `chmod +x filename.sh`, щоб зробити файл виконуваним |
| Використання `nano`, коли ви хотіли використати `cat` | Ви випадково відкриваєте файл для редагування, хоча хотіли просто прочитати його | Використовуйте `cat filename`, щоб переглянути вміст файлу без редагування |
| Редагування файлу на віддаленому сервері, думаючи, що він локальний | Ви можете випадково змінити налаштування робочого сервера (production) замість локальних тестових файлів, що призведе до збоїв | Завжди перевіряйте запит (наприклад, `user@server`) або виконуйте `hostname`, щоб підтвердити, на якій машині ви зараз редагуєте файли |
| Відкриття бінарного файлу (як-от зображення чи скомпільована програма) у nano та його збереження | Nano спробує прочитати бінарні дані як текст, і збереження назавжди зіпсує файл | Використовуйте nano тільки для звичайних текстових файлів (скрипти, конфіги, логи). Використовуйте `file filename`, щоб перевірити тип файлу перед відкриттям |

## Контрольні запитання

1. **Ви редагуєте важливий конфігураційний файл, і меню nano знизу каже вам натиснути `^O`, щоб зберегти (Write Out). Ви пробуєте ввести символ каретки (`^`), а потім літеру `O`, але редактор просто вписує «^O» у ваш файл. Що пішло не так?**
   <details>
   <summary>Відповідь</summary>
   Ви сприйняли символ `^` буквально замість того, щоб сприйняти його як клавішу-модифікатор. У термінальних програмах, таких як nano, символ `^` позначає клавішу Ctrl (Control) на вашій клавіатурі. Таким чином, `^O` означає, що ви повинні утримувати клавішу Ctrl і одночасно натиснути літеру O. Це стандартна конвенція в Unix-подібних системах для відображення комбінацій клавіш. Вводячи символи окремо, ви просто додавали текст у документ замість виконання команди збереження.
   </details>

2. **Ви витратили 15 хвилин на ретельне написання bash-скрипта в nano. Ви натискаєте Ctrl+X для виходу, але випадково натискаєте «N» на запит «Save modified buffer?». Що станеться з вашим скриптом, і що ви мали зробити інакше, щоб не втратити роботу?**
   <details>
   <summary>Відповідь</summary>
   Зміни у вашому скрипті будуть назавжди втрачені, оскільки натискання «N» каже nano відкинути всі модифікації, зроблені з моменту останнього збереження. На відміну від сучасних графічних редакторів з автозбереженням чи історією, термінальні редактори виконують саме те, що ви їм наказуєте в конкретний момент. Щоб запобігти цьому, варто виробити звичку вручну зберігати файл перед виходом. Для цього натисніть Ctrl+O (Write Out) і клавішу Enter для підтвердження назви файлу — це негайно запише зміни на диск. Після збереження індикатор «Modified» угорі зникне, і ви зможете безпечно вийти через Ctrl+X.
   </details>

3. **Ви завантажили від колеги скрипт, який закінчується на `.sh`. Ви робите його виконуваним і намагаєтеся запустити, але система видає помилку, що не знає, як виконати цей файл. Відкривши його в nano, ви бачите, що перший рядок — це просто `echo "Starting backup"`. Якого важливого елемента бракує, і чому він потрібен системі?**
   <details>
   <summary>Відповідь</summary>
   У скрипті бракує рядка шебангу, такого як `#!/bin/bash`, у самому верху файлу. Коли ви запускаєте скрипт напряму через `./script.sh`, Linux використовує цей рядок, щоб вирішити, який саме інтерпретатор має зчитувати файл. Без нього пряме виконання може завершитися помилкою, бо файл не вказує інтерпретатор. Якщо ж ви запускаєте скрипт явною командою, наприклад `bash script.sh`, bash уже обрано заздалегідь, тому шебанг у цьому випадку не є обов'язковим.
   </details>

4. **Ви написали ідеальний скрипт для автоматизації бекапів сервера, зберегли його як `backup.sh` і ввели `./backup.sh` для запуску. Замість того, щоб почати бекап, термінал суворо відповідає: `Permission denied`. Чому система заблокувала ваш скрипт і як це виправити?**
   <details>
   <summary>Відповідь</summary>
   Система заблокувала скрипт, оскільки за замовчуванням нові текстові файли мають лише права на читання та запис, але не на виконання. Це фундаментальна функція безпеки в Linux, розроблена для того, щоб запобігти випадковому чи зловмисному запуску довільних текстових файлів або завантажених даних як програм. Щоб це виправити, ви повинні явно надати файлу право на виконання командою `chmod +x backup.sh`. Це змінить режим файлу, сигналізуючи операційній системі, що ви, як власник, довіряєте цьому файлу і дозволяєте йому працювати як програмі.
   </details>

5. **Ви написали скрипт `backup.sh` і спробували запустити його через `./backup.sh`. Отримали «Permission denied». Потім виконали `chmod +x backup.sh` і спробували знову — тепер система каже «line 1: syntax error». Які ДВІ речі пішли не так і в якому порядку їх слід виправляти?**
   <details>
   <summary>Відповідь</summary>
   Перша проблема полягала у відсутності дозволів на виконання, що є стандартом для нових файлів з міркувань безпеки; це було правильно виправлено за допомогою `chmod +x`. Друга проблема в тому, що сам скрипт містить некоректні інструкції — швидше за все, це помилка в шебангу на першому рядку (наприклад, `#bin/bash` замість `#!/bin/bash`). Спершу завжди потрібно вирішувати питання з дозволами, тому що система навіть не намагатиметься прочитати синтаксис файлу, який їй не дозволено виконувати. Щойно дозволи надано, інтерпретатор може прочитати файл, знайти помилку в синтаксисі та видати корисне повідомлення, вказуючи безпосередньо на 1-й рядок. Для остаточного виправлення слід відкрити файл у nano і виправити одрук у першому рядку.
   </details>

## Практична вправа: Кухонна дошка оголошень

Створіть «дошку оголошень» для кухні ресторану, попрактикуйтеся в редагуванні та напишіть скрипт.

### Частина 1: Створення та редагування замітки

```bash
cd ~
nano kitchen-memo.txt
```

Введіть наступні п'ять рядків:

```
=== KITCHEN MEMO BOARD ===
1. Morning prep starts at 6 AM
2. New menu items arriving Thursday
3. Remember: clean as you go
4. Staff meeting at 3 PM Friday
```

Збережіть: `Ctrl + O`, Enter. Вийдіть: `Ctrl + X`.

Перевірте свою роботу:

```bash
cat kitchen-memo.txt
```

Ви маєте побачити всі п'ять рядків саме так, як ви їх надрукували.

### Частина 2: Редагування замітки

Відкрийте файл знову:

```bash
nano kitchen-memo.txt
```

Додайте шостий рядок в самому кінці:

```
5. Chef says: great work today, team!
```

Збережіть та вийдіть (`Ctrl + O`, Enter, `Ctrl + X`).

Перевірте:

```bash
cat kitchen-memo.txt
```

Усі шість рядків мають бути на місці.

### Частина 3: Написання скрипта для звіту

```bash
nano kitchen-report.sh
```

Введіть:

```bash
#!/bin/bash

echo "=== Kitchen Status Report ==="
echo "Date: $(date)"
echo "Chef on duty: $(whoami)"
echo ""
echo "--- Memo Board Contents ---"
cat kitchen-memo.txt
echo ""
echo "--- Files in current directory ---"
ls
echo ""
echo "Report complete. Kitchen is running smoothly!"
```

Збережіть та вийдіть.

Зробіть скрипт виконуваним та запустіть його:

```bash
chmod +x kitchen-report.sh
./kitchen-report.sh
```

Очікуваний результат (деталі можуть відрізнятися):

```
=== Kitchen Status Report ===
Date: Sun Mar 23 14:45:00 UTC 2026
Chef on duty: yourname

--- Memo Board Contents ---
=== KITCHEN MEMO BOARD ===
1. Morning prep starts at 6 AM
2. New menu items arriving Thursday
3. Remember: clean as you go
4. Staff meeting at 3 PM Friday
5. Chef says: great work today, team!

--- Files in current directory ---
hello.txt    kitchen-memo.txt    kitchen-report.sh    my-first-script.sh
...

Report complete. Kitchen is running smoothly!
```

### Частина 4: Очищення

```bash
rm hello.txt kitchen-memo.txt kitchen-report.sh my-first-script.sh
```

**Критерії успіху**: Ви створили файл у nano, відредагували його, написали bash-скрипт, який зчитує дані з іншого файлу, зробили його виконуваним і запустили. Ви більше не просто користуєтеся терміналом — ви *програмуєте* в ньому.
