## Чому цей модуль важливий

GitOps доводить ідею інфраструктури як коду до логічного завершення: Git стає єдиним джерелом істини для всього. Будь-які зміни в інфраструктурі відбуваються через Pull Requests, а не через прямі команди в терміналі. Цей підхід стає стандартом для роботи з Kubernetes і зробить вашу роботу набагато ефективнішою та безпечнішою.

## Що таке GitOps?

GitOps — це операційна модель, де:

1. **Git є джерелом істини** для бажаного стану системи.
2. **Зміни відбуваються через Git** (коміти, pull requests).
3. **Автоматизовані агенти** синхронізують реальний стан системи із записаним у Git.
4. **Відхилення (drift)** автоматично виправляються відповідно до стану в Git.

```
┌─────────────────────────────────────────────────────────────┐
│              ТРАДИЦІЙНИЙ CI/CD ПРОТИ GITOPS                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Традиційний CI/CD (Push-модель):                           │
│  ┌─────┐    ┌─────┐    ┌─────────┐    ┌─────────┐        │
│  │ Dev │───►│ Git │───►│   CI    │───►│ Кластер │        │
│  └─────┘    └─────┘    │ Пайплайн│    └─────────┘        │
│                        └─────────┘                         │
│  CI пайплайн "штовхає" зміни (потрібні доступи до кластера)│
│                                                             │
│  GitOps (Pull-модель):                                      │
│  ┌─────┐    ┌─────┐    ┌─────────┐    ┌─────────┐        │
│  │ Dev │───►│ Git │◄───│  GitOps │───►│ Кластер │        │
│  └─────┘    └─────┘    │  Агент  │    └─────────┘        │
│                        └─────────┘                         │
│  Агент "тягне" зміни з Git (агент живе всередині кластера) │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Чотири принципи GitOps

### 1. Декларативність

Усе описується декларативно:

```yaml
# Не "запусти 3 поди nginx"
# А "бажаний стан — 3 поди nginx"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  # ...
```

### 2. Версіонованість та незмінність

Усі зміни проходять через Git:

```bash
git log --oneline manifests/
abc123 Масштабування web до 5 реплік
d4e5f6g Додано кеш redis
g7h8i9j Початкове розгортання

# Кожна зміна:
# - Має версію (хеш коміту)
# - Незмінна (не можна просто видалити історію)
# - Має автора
# - Може бути перевірена (історія PR)
```

### 3. Автоматичне отримання змін (Pulled Automatically)

Агенти постійно перевіряють Git і застосовують зміни:

```
┌─────────────────────────────────────────────────────────────┐
│  Цикл GitOps-агента (кожні 30 сек - 5 хв):                 │
│                                                             │
│  1. Перевірити Git на наявність змін                       │
│  2. Порівняти стан у Git зі станом у кластері              │
│  3. Якщо є різниця: застосувати зміни до кластера          │
│  4. Повторити                                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 4. Постійне узгодження (Continuously Reconciled)

Будь-які ручні зміни автоматично скасовуються:

```bash
# Хтось вручну змінив продакшн (ай-ай-ай!)
kubectl scale deployment web --replicas=10

# GitOps агент помічає розбіжність:
# Git каже 3 репліки, в кластері зараз 10.
# Агент виправляє: повертає кількість до 3.

# Результат: Git завжди перемагає.
```

### Flux CD

Потужна альтернатива, зрілий (graduated) проєкт CNCF:

```yaml
# Flux GitRepository (джерело даних)
apiVersion: source.toolkit.fluxcd.io/v1
kind: GitRepository
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 1m
  url: https://github.com/myorg/my-app
  ref:
    branch: main

---
# Flux Kustomization (застосування маніфестів)
apiVersion: kustomize.toolkit.fluxcd.io/v1
kind: Kustomization
metadata:
  name: my-app
  namespace: flux-system
spec:
  interval: 5m
  path: ./kubernetes
  prune: true
  sourceRef:
    kind: GitRepository
    name: my-app
```

### Порівняння

| Функція | Argo CD | Flux CD |
|---------|---------|---------|
| Інтерфейс | Чудовий вебдашборд | Орієнтований на CLI |
| Мультиорендність | Вбудована | Через простори імен |
| RBAC | Розширений | Стандартний для Kubernetes |
| Підтримка Helm | Першокласна | Через контролери |
| Поріг входу | Середній | Вищий |
| Статус CNCF | Зрілий (Graduated) | Зрілий (Graduated) |

## Робочий процес GitOps

```
┌─────────────────────────────────────────────────────────────┐
│              РОБОЧИЙ ПРОЦЕС РОЗГОРТАННЯ GITOPS              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Розробник вносить зміни                                 │
│     └── Оновлює deployment.yaml (новий тег образу)         │
│                                                             │
│  2. Створює Pull Request                                    │
│     └── PR запускає: лінтинг, валідацію, сканування безпеки│
│                                                             │
│  3. Рецензування та схвалення                               │
│     └── Команда перевіряє, коментує, схвалює               │
│                                                             │
│  4. Злиття (Merge) в main                                   │
│     └── Git тепер містить новий бажаний стан               │
│                                                             │
│  5. GitOps агент помічає зміни                              │
│     └── Порівнює стан у Git зі станом у кластері           │
│                                                             │
│  6. Агент застосовує зміни                                  │
│     └── kubectl apply (або Helm upgrade тощо)              │
│                                                             │
│  7. Кластер досягає нового стану                            │
│     └── Нові поди запущені, старі видалені                 │
│                                                             │
│  Час від злиття до деплою: ~30 секунд - 5 хвилин           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Автоматизація оновлення образів

Сучасні інструменти GitOps можуть автоматично оновлювати теги образів:

```yaml
# Анотація Argo CD Image Updater
metadata:
  annotations:
    argocd-image-updater.argoproj.io/image-list: myapp=myrepo/myapp
    argocd-image-updater.argoproj.io/myapp.update-strategy: semver

# Конфіг Flux Image Automation
apiVersion: image.toolkit.fluxcd.io/v1beta1
kind: ImageUpdateAutomation
metadata:
  name: flux-system
spec:
  interval: 1m
  sourceRef:
    kind: GitRepository
    name: flux-system
  git:
    checkout:
      ref:
        branch: main
    commit:
      messageTemplate: 'Update image to {{.NewTag}}'
```

```
Потік:
1. CI збирає новий образ: myapp:v1.2.3
2. Пушить у реєстр (registry)
3. GitOps помічає новий тег
4. Оновлює Git-репо новим тегом
5. Синхронізує кластер на новий образ

Повна автоматизація з повним аудитом у Git!
```

## Відкат (Rollback) у GitOps

Відкат — це просто `git revert`:

```bash
# У продакшні знайшли баг! Жах!

# Варіант 1: Скасувати коміт у Git
git revert abc123
git push

# GitOps агент синхронізує зміни: стара версія відновлена
# Час на відкат: < 5 хвилин

# Варіант 2: Використати інтерфейс Argo CD
# Натиснути кнопку "Rollback" на додатку.
# Argo повернеться до попереднього стану синхронізації.

# Усі відкати зафіксовані в історії Git
git log --oneline
def456 Revert "Deploy v1.2.3"  # ← Факт відкату записано
abc123 Deploy v1.2.3           # ← Поганий деплой
```

## Чи знали ви?

- **Термін "GitOps" придумала компанія Weaveworks** у 2017 році. Усе почалося з блог-посту про те, як вони керували своїми кластерами Kubernetes.

- **GitOps виключає "kubectl apply" з вашого життя.** У чистому GitOps-середовищі людина ніколи не запускає kubectl проти продакшн-кластера. Усі зміни йдуть через Git.

- **Назва Argo CD** походить із грецької міфології. «Арго» — це корабель, на якому Ясон та аргонавти вирушили за золотим руном. CD означає Continuous Delivery (Безперервна доставка).

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Ручний kubectl у продакшні | Порушує аудит, викликає розбіжність | Тільки через GitOps агента |
| Секрети в Git | Витік конфіденційних даних | sealed-secrets або зовнішні сховища |
| Відсутність перевірки PR | Погані зміни потрапляють у продакшн | Обов'язкове схвалення (approvals) |
| Занадто часта синхронізація | Нестабільність кластера | Інтервали 1-5 хвилин |
| Відсутність перевірок стану (health checks) | Непрацюючі деплої залишаються | Налаштуйте health probes |

## Тест

1. **Яка ключова різниця між push-моделлю та pull-моделлю розгортання?**
   <details>
   <summary>Відповідь</summary>
   Push-модель: CI пайплайн "штовхає" зміни в кластер (потрібні креденшали кластера ззовні). Pull-модель (GitOps): Агент всередині кластера сам затягує зміни з Git. GitOps безпечніший, бо доступи до кластера не покидають його меж.
   </details>

2. **Що станеться, якщо хтось вручну змінить ресурси у продакшн-кластері при налаштованому GitOps?**
   <details>
   <summary>Відповідь</summary>
   GitOps агент помітить розбіжність (drift) між станом у Git (бажаним) та станом у кластері (реальним) і автоматично виправить кластер, повернувши його до стану, описаного в Git. Ручні зміни будуть перезаписані.
   </details>

3. **Як відбувається відкат (rollback) у моделі GitOps?**
   <details>
   <summary>Відповідь</summary>
   Виконується команда `git revert <коміт>` і робиться push у репозиторій. GitOps агент бачить повернення до старого стану в Git і синхронізує кластер. Усе прозоро та зафіксовано в історії Git.
   </details>

4. **Чому GitOps вважається безпечнішим за традиційний CI/CD?**
   <details>
   <summary>Відповідь</summary>
   У традиційному CI/CD пайплайнам потрібні секрети для доступу до кластера (які зберігаються в CI-системі). В GitOps секрети для доступу до кластера не потрібні нікому ззовні — тільки агент всередині кластера має права на зміни. Це мінімізує ризик витоку креденшалів.
   </details>

## Практична вправа

**Завдання**: Симуляція концепцій GitOps без встановлення Argo CD.

```bash
# Ця вправа демонструє логіку роботи GitOps вручну
# У справжньому GitOps це робить агент автоматично

# 1. Створіть "Git репозиторій" (директорію)
mkdir -p ~/gitops-demo/manifests
cd ~/gitops-demo

# 2. Створіть початковий бажаний стан
cat << 'EOF' > manifests/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: gitops-demo
spec:
  replicas: 2
  selector:
    matchLabels:
      app: gitops-demo
  template:
    metadata:
      labels:
        app: gitops-demo
    spec:
      containers:
      - name: nginx
        image: nginx:1.24
EOF

# 3. Синхронізація (симуляція роботи агента)
kubectl apply -f manifests/

# 4. Перевірка
kubectl get deployment gitops-demo

# 5. Симуляція відхилення (ручна зміна)
kubectl scale deployment gitops-demo --replicas=5

# 6. Перевірка розбіжності
kubectl get deployment gitops-demo
# Показує 5 реплік

# 7. Узгодження (GitOps виправляє ситуацію)
kubectl apply -f manifests/
# Знову 2 репліки! Git переміг.

# 8. "Коміт у Git" (зміна коду)
# (Для macOS/Linux)
sed -i 's/nginx:1.24/nginx:1.25/' manifests/deployment.yaml

# 9. Нова синхронізація (агент бачить зміни)
kubectl apply -f manifests/

# 10. Перевірка оновлення
kubectl get deployment gitops-demo -o jsonpath='{.spec.template.spec.containers[0].image}'
# Має бути nginx:1.25

# 11. Прибирання
kubectl delete -f manifests/
rm -rf ~/gitops-demo
```

**Критерії успіху**: Розуміння того, як працює синхронізація та виправлення відхилень (drift correction).

## Підсумок

**GitOps** — це сучасний стандарт експлуатації Kubernetes:

**Основні принципи**:
- Git — єдине джерело істини.
- Зміни тільки через Pull Requests.
- Автоматична синхронізація з кластером.
- Автоматичне виправлення ручних змін.

**Інструменти**:
- Argo CD: Повнофункціональний, з чудовим інтерфейсом.
- Flux CD: Зрілий проєкт CNCF, орієнтований на Kubernetes-native підхід.

**Переваги**:
- Повний аудит (історія Git).
- Легкий відкат (git revert).
- Вища безпека (секрети кластера не виходять за його межі).
- Самодокументована інфраструктура.

**Головна думка**: У світі GitOps ви ніколи не робите `kubectl apply` вручну у продакшні. Ви комітите в Git, а агент робить усе інше.
