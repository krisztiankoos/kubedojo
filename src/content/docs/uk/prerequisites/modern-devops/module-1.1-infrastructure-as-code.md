## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Пояснити**, що означає Infrastructure as Code і чому це замінило ручну конфігурацію серверів
- **Порівняти** інструменти IaC (Terraform, Ansible, Pulumi) і пояснити, коли використовувати кожен
- **Написати** просту декларативну конфігурацію і пояснити, чим вона відрізняється від скрипту
- **Визначити** антипатерни IaC (clickops, імперативні скрипти для декларативних задач, дрейф конфігурації)

---

## Чому цей модуль важливий

До появи інфраструктури як коду (Infrastructure as Code, IaC) налаштування серверів було ручним процесом, схильним до помилок, який неможливо було точно повторити. Фраза «на моїй машині працює» була універсальним виправданням. IaC змінив усе — інфраструктура стала придатною для версіонування, тестування та повторення. Розуміння IaC є обов'язковим, оскільки сам Kubernetes є системою, що працює за принципами IaC.

## Старий шлях: ClickOps

Уявіть собі: 2005 рік. Вам потрібно налаштувати вебсервер.

```
Ручний процес:
1. Замовити фізичний сервер (очікування 2-4 тижні)
2. Дочекатися, поки його встановлять у стійку в дата-центрі (1 тиждень)
3. Підключитися через SSH та встановити пакети
4. Налаштувати все, редагуючи файли вручну
5. Сподіватися, що ви пам'ятаєте, що саме зробили
6. Молитися, щоб нічого не зламалося

Документація: "Запитайте в Олега, він це налаштовував"
```

**Проблеми**:
- Немає запису про те, що було зроблено.
- Неможливо точно відтворити налаштування на іншому сервері.
- Різні «однакові» сервери поводяться по-різному.
- Олег іде у відпустку — і все ламається.

## Інфраструктура як код (IaC)

IaC означає **опис інфраструктури у файлах, які можна версіонувати, поширювати та виконувати автоматично**.

```
┌─────────────────────────────────────────────────────────────┐
│              ІНФРАСТРУКТУРА ЯК КОД                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Традиційно:                                                │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐            │
│  │ Людина  │ ───► │ Консоль │ ───► │ Сервер  │            │
│  │         │      │  (GUI)  │      │         │            │
│  └─────────┘      └─────────┘      └─────────┘            │
│                                                             │
│  З IaC:                                                     │
│  ┌─────────┐      ┌─────────┐      ┌─────────┐            │
│  │  Код    │ ───► │Інструмент│ ───► │ Сервер  │            │
│  │(файли)  │      │(Terraform)│    │         │            │
│  └─────────┘      └─────────┘      └─────────┘            │
│       │                                                     │
│       ▼                                                     │
│  ┌─────────┐                                               │
│  │   Git   │  Версіоновано, перевірено, відтворювано       │
│  └─────────┘                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Ключові принципи

### 1. Декларативний проти Імперативного

```
Імперативний (Як зробити):
"Встанови nginx, потім відредагуй /etc/nginx/nginx.conf,
потім перезапусти nginx"

Декларативний (Що отримати):
"Я хочу, щоб працював nginx з оцим конфігом"
```

Декларативний підхід кращий — ви описуєте бажаний стан, а інструмент сам розбирається, як його досягти.

### 2. Ідемпотентність

Запуск одного і того самого коду багато разів дає однаковий результат:

```bash
# Запуск цього 10 разів створить 10 серверів (ПОГАНО)
створити_сервер web-1

# Запуск цього 10 разів гарантує існування 1 сервера (ДОБРЕ)
забезпечити_наявність_сервера web-1
```

### 3. Контроль версій

```bash
git log --oneline infrastructure/
abc123 Додано репліку бази даних для продакшну
def456 Збільшено кількість вебсерверів до 5
ghi789 Початкове налаштування інфраструктури

# "Хто змінив продакшн?" — просто подивіться git blame
```

## Terraform: Галузевий стандарт

Terraform від HashiCorp — найпоширеніший інструмент IaC:

```hcl
# main.tf - конфігурація Terraform

# Визначаємо провайдера (де створювати ресурси)
provider "aws" {
  region = "eu-central-1"
}

# Визначаємо ресурс
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t2.micro"

  tags = {
    Name = "web-server"
    Environment = "production"
  }
}

# Визначаємо вивід
output "public_ip" {
  value = aws_instance.web.public_ip
}
```

```bash
# Робочий процес Terraform
terraform init      # Завантажити провайдери
terraform plan      # Подивитися майбутні зміни
terraform apply     # Створити інфраструктуру
terraform destroy   # Видалити все створене
```

### Чому Terraform переміг

| Функція | Terraform | CloudFormation |
|---------|-----------|----------------|
| Підтримка хмар | Будь-яка хмара | Тільки AWS |
| Керування станом | Вбудовано (state file) | Керується AWS |
| Синтаксис | HCL (зрозумілий) | JSON/YAML (багатослівний) |
| Поріг входу | Середній | Специфічний для AWS |
| Спільнота | Величезна | Обмежена AWS |

## Ansible: Просте налаштування

Ansible використовує YAML-файли ("playbooks") для конфігурації машин:

```yaml
# playbook.yml - Ansible playbook
---
- name: Налаштувати вебсервер
  hosts: webservers
  become: yes  # Запускати як root

  tasks:
    - name: Встановити nginx
      apt:
        name: nginx
        state: present
        update_cache: yes

    - name: Скопіювати конфігурацію
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: Restart nginx

    - name: Переконатися, що nginx запущено
      service:
        name: nginx
        state: started
        enabled: yes

  handlers:
    - name: Restart nginx
      service:
        name: nginx
        state: restarted
```

```bash
# Запустити плейбук
ansible-playbook -i inventory.ini playbook.yml
```

**Ключова перевага**: Не потребує агентів на серверах. Потрібен лише SSH-доступ.

## IaC для Kubernetes

Kubernetes — ЦЕ і є інфраструктура як код:

```yaml
# deployment.yaml - бажаний стан
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
```

```bash
# Застосувати бажаний стан
kubectl apply -f deployment.yaml

# Kubernetes приводить реальний стан у відповідність до бажаного.
# Це і є IaC у дії!
```

Зв'язок очевидний: **Kubernetes використовує ті самі декларативні та ідемпотентні принципи, що й Terraform або Ansible.**

## Найкращі практики IaC

### 1. Все в Git

```bash
infrastructure/
├── terraform/
│   ├── main.tf
│   ├── variables.tf
│   └── outputs.tf
├── kubernetes/
│   ├── deployments/
│   └── services/
└── ansible/
    └── playbooks/
```

### 2. Використовуйте модулі та повторювані компоненти

```hcl
# Не повторюйтеся (DRY)
module "web_server" {
  source = "./modules/ec2-instance"

  name          = "web-1"
  instance_type = "t2.micro"
}

module "api_server" {
  source = "./modules/ec2-instance"

  name          = "api-1"
  instance_type = "t2.small"
}
```

### 3. Розділяйте середовища (Environments)

```bash
environments/
├── dev/
│   └── main.tf      # Маленькі інстанси, 1 репліка
├── staging/
│   └── main.tf      # Середні інстанси, для тестів
└── prod/
    └── main.tf      # Великі інстанси, висока доступність
```

### 4. Ніколи не редагуйте вручну

```
Золоте правило: Якщо цього немає в коді — цього не існує.

Ручні зміни = розбіжність конфігурацій (drift) = баги о 3-й годині ночі.
```

## Робочий процес IaC

```
┌─────────────────────────────────────────────────────────────┐
│              РОБОЧИЙ ПРОЦЕС IaC                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Пишемо   ───►  2. Рецензуємо ───►  3. Тестуємо        │
│     код            (PR/MR)           (Plan)                │
│       │                                 │                   │
│       │                                 ▼                   │
│  6. Моніторимо ◄── 5. Застосовуємо ◄── 4. Схвалюємо       │
│     стан           зміни             (Merge)               │
│                                                             │
│  Усі зміни проходять через перевірку коду                  │
│  Усі зміни можна відстежити (аудит)                        │
│  Усі зміни можна скасувати (відкат)                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Чи знали ви?

- **NASA використовує Terraform** для керування своєю хмарною інфраструктурою. Якщо це достатньо надійно для космосу, то підійде і для вашого стартапу.

- **Назва Ansible** походить із науково-фантастичних романів Урсули Ле Гуїн, де «ансібль» — це пристрій для миттєвого зв'язку через космічний простір.

- **«Худоба, а не домашні улюбленці» (Cattle, not pets)** — це фундаментальний принцип IaC. Ставтеся до серверів як до худоби (замінні, пронумеровані), а не як до улюбленців (іменовані, незамінні). Ви повинні мати змогу видалити та перестворити будь-який сервер без жодних хвилювань.

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Ручні зміни після деплою через IaC | Розбіжність конфігурацій (drift) | Передеплойте з коду |
| Невикористання контролю версій | Відсутність аудиту та відкатів | Зберігайте все в Git |
| "Зашиті" секрети (passwords) у коді | Витік даних | Використовуйте менеджери секретів |
| Монолітні конфігурації | Важко підтримувати | Використовуйте модулі |
| Відсутність бекапу файлу стану (state) | Втрата контролю над інфраструктурою | Використовуйте віддалене сховище стану |

## Тест

1. **Що означає «ідемпотентність» в IaC?**
   <details>
   <summary>Відповідь</summary>
   Запуск одного і того самого коду кілька разів дає однаковий результат. Незалежно від того, чи застосуєте ви план Terraform один раз чи десять, кінцевий стан інфраструктури буде ідентичним.
   </details>

2. **Яка різниця між Terraform та Ansible?**
   <details>
   <summary>Відповідь</summary>
   Terraform створює інфраструктуру (віртуальні машини, мережі, бази даних). Ansible налаштовує вже наявні машини (встановлює ПЗ, керує конфігами). Вони часто використовуються разом.
   </details>

3. **Чому декларативний підхід кращий за імперативний?**
   <details>
   <summary>Відповідь</summary>
   Декларативний підхід описує «що» ви хочете отримати, а не «як» це зробити. Інструмент сам займається деталями реалізації, що робить код простішим і стійкішим до початкових умов.
   </details>

4. **Як Kubernetes пов'язаний з IaC?**
   <details>
   <summary>Відповідь</summary>
   Kubernetes — це і є IaC в чистому вигляді. Ви декларуєте бажаний стан у YAML-файлах, а Kubernetes постійно працює над тим, щоб реальний стан відповідав вашому опису. Застосовуються ті самі принципи: декларативність, ідемпотентність, контроль версій.
   </details>

## Практична вправа

**Завдання**: Відчуйте принципи IaC за допомогою `kubectl`.

```bash
# Ця вправа використовує Kubernetes для демонстрації концепцій IaC

# 1. Створіть деплоймент декларативно
cat << 'EOF' > deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: iac-demo
spec:
  replicas: 2
  selector:
    matchLabels:
      app: iac-demo
  template:
    metadata:
      labels:
        app: iac-demo
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
EOF

# 2. Застосуйте його (IaC у дії)
kubectl apply -f deployment.yaml

# 3. Перевірте стан
kubectl get deployment iac-demo

# 4. Застосуйте ще раз (ідемпотентність)
kubectl apply -f deployment.yaml
# "deployment.apps/iac-demo unchanged" - результат той самий!

# 5. Змініть код
# (Для macOS/Linux)
sed -i 's/replicas: 2/replicas: 4/' deployment.yaml

# 6. Застосуйте зміни
kubectl apply -f deployment.yaml

# 7. Перевірте результат
kubectl get deployment iac-demo
# Тепер показує 4 репліки

# 8. Контроль версій (симуляція)
# У реальному світі: git add deployment.yaml && git commit

# 9. Прибирання
kubectl delete -f deployment.yaml
rm deployment.yaml
```

**Критерії успіху**: Розуміння того, як декларативні файли + команда apply утворюють робочий процес IaC.

## Підсумок

**Інфраструктура як код (IaC)** трансформує керування системами:

**Основні принципи**:
- Декларативність замість імперативності.
- Ідемпотентні операції.
- Контроль версій (Git).
- Перевірка змін (Code Review).

**Ключові інструменти**:
- Terraform: Створення хмарних ресурсів.
- Ansible: Конфігурація машин.
- Kubernetes: Оркестрація контейнерів (IaC вбудовано).

**Чому це важливо**:
- Відтворювані середовища.
- Історія всіх змін (аудит).
- Швидке відновлення після збоїв.
- Спільна робота через Git.

**Зв'язок з Kubernetes**: Усе, що ви робите в Kubernetes, базується на принципах IaC. Ваші YAML-файли — це і є код вашої інфраструктури.
