## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Пояснити** принцип «зсуву вліво» в безпеці та чому дешевше виявляти проблеми рано
- **Визначити** топ ризиків безпеки Kubernetes (неправильна конфігурація, відкриті дашборди, надмірний RBAC)
- **Описати** безпечний CI/CD пайплайн зі скануванням образів, управлінням секретами та застосуванням політик
- **Порівняти** інструменти DevSecOps (Trivy, OPA/Gatekeeper, Falco) і пояснити, від чого захищає кожен

---

## Чому цей модуль важливий

Раніше безпека була чимось другорядним — окрема команда казала «ні» в самому кінці розробки. Це не працює в хмарних середовищах, де розгортання відбуваються декілька разів на день. DevSecOps інтегрує безпеку на кожному етапі життєвого циклу розробки. Для Kubernetes, де помилки конфігурації є причиною №1 інцидентів безпеки, це критично важливо.

## Безпека контейнерів

### 1. Безпека образів

```dockerfile
# ПОГАНО: Велика поверхня атаки, робота від root
FROM ubuntu:latest
RUN apt-get update && apt-get install -y nginx
COPY app /app
CMD ["nginx"]

# ДОБРЕ: Мінімальний образ, не-root користувач
FROM nginx:1.25-alpine
RUN adduser -D -u 1000 appuser
COPY --chown=appuser:appuser app /app
USER appuser
EXPOSE 8080
```

### 2. Сканування образів

```bash
# Trivy - найпопулярніший сканер із відкритим кодом
trivy image nginx:1.25

# Приклад виводу:
# nginx:1.25 (debian 12.0)
# Total: 142 (UNKNOWN: 0, LOW: 89, MEDIUM: 45, HIGH: 7, CRITICAL: 1)
```

### 3. Підпис образів

```bash
# Підпис образу, щоб гарантувати його незмінність
# Використовуємо cosign (sigstore)
cosign sign myregistry/myapp:v1.0

# Перевірка перед деплоєм
cosign verify myregistry/myapp:v1.0
```

## Безпека в Kubernetes

### Типові помилки конфігурації

```yaml
# ПОГАНО: Под із надмірними правами
apiVersion: v1
kind: Pod
metadata:
  name: insecure-pod
spec:
  containers:
  - name: app
    image: myapp
    securityContext:
      privileged: true          # Ніколи так не робіть!
      runAsUser: 0              # Не запускайте від root
    volumeMounts:
    - name: host
      mountPath: /host          # Не монтуйте хостову ФС
  volumes:
  - name: host
    hostPath:
      path: /

# ДОБРЕ: Безпечна конфігурація пода
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    fsGroup: 1000
  containers:
  - name: app
    image: myapp
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
    resources:
      limits:
        memory: "128Mi"
        cpu: "500m"
```

### Стандарти безпеки подів (Pod Security Standards)

У Kubernetes 1.25+ використовуються вбудовані рівні безпеки:

```yaml
# Застосування стандартів на рівні простору імен
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/audit: restricted
```

| Рівень | Опис |
|-------|-------------|
| privileged | Жодних обмежень (небезпечно) |
| baseline | Мінімальні обмеження, запобігає відомим ескалаціям прав |
| restricted | Максимальні обмеження, згідно з кращими практиками |

## Мережева безпека

```yaml
# Network Policy: Дозволити тільки специфічний трафік
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: api-network-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: api
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 5432
```

## Інструменти сканування безпеки

### KubeLinter (Конфігурація)

```bash
# Перевірка YAML маніфестів на помилки
kube-linter lint deployment.yaml

# Приклад виводу:
# - контейнер "app" не має файлової системи тільки для читання
# - контейнер "app" запускається від root
```

### Kubescape (Комплексна перевірка)

```bash
# Повний скан безпеки за стандартами NSA-CISA
kubescape scan framework nsa

# Перевіряє:
# - Помилки конфігурації
# - Проблеми RBAC
# - Мережеві політики
# - Вразливості образів
```

### Trivy (Універсальний інструмент)

```bash
# Скан образу
trivy image myapp:v1

# Скан маніфестів
trivy config .

# Скан запущеного кластера
trivy k8s --report summary cluster
```

## Кращі практики RBAC

```yaml
# Принцип найменших привілеїв
# Надавайте тільки ті права, які дійсно потрібні

# ПОГАНО: Права адміна кластера для всіх
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: developer-admin
subjects:
- kind: User
  name: developer@company.com
roleRef:
  kind: ClusterRole
  name: cluster-admin    # Занадто багато влади!

# ДОБРЕ: Мінімальні права в межах простору імен
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: development
  name: developer
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "create", "update"]
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]
```

## Чи знали ви?

- **Понад 90% інцидентів безпеки в Kubernetes** спричинені помилками конфігурації, а не вразливостями нульового дня. Сканування конфігів дає більше результату, ніж ви думаєте.

- **Вразливість Log4Shell (2021)** могла бути виявлена автоматичним скануванням залежностей. Компанії з налагодженим SCA дізналися, які їхні додатки під загрозою, за лічені години.

- **Falco обробляє мільярди подій** у таких компаніях, як Shopify. Він може виявити запуск shell-оболонки в контейнері за мілісекунди.

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Секрети в Git | Постійний ризик витоку | Використовуйте менеджери секретів |
| Запуск від імені root | Ризик виходу за межі контейнера | Завжди runAsNonRoot |
| Відсутність Network Policies | Вільне переміщення атакуючого | Політика "заборонити все за замовчуванням" |
| Тег `latest` | Неможливість відстежити вразливості | Використовуйте конкретні версії |
| Відсутність сканування | Невідомі дірки в захисті | Скануйте в CI/CD пайплайні |
| Права адміна всюди | Величезний радіус ураження | RBAC з мінімальними правами |

## Тест

1. **Що означає «Shift Left» у контексті DevSecOps?**
   <details>
   <summary>Відповідь</summary>
   Виявлення проблем безпеки якомога раніше в життєвому циклі розробки (ближче до написання коду, далі від продакшну). Раннє виявлення значно дешевше і простіше у виправленні.
   </details>

2. **Чому контейнери не повинні працювати від імені root?**
   <details>
   <summary>Відповідь</summary>
   Якщо зловмисник зламає додаток у контейнері, права root значно полегшують «втечу» з контейнера на хост-машину. Обмеження прав мінімізує шкоду від потенційного зламу.
   </details>

3. **Яка різниця між SAST та DAST?**
   <details>
   <summary>Відповідь</summary>
   SAST (Static Analysis) сканує вихідний код без його запуску. DAST (Dynamic Analysis) тестує вже запущений додаток. SAST знаходить помилки в логіці коду, DAST — вразливості в рантаймі, як-от SQL-ін'єкції.
   </details>

4. **Яку проблему вирішує інструмент Sealed Secrets?**
   <details>
   <summary>Відповідь</summary>
   Він дозволяє зберігати зашифровані секрети прямо в Git. Тільки ваш Kubernetes кластер має ключ для їх розшифрування. Це дозволяє повноцінно використовувати GitOps навіть для конфіденційних даних.
   </details>

## Практична вправа

**Завдання**: Навчіться розрізняти безпечні та небезпечні конфігурації.

```bash
# 1. Створіть небезпечний деплоймент
cat << 'EOF' > insecure-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: insecure-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: insecure
  template:
    metadata:
      labels:
        app: insecure
    spec:
      containers:
      - name: app
        image: nginx:latest
        securityContext:
          privileged: true
          runAsUser: 0
        ports:
        - containerPort: 80
EOF

# 2. Перевірте його (симуляція сканування)
echo "Чекліст безпеки (Security Review):"
echo "[ ] Образ використовує конкретний тег (не :latest) - НІ"
echo "[ ] Контейнер працює не від root - НІ"
echo "[ ] privileged: false - НІ"
echo "[ ] Встановлено ліміти ресурсів - НІ"

# 3. Створіть безпечну версію
cat << 'EOF' > secure-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: secure-app
spec:
  replicas: 1
  selector:
    matchLabels:
      app: secure
  template:
    metadata:
      labels:
        app: secure
    spec:
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        fsGroup: 1000
      containers:
      - name: app
        image: nginx:1.25-alpine
        securityContext:
          allowPrivilegeEscalation: false
          readOnlyRootFilesystem: true
          capabilities:
            drop:
            - ALL
        ports:
        - containerPort: 8080
        resources:
          limits:
            memory: "128Mi"
            cpu: "500m"
          requests:
            memory: "64Mi"
            cpu: "250m"
EOF

# 4. Порівняйте файли
diff insecure-deployment.yaml secure-deployment.yaml || true

# 5. Прибирання
rm insecure-deployment.yaml secure-deployment.yaml
```

**Критерії успіху**: Розуміння різниці між небезпечними та захищеними налаштуваннями подів.

- Безпека як код.
- Автоматичне сканування.
- Принцип найменших привілеїв.

**Безпека в пайплайні**:
- SAST: Скан коду.
- SCA: Скан бібліотек.
- Контейнери: Скан образів.
- Конфіги: Скан K8s YAML.

**Безпека Kubernetes**:
- Стандарти безпеки подів.
- Мережеві політики.
- RBAC (мінімальні права).
- Керування секретами.

**Інструменти**:
- Trivy: Образи та конфіги.
- KubeLinter/Kubescape: Конфіги K8s.
- Falco: Виявлення загроз у рантаймі.
- Vault/Sealed Secrets: Секрети.

**Головна думка**: Безпека — це не перешкода, це якість вашого продукту. Вона є відповідальністю кожного члена команди.
