## Чому цей модуль важливий

У міру впровадження Kubernetes в організаціях виникає проблема: Kubernetes занадто складний, і далеко не кожен розробник хоче (або повинен) знати його глибоко. Платформна інженерія (Platform Engineering) розв'язує цю проблему, створюючи внутрішні платформи, які приховують складність. Це дозволяє розробникам зосередитися на коді, тоді як платформна команда опікується інфраструктурою. Розуміння цього тренду допоможе вам правильно позиціювати свої навички на ринку.

## Що таке платформна інженерія?

Платформна інженерія — це **дисципліна проєктування та побудови інструментів і робочих процесів, які забезпечують самообслуговування розробників (self-service) у хмарну епоху**.

```
┌─────────────────────────────────────────────────────────────┐
│              МОДЕЛЬ ПЛАТФОРМИ                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                    РОЗРОБНИКИ                               │
│                        │                                    │
│                        │ "Мені потрібна база даних"         │
│                        │ "Задеплой мій додаток"             │
│                        │ "Покажи мені логи"                 │
│                        ▼                                    │
│  ┌─────────────────────────────────────────────────────┐   │

│  │                                                      │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐         │   │
│  │  │ Портал   │  │ Шаблони  │  │ Обмеження│         │   │
```

## Ключові компоненти

### 1. Портал розробника (Developer Portal)

Єдине місце, де розробники можуть знаходити, створювати та керувати сервісами.

```
┌─────────────────────────────────────────────────────────────┐
│  ПОРТАЛ РОЗРОБНИКА (наприклад, Backstage)                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  КАТАЛОГ СЕРВІСІВ                                           │
│  ├── frontend-app (React, команда: frontend)               │
│  ├── api-service (Go, команда: backend)                    │
│  ├── user-service (Python, команда: identity)              │
│  └── payment-service (Java, команда: payments)             │
│                                                             │
│  ШАБЛОНИ (Scaffolding)                                      │
│  ├── [Створити новий мікросервіс]                          │
│  ├── [Створити нову базу даних]                            │
│  └── [Створити пайплайн даних]                             │
│                                                             │
│  ДОКУМЕНТАЦІЯ                                               │
│  ├── Як почати роботу                                      │
│  ├── API документація                                      │
│  └── Інструкції з експлуатації (Runbooks)                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. Золоті шляхи (Golden Paths)

Готові, рекомендовані способи виконання типових завдань:

```yaml
# Замість: "Розбирайся сам, як деплоїти в Kubernetes"
# Золотий шлях: "Використовуй цей шаблон"

# scaffold/microservice-template/template.yaml
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: microservice-template
  title: Створити новий мікросервіс
spec:
  parameters:
    - name: serviceName
      description: Назва сервісу
    - name: team
      description: Назва команди
    - name: language
      options: [go, python, java, node]

  steps:
    - id: create-repo
      action: github:create-repo
    - id: add-ci-cd
      action: add-github-actions
    - id: register-service
      action: backstage:register
```

### 3. Інфраструктура самообслуговування

```yaml
# Запит розробника (спрощено)
apiVersion: platform.company.io/v1
kind: DatabaseRequest
metadata:
  name: my-postgres
spec:
  type: postgresql
  size: small
  team: backend

# Платформа автоматично робить:
# - Створення БД у хмарі
# - Налаштування бекапів
# - Додавання в моніторинг
# - Видачу доступів (credentials)
# - Мережеві політики
```

## Crossplane: Інфраструктура через API K8s

Crossplane дозволяє описувати хмарну інфраструктуру як ресурси Kubernetes:

```yaml
# Платформна команда створює складне визначення
apiVersion: apiextensions.crossplane.io/v1
kind: CompositeResourceDefinition
metadata:
  name: databases.platform.company.io
spec:
  group: platform.company.io
  names:
    kind: Database
  versions:
    - name: v1
      schema:
        openAPIV3Schema:
          type: object
          properties:
            spec:
              type: object
              properties:
                size:
                  type: string
                  enum: [small, medium, large]

---
# Розробник просто робить запит:
apiVersion: platform.company.io/v1
kind: Database
metadata:
  name: my-db
spec:
  size: small

# Платформа сама створює складні ресурси в AWS/Azure/GCP
```

## Чи знали ви?

- **Backstage народився у Spotify**, щоб керувати понад 1500 мікросервісів. Вони відкрили код у 2020 році, і зараз це проєкт CNCF, який використовують тисячі компаній.

- **Платформна інженерія — тренд №1** у стратегічних технологічних прогнозах Gartner на 2024 рік. Вони пророкують, що до 2026 року 80% великих ІТ-компаній матимуть платформні команди.

- **Платформа Netflix** дозволяє розробникам деплоїти сотні разів на день у тисячі мікросервісів майже без участі адмінів.

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Будувати без участі розробників | Платформа, якою ніхто не користується | Спілкуйтеся з розробниками спочатку |
| Занадто сильна абстракція | Неможливо налагодити (debug) | Залишайте "чорні ходи" для профі |
| Відсутність золотих шляхів | Кожна команда робить по-своєму | Визначайте та просувайте кращі практики |
| Платформа як бюрократичний бар'єр | Сповільнює всіх | Платформа має прискорювати, а не блокувати |
| "Побудуємо — і вони прийдуть" | Низька популярність | Ставтеся до платформи як до продукту, рекламуйте її |

## Тест

1. **Яку основну проблему вирішує платформна інженерія?**
   <details>
   <summary>Відповідь</summary>
   Когнітивне перевантаження розробників. Розробникам не обов'язково знати всі тонкощі Kubernetes, хмарних мереж чи моніторингу. Платформа надає зручні абстракції, які приховують складність, зберігаючи при цьому високі стандарти якості та безпеки.
   </details>

2. **Що таке "Золотий шлях" (Golden Path)?**
   <details>
   <summary>Відповідь</summary>
   Це готовий, перевірений та рекомендований спосіб виконання завдання (наприклад, створення нового сервісу). Замість того, щоб розбиратися в деталях деплою, розробник використовує шаблон, де вже враховані всі кращі практики.
   </details>

3. **Чим платформна інженерія відрізняється від класичного DevOps?**
   <details>
   <summary>Відповідь</summary>
   DevOps казав "ти збудував — ти й запускаєш" (кожна команда робить усе). Платформна інженерія створює окрему команду, яка будує інструментарій для всіх інших команд, щоб продуктові команди могли зосередитися саме на бізнес-логіці, а не на інфраструктурних "велосипедах".
   </details>

4. **Що таке Backstage?**
   <details>
   <summary>Відповідь</summary>
   Це портал розробника з відкритим кодом від Spotify. Він містить каталог сервісів, шаблони для створення нових компонентів та систему документації. Це стандарт для побудови внутрішніх платформ (IDP).
   </details>

## Практична вправа

**Завдання**: Відчуйте силу абстракцій у Kubernetes.

```bash
# Ця вправа демонструє, як платформи приховують складність

# 1. "Складний шлях" (те, що платформа бере на себе)
cat << 'EOF' > complex-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
  labels:
    app: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "100m"
          limits:
            memory: "128Mi"
            cpu: "200m"
        livenessProbe:
          httpGet:
            path: /
            port: 80
        readinessProbe:
          httpGet:
            path: /
            port: 80
---
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 80
EOF

kubectl apply -f complex-deployment.yaml

# 2. "Шлях платформи" (спрощений інтерфейс)
# Уявіть платформу, де розробник каже лише:
cat << 'EOF'
# platform-request.yaml (гіпотетично)
name: myapp
image: nginx:1.25
replicas: 3
expose: true
EOF

# Платформа сама перетворює це на повний набір маніфестів K8s.
# Розробнику не потрібно думати про:
# - Ліміти ресурсів
# - Перевірки стану (health checks)
# - Конфігурацію сервісів
# - Правильні мітки та селектори

# 3. Перевірте, що створив цей запит
kubectl get deployment myapp
kubectl get service myapp
kubectl get pods -l app=myapp

# 4. Саме цим займаються платформні команди:
# - Створюють прості інтерфейси для розробників
# - Зашивають "кращі практики" в шаблони
# - Встановлюють безпечні межі (ліміти, безпека тощо)

# 5. Прибирання
kubectl delete -f complex-deployment.yaml
rm complex-deployment.yaml
```

**Критерії успіху**: Розуміння того, навіщо потрібні абстракції над складними системами.

## Підсумок

**Платформна інженерія** — це про покращення життя розробників:

**Головна ідея**:
- Внутрішня платформа розробника (IDP).
- Самообслуговування з розумними обмеженнями.
- "Платформа як Продукт".

**Ключові компоненти**:
- Портал розробника (Backstage).
- Золоті шляхи (шаблони).
- Інфраструктура як сервіс.

**Інструменти**:
- Backstage для порталів.
- Crossplane для інфраструктури.
- Kubernetes як фундамент.

**Чому це важливо**:
- Зменшує когнітивне навантаження.
- Підвищує продуктивність розробників.
- Забезпечує дотримання стандартів і безпеки.
- Дозволяє масштабувати практики DevOps у великих компаніях.
