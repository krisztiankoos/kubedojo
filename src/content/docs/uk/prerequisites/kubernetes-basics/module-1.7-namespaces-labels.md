## Чому цей модуль важливий

Зі зростанням кластера організація ресурсів стає критично важливою. Простори імен (Namespaces) забезпечують ізоляцію та логічне групування. Мітки (Labels) дозволяють вибирати та організовувати ресурси. Разом вони допомагають тримати великі кластери керованими.

## Простори імен (Namespaces)

### Що таке простори імен?

Простори імен — це віртуальні кластери всередині одного фізичного кластера:

```
┌─────────────────────────────────────────────────────────────┐
│              КЛАСТЕР KUBERNETES                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────┐  ┌──────────────────┐               │
│  │  Простір імен:   │  │  Простір імен:   │               │
│  │    "frontend"    │  │    "backend"     │               │
│  │  ┌────┐ ┌────┐   │  │  ┌────┐ ┌────┐   │               │
│  │  │web │ │api │   │  │  │db  │ │cache│  │               │
│  │  └────┘ └────┘   │  │  └────┘ └────┘   │               │
│  │                  │  │                  │               │
│  │  Ізольований від │  │  Ізольований від │               │
│  │  інших просторів │  │  інших просторів │               │
│  └──────────────────┘  └──────────────────┘               │
│                                                             │
│  ┌──────────────────┐                                      │
│  │  Простір імен:   │  Стандартні простори імен:           │
│  │   "kube-system"  │  • default                          │
│  │ Системні поди    │  • kube-system                       │
│  └──────────────────┘  • kube-public                       │
│                        • kube-node-lease                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Створення просторів імен

```bash
# Створити простір імен
kubectl create namespace my-app
kubectl create ns my-app          # Коротка форма

# Переглянути список
kubectl get namespaces
kubectl get ns
```

```yaml
# namespace.yaml
apiVersion: v1
kind: Namespace
metadata:
  name: my-app
  labels:
    team: backend
```

### Робота з просторами імен

```bash
# Виконати команду в конкретному просторі імен
kubectl get pods -n my-app
kubectl get pods --namespace=my-app

# У всіх просторах імен одночасно
kubectl get pods -A
kubectl get pods --all-namespaces

# Встановити простір імен за замовчуванням для поточного контексту
kubectl config set-context --current --namespace=my-app

# Перевірити поточне значення за замовчуванням
kubectl config view --minify | grep namespace
```

### Стандартні простори імен

| Назва | Призначення |
|-------|-------------|
| `default` | Сюди потрапляють ресурси, якщо простір імен не вказано явно |
| `kube-system` | Для компонентів самого Kubernetes (API server, scheduler тощо) |
| `kube-public` | Для даних, які мають бути доступні всім (навіть без автентифікації) |
| `kube-node-lease` | Для даних про стан вузлів кластера (heartbeats) |

## Мітки (Labels)

### Що таке мітки?

Мітки — це пари ключ-значення, які прикріплюються до ресурсів:

```yaml
metadata:
  name: my-pod
  labels:
    app: nginx
    environment: production
    team: frontend
    version: v1.2.3
```

### Використання міток

```bash
# Додати мітки при створенні
kubectl run nginx --image=nginx --labels="app=nginx,env=prod"

# Додати мітку до існуючого ресурсу
kubectl label pod nginx tier=frontend

# Видалити мітку (знак мінус в кінці ключа)
kubectl label pod nginx tier-

# Оновити мітку (перезаписати значення)
kubectl label pod nginx env=staging --overwrite
```

### Фільтрація за мітками

```bash
# Фільтрація за однією міткою
kubectl get pods -l app=nginx
kubectl get pods --selector=app=nginx

# За декількома мітками (умова "ТА" / AND)
kubectl get pods -l app=nginx,env=prod

# Фільтрація за списком значень
kubectl get pods -l 'env in (prod, staging)'
kubectl get pods -l 'app notin (test)'
kubectl get pods -l 'tier'              # Тільки ті, що мають таку мітку
kubectl get pods -l '!tier'             # Тільки ті, що НЕ мають такої мітки

# Показати мітки в окремій колонці
kubectl get pods --show-labels
```

## Мітки проти Анотацій (Labels vs Annotations)

| Характеристика | Мітки (Labels) | Анотації (Annotations) |
|----------------|----------------|------------------------|
| Використання | Для вибору та фільтрації | Для метаданих (описів) |
| Обмеження | Суворі обмеження на символи | Можуть містити великі дані |
| Робота K8s | Використовуються K8s для зв'язків | Для людей та зовнішніх інструментів |
| Довжина | Дуже обмежена | Майже необмежена |

```yaml
metadata:
  labels:                    # Для вибору
    app: nginx
    version: v1
  annotations:               # Для інформації
    description: "Головний вебсервер"
    git-commit: "abc123"
    monitoring.enabled: "true"
```

## Найкращі практики

### Рекомендовані мітки (стандарт CNCF)

```yaml
metadata:
  labels:
    app.kubernetes.io/name: nginx
    app.kubernetes.io/version: "1.25"
    app.kubernetes.io/component: frontend
    app.kubernetes.io/part-of: website
    app.kubernetes.io/managed-by: helm
```

### Стратегія використання просторів імен

```
Окруження розробки:
  - dev
  - staging
  - prod

За командами:
  - team-frontend
  - team-backend
  - team-data

За застосунками:
  - app-website
  - app-api
  - app-admin
```

## Чи знали ви?

- **Деякі ресурси не належать до просторів імен.** Вузли (Nodes), PersistentVolumes та самі Namespaces мають глобальну область видимості в межах кластера.

- **Мітки мають обмеження.** Ключі та значення мають ліміти на кількість символів та дозволені знаки.

- **Простори імен не забезпечують безпеку самі по собі.** Поди в різних просторах імен можуть вільно спілкуватися між собою, якщо не налаштовані NetworkPolicies.

- **Простір імен `default` — це пастка.** Використовуйте явні простори імен, щоб не змішувати ресурси різних проєктів.

## Тест

1. **Яка різниця між мітками (labels) та анотаціями (annotations)?**
   <details>
   <summary>Відповідь</summary>
   Мітки використовуються для вибору ресурсів (Сервісами, Деплойментами тощо) і мають обмеження за форматом. Анотації призначені для довільних метаданих (використовуються людьми або інструментами) і можуть містити набагато більше інформації.
   </details>

2. **Як побачити поди в усіх просторах імен одночасно?**
   <details>
   <summary>Відповідь</summary>
   `kubectl get pods -A` або `kubectl get pods --all-namespaces`.
   </details>

3. **Як вибрати поди, що мають мітку app=nginx ТА env=prod?**
   <details>
   <summary>Відповідь</summary>
   `kubectl get pods -l app=nginx,env=prod`. Кома між мітками означає логічне "ТА" (AND).
   </details>

4. **Чи є простори імен бар'єром для безпеки?**
   <details>
   <summary>Відповідь</summary>
   За замовчуванням — ні. Поди можуть спілкуватися через межі просторів імен. Простори імен — це інструмент організації та квот, а не безпеки. Для обмеження мережевого доступу потрібні NetworkPolicies.
   </details>

## Практична вправа

**Завдання**: Попрактикуватися з просторами імен та мітками.

```bash
# 1. Створіть простори імен
kubectl create namespace frontend
kubectl create namespace backend

# 2. Запустіть поди з мітками в різних просторах імен
kubectl run web --image=nginx -n frontend --labels="app=web,tier=frontend"
kubectl run api --image=nginx -n backend --labels="app=api,tier=backend"
kubectl run cache --image=redis -n backend --labels="app=cache,tier=backend"

# 3. Список усіх подів
kubectl get pods -A

# 4. Фільтрація за міткою
kubectl get pods -A -l tier=backend

# 5. Показати всі мітки
kubectl get pods -A --show-labels

# 6. Додати мітку існуючому поду
kubectl label pod web -n frontend version=v1

# 7. Встановити простір імен за замовчуванням
kubectl config set-context --current --namespace=frontend
kubectl get pods  # Тепер показує frontend за замовчуванням

# 8. Повернути на default
kubectl config set-context --current --namespace=default

# 9. Прибирання
kubectl delete namespace frontend backend
```

**Критерії успіху**: Вміння фільтрувати поди за простором імен та мітками.

## Підсумок

**Простори імен (Namespaces)**:
- Віртуальні кластери всередині кластера.
- Забезпечують організацію та обмеження видимості.
- Не є бар'єром безпеки за замовчуванням.
- Використовуйте `-n` для вибору або `-A` для всіх.

**Мітки (Labels)**:
- Пари ключ-значення на ресурсах.
- Дозволяють вибирати ресурси (Сервіси, Деплойменти).
- Використовуйте `-l ключ=значення` для фільтрації.
- Кома розділяє умови "ТА".

**Найкращі практики**:
- Давайте просторам імен зрозумілі назви.
- Дотримуйтесь єдиних стандартів іменування міток.
- Ніколи не працюйте в `default` для реальних завдань.
