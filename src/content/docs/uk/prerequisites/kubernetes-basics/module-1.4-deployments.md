## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Створювати** Деплойменти як імперативно, так і декларативно
- **Масштабувати** застосунки вгору та вниз і пояснити, що відбувається з базовими Подами
- **Виконувати** плаваючі оновлення та відкати, і діагностувати застрягле розгортання
- **Пояснити** ієрархію Deployment → ReplicaSet → Pod і навіщо існує кожен рівень

---

## Чому цей модуль важливий

У реальних проєктах ви майже ніколи не створюєте Поди напряму. Замість цього ви використовуєте Деплойменти (Deployments). Деплойменти керують Подами за вас: автоматично обробляють масштабування, оновлення та самовідновлення.

---

## Навіщо потрібні Деплойменти?

Використання Подів без контролерів має суттєві недоліки:

```
┌─────────────────────────────────────────────────────────────┐
│              ПОДИ НАПРЯМУ ПРОТИ ДЕПЛОЙМЕНТІВ                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  "ГОЛИЙ" ПОД (NAKED POD):                                   │
│  - Под вмирає → залишається мертвим                        │
│  - Вузол виходить з ладу → Под втрачено                    │
│  - Складно масштабувати                                     │
│  - Оновлення потребують видалення та створення вручну       │
│                                                             │
│  З ДЕПЛОЙМЕНТОМ:                                            │
│  - Под вмирає → автоматично створюється новий              │
│  - Вузол виходить з ладу → поди переносяться на інший      │
│  - Масштабування однією командою                            │
│  - Оновлення без простоїв (rolling updates)                 │
│  - Відкат до попередніх версій                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Створення Деплойментів

### Імперативно (для швидких тестів)

```bash
# Створити деплоймент
kubectl create deployment nginx --image=nginx

# Вказати кількість реплік
kubectl create deployment nginx --image=nginx --replicas=3

# Переглянути YAML без створення
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml
```

### Декларативно (професійний шлях)

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  replicas: 3                    # Бажана кількість копій подів
  selector:                      # Як деплоймент знаходить свої поди
    matchLabels:
      app: nginx
  template:                      # Шаблон пода
    metadata:
      labels:
        app: nginx               # Має збігатися з selector
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
```

```bash
kubectl apply -f deployment.yaml
```

## Основні операції

### Перегляд Деплойментів

```bash
# Список деплойментів
kubectl get deployments
kubectl get deploy              # Коротка форма

# Детальна інформація
kubectl describe deployment nginx

# Переглянути пов'язані ресурси
kubectl get deploy,rs,pods
```

### Масштабування (Scaling)

```bash
# Змінити кількість реплік
kubectl scale deployment nginx --replicas=5

# Або відредагувати YAML та застосувати
kubectl edit deployment nginx
# Змініть replicas, збережіть файл

# Стежте за появою подів
kubectl get pods -w
```

### Оновлення (Rolling Update)

```bash
# Оновити образ контейнера
kubectl set image deployment/nginx nginx=nginx:1.26

# Або через редагування деплойменту
kubectl edit deployment nginx

# Стежити за процесом оновлення
kubectl rollout status deployment nginx

# Переглянути історію оновлень
kubectl rollout history deployment nginx
```

### Відкат (Rollback)

```bash
# Скасувати останню зміну
kubectl rollout undo deployment nginx

# Повернутися до конкретної версії
kubectl rollout history deployment nginx
kubectl rollout undo deployment nginx --to-revision=2
```

## Стратегія Rolling Update

Деплойменти оновлюють Поди поступово:

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%        # Скільки додаткових подів можна створити
      maxUnavailable: 25%  # Скільки подів можуть бути недоступні
```

```
┌─────────────────────────────────────────────────────────────┐
│              ПРОЦЕС ROLLING UPDATE                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Початковий стан (3 репліки, v1):                          │
│  [v1] [v1] [v1]                                            │
│                                                             │
│  Початок оновлення до v2:                                  │
│  [v1] [v1] [v1] [v2]    ← Створено новий под               │
│                                                             │
│  Новий под готовий:                                        │
│  [v1] [v1] [v2] [v2]    ← Старий под видаляється           │
│                                                             │
```

## Розбір YAML Деплойменту

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  replicas: 3                    # Бажана кількість подів
  selector:
    matchLabels:
      app: nginx                 # Має збігатися з мітками шаблону
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  template:                      # Шаблон пода (структура як у Pod)
    metadata:
      labels:
        app: nginx               # Мітки для пошуку сервісами
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
        ports:
        - containerPort: 80
        resources:
          requests:
            memory: "64Mi"
            cpu: "100m"
          limits:
            memory: "128Mi"
            cpu: "200m"
```

## Самовідновлення в дії

```bash
# Створіть деплоймент
kubectl create deployment nginx --image=nginx --replicas=3

# Перегляньте поди
kubectl get pods

# Видаліть один под
kubectl delete pod <назва-пода>

# Одразу перевірте список подів знову
kubectl get pods
# Новий под уже створюється!

# Деплоймент завжди підтримує бажаний стан
kubectl get deployment nginx
# READY покаже 3/3
```

## Чи знали ви?

- **Деплойменти не керують Подами напряму.** Вони керують ReplicaSet-ами, а ті вже — Подами. Це дозволяє робити відкати (rollbacks).

- **Кожне оновлення створює новий ReplicaSet.** Старі ReplicaSet-и зберігаються (з 0 реплік) для історії відкатів.

- **Налаштування `maxSurge: 0, maxUnavailable: 0` неможливе.** Ви не можете оновити систему, не створивши нових подів або не видаливши старі.

- **`kubectl rollout restart`** запускає нове оновлення без зміни конфігурації. Корисно для оновлення образів, якщо тег не змінився.

---

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Selector не збігається з мітками шаблону | Деплоймент не зможе керувати подами | Перевірте, щоб labels збігалися точно |
| Тег :latest | Неможливо зробити зрозумілий відкат | Використовуйте версії (теги) |
| Відсутність лімітів ресурсів | Може вплинути на інші поди | Завжди встановлюйте resources |
| Не чекаєте завершення оновлення | Можна пропустити помилки розгортання | Використовуйте `rollout status` |

## Тест

1. **Який зв'язок між Deployment, ReplicaSet та Pod?**
   <details>
   <summary>Відповідь</summary>
   Deployment керує ReplicaSet-ами. ReplicaSet керує Подами. Ви взаємодієте з Деплойментом, а він автоматизує все інше. ReplicaSet-и дозволяють робити відкат, зберігаючи попередні версії.
   </details>

2. **Що станеться, якщо видалити Под, яким керує Деплоймент?**
   <details>
   <summary>Відповідь</summary>
   Деплоймент (через ReplicaSet) автоматично створить новий Под, щоб підтримувати бажану кількість реплік. Це називається самовідновленням (self-healing).
   </details>

3. **Як зробити відкат Деплойменту до попередньої версії?**
   <details>
   <summary>Відповідь</summary>
   Командою `kubectl rollout undo deployment НАЗВА_ДЕПЛОЙМЕНТУ`. Для відкату до конкретної версії: `kubectl rollout undo deployment НАЗВА --to-revision=N`.
   </details>

4. **За що відповідає параметр `maxSurge` при оновленні?**
   <details>
   <summary>Відповідь</summary>
   За максимальну кількість подів, які можуть бути створені понад бажану кількість реплік під час оновлення. Наприклад, `maxSurge: 25%` при 4 репліках дозволяє тимчасово мати 5 подів.
   </details>

## Практична вправа

**Завдання**: Створити Деплоймент, масштабувати його, оновити та зробити відкат.

```bash
# 1. Створіть деплоймент
kubectl create deployment web --image=nginx:1.24

# 2. Масштабуйте до 3 реплік
kubectl scale deployment web --replicas=3

# 3. Перевірте статус
kubectl get deploy,rs,pods

# 4. Оновіть образ
kubectl set image deployment/web nginx=nginx:1.25

# 5. Стежте за оновленням
kubectl rollout status deployment web

# 6. Перевірте історію
kubectl rollout history deployment web

# 7. Симулюйте проблему — зробіть відкат
kubectl rollout undo deployment web

# 8. Перевірте версію після відкату
kubectl get deployment web -o jsonpath='{.spec.template.spec.containers[0].image}'
# Має повернути nginx:1.24

# 9. Прибирання
kubectl delete deployment web
```

**Критерії успіху**: Створення, масштабування, оновлення та відкат виконані успішно.

## Підсумок

Деплойменти керують вашими застосунками:

**Функції**:
- Декларативні оновлення.
- Оновлення без простоїв (rolling updates).
- Можливість відкату.
- Самовідновлення.
- Легке масштабування.

**Основні команди**:
- `kubectl create deployment`
- `kubectl scale deployment`
- `kubectl set image`
- `kubectl rollout status`
- `kubectl rollout undo`

**Найкращі практики**:
- Завжди використовуйте Деплойменти замість окремих Подів.
- Використовуйте конкретні версії образів.
- Встановлюйте запити та ліміти ресурсів.
