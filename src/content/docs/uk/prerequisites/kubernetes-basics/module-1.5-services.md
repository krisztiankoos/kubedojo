## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Створювати** Сервіси для виставлення подів і пояснити, чому подам потрібні Сервіси (IP подів — тимчасові)
- **Обирати** між ClusterIP, NodePort та LoadBalancer і пояснити, коли використовувати кожен
- **Тестувати** з'єднання з сервісом зсередини кластера за допомогою `curl` та DNS-імен
- **Дебажити** сервіс, що не може дістатися до подів, перевіряючи мітки, селектори та ендпоінти

---

## Чому цей модуль важливий

Поди ефемерні — вони з'являються і зникають, і кожен раз отримують нову IP-адресу. Сервіси (Services) забезпечують стабільну мережу: фіксовану IP-адресу та DNS-ім'я, які направляють трафік до ваших Подів, незалежно від того, скільки їх і як часто вони змінюються.

## Створення Сервісів

### Імперативно (швидко)

```bash
# Відкрити доступ до деплойменту
kubectl expose deployment nginx --port=80

# З вказанням конкретного типу
kubectl expose deployment nginx --port=80 --type=NodePort

# Перевірити сервіс
kubectl get services
kubectl get svc              # Коротка форма
```

### Декларативно (професійно)

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx
spec:
  selector:
    app: nginx               # Має збігатися з мітками подів
  ports:
  - port: 80                 # Порт сервісу
    targetPort: 80           # Порт контейнера
  type: ClusterIP            # Тип за замовчуванням
```

```bash
kubectl apply -f service.yaml
```

## Типи Сервісів

### ClusterIP (За замовчуванням)

Доступ тільки всередині кластера:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: internal-api
spec:
  type: ClusterIP            # Можна не вказувати
  selector:
    app: api
  ports:
  - port: 80
    targetPort: 8080
```

```bash
# Доступ тільки зсередини кластера
curl http://internal-api:80
```

### NodePort

Відкриває доступ на IP кожного вузла (node) через статичний порт:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-nodeport
spec:
  type: NodePort
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080          # Опційно: діапазон 30000-32767
```

```bash
# Доступ ззовні кластера
curl http://<IP-вузла>:30080
```

### LoadBalancer

Створює зовнішній балансувальник навантаження (у хмарних середовищах):

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80
    targetPort: 80
```

```bash
# Отримати зовнішню IP (тільки в хмарі)
kubectl get svc web-lb
# Колонка EXTERNAL-IP покаже адресу балансувальника
```

## Схема роботи Сервісів

```
┌─────────────────────────────────────────────────────────────┐
│              ТИПИ СЕРВІСІВ                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ClusterIP (Тільки внутрішній)                              │
│  ┌─────────────────────────────────────┐                  │
│  │  ClusterIP:80 ──► Под:8080          │                  │
│  │               ──► Под:8080          │                  │
│  │  (Доступний тільки в межах кластера)│                  │
│  └─────────────────────────────────────┘                  │
│                                                             │
│  NodePort (Зовнішній через Вузол)                           │
│  ┌─────────────────────────────────────┐                  │
│  │  <IPВузла>:30080 ──► ClusterIP:80 ──► Поди             │
│  │  (Доступний ззовні)                 │                  │
│  └─────────────────────────────────────┘                  │
│                                                             │
│  LoadBalancer (Хмарний зовнішній)                           │
│  ┌─────────────────────────────────────┐                  │
│  │  <ЗовнішняIP>:80 ──► NodePort ──► ClusterIP ──► Поди   │
│  │  (Хмарний провайдер керує LB)       │                  │
│  └─────────────────────────────────────┘                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Виявлення сервісів (DNS)

Kubernetes автоматично створює DNS-записи для Сервісів:

```
<назва-сервісу>.<простір-імен>.svc.cluster.local
```

```bash
# З будь-якого пода ви можете звернутися до:
curl nginx                           # У тому ж просторі імен
curl nginx.default                   # Явне вказання простору імен
curl nginx.default.svc               # Більш детально
curl nginx.default.svc.cluster.local # Повна назва (FQDN)
```

### Приклад

```bash
# Створіть деплоймент і сервіс
kubectl create deployment nginx --image=nginx
kubectl expose deployment nginx --port=80

# Перевірте DNS з іншого пода
kubectl run test --image=busybox --rm -it -- wget -qO- nginx
# Поверне HTML-код nginx!

# Перевірте повне DNS-ім'я
kubectl run test --image=busybox --rm -it -- nslookup nginx.default.svc.cluster.local
```

## Селектори: Як Сервіси знаходять Поди

Сервіси використовують селектори міток (label selectors):

```yaml
# Сервіс
spec:
  selector:
    app: nginx
    tier: frontend

# Под (має мати ВСІ вказані мітки)
metadata:
  labels:
    app: nginx
    tier: frontend
```

```bash
# Перевірити, на які поди направляє трафік сервіс
kubectl get endpoints nginx
# Покаже IP:Порт підходящих подів
```

## Відображення портів (Port Mapping)

```yaml
spec:
  ports:
  - port: 80           # Порт сервісу (куди звертаються клієнти)
    targetPort: 8080   # Порт контейнера (де слухає програма)
    protocol: TCP      # TCP (за замовчуванням) або UDP
```

```
┌─────────────────────────────────────────────────────────────┐
│  Клієнт ──► Сервіс:80 ──► Под:8080                         │
│                │                 │                          │
│             "port"         "targetPort"                     │
└─────────────────────────────────────────────────────────────┘
```

## Чи знали ви?

- **Сервіси використовують iptables або IPVS.** kube-proxy налаштовує правила, які перенаправляють трафік із IP Сервісу на IP Подів. Насправді немає окремого процесу-проксі для кожного запиту.

- **ClusterIP є віртуальним.** У жодного мережевого інтерфейсу немає цієї IP-адреси. Вона існує лише в правилах маршрутизації (iptables).

- **NodePort працює на ВСІХ вузлах.** Навіть вузли, де немає цільових подів, будуть коректно перенаправляти трафік.

- **Балансування навантаження зазвичай випадкове.** Кожне нове з'єднання може потрапити на інший под.

---

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Селектор не збігається з мітками подів | У сервісу немає кінцевих точок (endpoints) | Перевірте `kubectl get endpoints` |
| Неправильний targetPort | З'єднання відхилено | Має збігатися з портом, на якому слухає додаток |
| Використання IP пода замість сервісу | Зв'язок переривається при перезапуску пода | Завжди використовуйте назву сервісу |

## Тест

1. **Чому варто використовувати Сервіси замість прямих IP-адрес Подів?**
   <details>
   <summary>Відповідь</summary>
   IP-адреси подів змінюються при кожному перезапуску. Сервіси надають стабільну IP-адресу та DNS-ім'я, які залишаються незмінними. Також сервіси автоматично балансують навантаження між декількома подами.
   </details>

2. **Яка різниця між ClusterIP та NodePort?**
   <details>
   <summary>Відповідь</summary>
   ClusterIP доступний тільки всередині кластера. NodePort відкриває сервіс на IP-адресі кожного вузла кластера через статичний порт (30000-32767), що робить його доступним ззовні.
   </details>

3. **Як Сервіси визначають, на які Поди направляти трафік?**
   <details>
   <summary>Відповідь</summary>
   За допомогою селекторів міток (label selectors). Поле `selector` у специфікації Сервісу вказує, які мітки мають бути у пода. Тільки поди з відповідними мітками отримають трафік.
   </details>

4. **Яке DNS-ім'я може використати под, щоб звернутися до сервісу "api" у просторі імен "backend"?**
   <details>
   <summary>Відповідь</summary>
   `api.backend`, `api.backend.svc` або повне ім'я `api.backend.svc.cluster.local`. Якщо поди в одному просторі імен, достатньо просто `api`.
   </details>

## Практична вправа

**Завдання**: Створити деплоймент та відкрити до нього доступ через Сервіс.

```bash
# 1. Створіть деплоймент
kubectl create deployment web --image=nginx --replicas=3

# 2. Відкрийте доступ як ClusterIP
kubectl expose deployment web --port=80

# 3. Перевірте сервіс та кінцеві точки
kubectl get svc web
kubectl get endpoints web

# 4. Протестуйте зсередини кластера
kubectl run test --image=busybox --rm -it -- wget -qO- web

# 5. Створіть сервіс типу NodePort
kubectl expose deployment web --port=80 --type=NodePort --name=web-external

# 6. Дізнайтеся призначений порт
kubectl get svc web-external
# Знайдіть порт у діапазоні 30000-32767

# 7. Прибирання
kubectl delete deployment web
kubectl delete svc web web-external
```

**Критерії успіху**: Внутрішній сервіс працює, `endpoints` показують IP-адреси подів.

## Підсумок

Сервіси забезпечують стабільну мережу:

**Типи**:
- ClusterIP — тільки внутрішній (за замовчуванням).
- NodePort — зовнішній через порт вузла.
- LoadBalancer — зовнішній через хмарний балансувальник.

**Ключові концепції**:
- Селектори знаходять поди за мітками.
- DNS-імена для зручного звернення.
- Відображення портів (port → targetPort).
- Ендпоінти (Endpoints) показують реальні адреси подів.

**Команди**:
- `kubectl expose deployment НАЗВА --port=ПОРТ`
- `kubectl get svc`
- `kubectl get endpoints`
