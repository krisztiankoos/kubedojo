## Чому цей модуль важливий

Поди (Pods) — це найменша одиниця розгортання в Kubernetes. Кожен контейнер, який ви запускаєте в K8s, працює всередині Пода. Розуміння Подів є фундаментальним — все інше в Kubernetes будується на цій концепції.

---

## Що таке Под (Pod)?

Под — це оболонка навколо одного або декількох контейнерів:

```
┌─────────────────────────────────────────────────────────────┐
│                          POD                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                    Под                               │   │
│  │  ┌─────────────┐     ┌─────────────┐               │   │
│  │  │  Контейнер  │     │  Контейнер  │  (опційно)    │   │
│  │  │ (основний)  │     │ (sidecar)   │               │   │
│  │  └─────────────┘     └─────────────┘               │   │
│  │                                                     │   │
│  │  Спільне:                                          │   │
│  │  • Мережевий простір (один IP, localhost працює)   │   │
│  │  • Сховища (volumes)                                │   │
│  │  • Життєвий цикл Пода                              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Контейнери в Поді:                                        │
│  - Мають однакову IP-адресу                                │
│  - Спілкуються між собою через localhost                   │
│  - Використовують спільні змонтовані томи                  │
│  - Плануються разом на один і той самий вузол              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Ключові тези

- **Один контейнер на Под — це стандарт.** Багатоконтейнерні Поди використовуються для специфічних патернів (sidecar, adapters).
- **Поди ефемерні.** Їх можна видалити або замінити в будь-який момент.
- **Под отримує унікальну IP-адресу.** Інші Поди можуть звертатися до нього за цією IP.
- **Поди не самолікуються.** Якщо Под вмирає, він залишається мертвим (якщо ним не керує контролер, наприклад, Deployment).

## Створення Подів

### Імперативно (для швидкого тестування)

```bash
# Простий под
kubectl run nginx --image=nginx

# З вказанням порту
kubectl run nginx --image=nginx --port=80

# З мітками (labels)
kubectl run nginx --image=nginx --labels="app=web,tier=frontend"

# Тестовий запуск для перегляду YAML
kubectl run nginx --image=nginx --dry-run=client -o yaml
```

### Декларативно (професійний шлях)

```yaml
# pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.25
    ports:
    - containerPort: 80
```

```bash
kubectl apply -f pod.yaml
```

## Структура Pod YAML

```yaml
apiVersion: v1          # Версія API
kind: Pod               # Тип ресурсу
metadata:               # Метадані
  name: nginx           # Назва пода (обов'язково)
  namespace: default    # Простір імен (опційно)
  labels:               # Пари ключ-значення для вибору
    app: nginx
    environment: dev
  annotations:          # Метадані не для ідентифікації
    description: "Мій под nginx"
spec:                   # Бажаний стан
  containers:           # Список контейнерів
  - name: nginx         # Назва контейнера
    image: nginx:1.25   # Образ контейнера
    ports:              # Порти, що відкриваються
    - containerPort: 80
    env:                # Змінні середовища
    - name: MY_VAR
      value: "my-value"
    resources:          # Запити та ліміти ресурсів
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"
```

## Життєвий цикл Пода

```
┌─────────────────────────────────────────────────────────────┐
│              СТАДІЇ ЖИТТЄВОГО ЦИКЛУ ПОДА                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Pending ──► Running ──► Succeeded/Failed                  │
│     │           │                                           │
│     │           ▼                                           │
│     │       Unknown (вузол втратив зв'язок)                │
│     │                                                       │
│     ▼                                                       │
│  Failed (помилка завантаження образу тощо)                 │
│                                                             │
│  СТАТУСИ:                                                  │
│  • Pending    - Очікування планування або завантаження     │
│  • Running    - Хоча б один контейнер запущено             │
│  • Succeeded  - Усі контейнери успішно завершили роботу    │
│  • Failed     - Хоча б один контейнер завершився з помилкою│
│  • Unknown    - Неможливо визначити стан (проблеми вузла)  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Стани контейнера

```bash
kubectl get pod nginx -o jsonpath='{.status.containerStatuses[0].state}'
```

- **Waiting**: Контейнер ще не запущено (йде завантаження образу тощо).
- **Running**: Контейнер виконується.
- **Terminated**: Контейнер завершив роботу (успішно чи з помилкою).

## Поширені операції з Подами

### Перегляд Подів

```bash
# Список подів
kubectl get pods
kubectl get pods -o wide          # Більше інфо (IP, вузол)
kubectl get pods --show-labels    # Показати мітки

# Детальна інформація
kubectl describe pod nginx

# Стежити за статусом
kubectl get pods -w
```

### Налагодження (Debug)

```bash
# Перегляд логів
kubectl logs nginx
kubectl logs nginx -f             # Стрімінг (follow)
kubectl logs nginx --previous     # Логи попереднього екземпляра

# Виконання команд
kubectl exec nginx -- ls /
kubectl exec -it nginx -- bash    # Інтерактивна оболонка

# Отримання подій
kubectl get events --field-selector involvedObject.name=nginx
```

### Доступ до Подів

```bash
# Прокидання порту (Port forward)
kubectl port-forward pod/nginx 8080:80
# Тепер доступ за адресою localhost:8080

# Отримати IP пода
kubectl get pod nginx -o jsonpath='{.status.podIP}'
```

### Зміна Подів

```bash
# Редагування (обмежене для подів)
kubectl edit pod nginx

# Видалення та повторне створення
kubectl delete pod nginx
kubectl apply -f pod.yaml

# Примусове видалення пода, що "застряг"
kubectl delete pod nginx --force --grace-period=0
```

## Запити та ліміти ресурсів (Requests and Limits)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: resource-demo
spec:
  containers:
  - name: demo
    image: nginx
    resources:
      requests:          # Гарантований мінімум
        memory: "64Mi"
        cpu: "250m"      # 0.25 ядра CPU
      limits:            # Максимально дозволено
        memory: "128Mi"
        cpu: "500m"
```

```
┌─────────────────────────────────────────────────────────────┐
│              REQUESTS ПРОТИ LIMITS                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  REQUEST = Гарантований мінімум ресурсів                    │
│  - Використовується для прийняття рішень про планування     │
│  - Под не буде запущено, якщо на вузлі немає стільки місця │
│                                                             │
│  LIMIT = Максимально дозволений обсяг ресурсів              │
│  - Контейнер може використовувати до цієї межі              │
│  - CPU: обмежується (throttled) при перевищенні             │
│  - Memory: вбивається (OOMKilled) при перевищенні           │
│                                                             │
│  Приклад:                                                   │
│  requests:                                                  │
│    cpu: "250m"     # 0.25 ядра гарантовано                 │
│    memory: "64Mi"  # 64 MB гарантовано                     │
│  limits:                                                    │
│    cpu: "500m"     # Може зрости до 0.5 ядра               │
│    memory: "128Mi" # Буде вбито, якщо перевищить 128 MB    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Чи знали ви?

- **IP-адреси Подів внутрішні для кластера.** Ви не можете отримати доступ до IP Пода безпосередньо ззовні кластера.

- **Поди — це "худоба", а не "домашні улюбленці".** Вони розроблені для заміни. Ніколи не розраховуйте на існування конкретного екземпляра Пода.

- **localhost працює між контейнерами в Поді.** Вони ділять мережевий простір імен.

- **Под без контролера називається "голим" (naked pod).** Якщо він помре, ніщо його не відновить. У продакшні завжди використовуйте Деплойменти.

## Типові помилки

| Помилка | Наслідки | Рішення |
|---------|----------|---------|
| Використання подів напряму | Немає самовідновлення чи масштабування | Використовуйте Деплойменти |
| Відсутність лімітів ресурсів | Може "з'їсти" ресурси інших подів | Завжди встановлюйте resources |
| Тег :latest | Непередбачувані версії образу | Використовуйте конкретні теги |
| Ігнорування статусу пода | Можна пропустити збої | Перевіряйте `kubectl get pods` |

## Тест

1. **Що таке Под (Pod)?**
   <details>
   <summary>Відповідь</summary>
   Под — це найменша одиниця розгортання в Kubernetes. Це оболонка навколо одного або декількох контейнерів, які мають спільну мережу та сховища. Контейнери в одному поді мають спільну IP-адресу та можуть спілкуватися через localhost.
   </details>

2. **Чому не варто використовувати "голі" поди (naked pods) у продакшні?**
   <details>
   <summary>Відповідь</summary>
   "Голі" поди не вміють самовідновлюватися. Якщо под помре або вузол вийде з ладу, под зникне назавжди. Потрібно використовувати Деплойменти або інші контролери, які автоматично створюють нові поди на заміну тим, що вийшли з ладу.
   </details>

3. **Яка різниця між resource requests та limits?**
   <details>
   <summary>Відповідь</summary>
   Requests — це мінімальні ресурси, які гарантуються поду (використовуються для вибору вузла). Limits — це максимум, який поду дозволено використовувати. При перевищенні ліміту CPU под сповільнюється, при перевищенні ліміту пам'яті — вбивається (OOMKilled).
   </details>

4. **Як контейнери в одному поді спілкуються між собою?**
   <details>
   <summary>Відповідь</summary>
   Через localhost (127.0.0.1). Оскільки вони ділять один мережевий простір, вони бачать порти один одного на локальному інтерфейсі.
   </details>

## Практична вправа

**Завдання**: Створити, оглянути та налагодити Под.

```bash
# 1. Створіть YAML маніфест пода
cat << 'EOF' > my-pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
  labels:
    app: demo
spec:
  containers:
  - name: demo
    image: nginx:1.25
    ports:
    - containerPort: 80
    resources:
      requests:
        memory: "64Mi"
        cpu: "100m"
      limits:
        memory: "128Mi"
        cpu: "200m"
EOF

# 2. Створіть под
kubectl apply -f my-pod.yaml

# 3. Стежте за запуском
kubectl get pods -w

# 4. Отримайте детальну інформацію
kubectl describe pod my-pod

# 5. Перегляньте логи
kubectl logs my-pod

# 6. Виконайте команду в поді
kubectl exec my-pod -- nginx -v

# 7. Прокиньте порт і протестуйте
kubectl port-forward pod/my-pod 8080:80 &
curl localhost:8080
kill %1  # Зупинити port-forward

# 8. Прибирання
kubectl delete -f my-pod.yaml
rm my-pod.yaml
```

**Критерії успіху**: Под запущено, логи доступні, port-forward працює.

## Підсумок

Поди — це фундамент:

**Що це таке**:
- Найменша одиниця розгортання.
- Оболонка для контейнерів.
- Спільна мережа та диски.

**Основні операції**:
- `kubectl run` / `kubectl apply -f`
- `kubectl get pods`
- `kubectl describe pod`
- `kubectl logs`
- `kubectl exec`

**Найкращі практики**:
- Використовуйте контролери (Deployments), а не поди напряму.
- Завжди встановлюйте запити (requests) та ліміти (limits) ресурсів.
- Використовуйте конкретні теги образів.
