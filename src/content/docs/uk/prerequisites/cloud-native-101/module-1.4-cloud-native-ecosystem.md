## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Орієнтуватися** в ландшафті CNCF та пояснити, які категорії інструментів існують
- **Зіставляти** типові проблеми (спостережуваність, безпека, мережа) з інструментами, що їх вирішують
- **Пояснити** процес градуації CNCF та що він означає для зрілості інструменту
- **Визначити** 5-6 інструментів, які найчастіше зустрічатимуться на першій роботі з K8s

---

## Чому цей модуль важливий

Kubernetes не існує в ізоляції. Він є центром величезної екосистеми проєктів, інструментів та практик. Розуміння цієї екосистеми допомагає вам:

1. Знати, які інструменти існують для розв'язання різних проблем
2. Розуміти описи вакансій та дискусії в команді
3. Планувати свій шлях навчання
4. Уникати «винайдення велосипеда»

Ви не будете вчити всі ці інструменти тут — достатньо знати про їхнє існування та призначення.

## Категорії інструментів

### 1. Оркестрація (Ядро)

| Інструмент | Що він робить |
|------------|---------------|
| **Kubernetes** | Оркестрація контейнерів (центр усього) |
| **Helm** | Менеджер пакетів для K8s (як apt/yum для K8s) |
| **Kustomize** | Керування конфігурацією K8s без шаблонів |

### 2. Рантайм контейнерів

| Інструмент | Що він робить |
|------------|---------------|
| **containerd** | Галузевий стандарт середовища виконання контейнерів |
| **CRI-O** | Легкий рантайм спеціально для K8s |

### 3. Мережі

| Інструмент | Що він робить |
|------------|---------------|
| **Cilium** | CNI на базі eBPF для мереж та безпеки |
| **Calico** | Популярний CNI для мережевих політик |
| **Flannel** | Проста мережева надбудова (overlay) |
| **Istio** | Service mesh (керування трафіком, безпека) |
| **Linkerd** | Легкий та швидкий service mesh |
| **Envoy** | Сервіс-проксі (основа багатьох service mesh) |

### 4. Спостережуваність (Observability)

| Інструмент | Що він робить |
|------------|---------------|
| **Prometheus** | Збір метрик та сповіщення (alerting) |
| **Grafana** | Візуалізація та дашборди |
| **Jaeger** | Розподілене трасування (tracing) |
| **Fluentd/Fluent Bit** | Збір та пересилання логів |
| **Loki** | Агрегація логів (у стилі Prometheus) |
| **OpenTelemetry** | Уніфікований фреймворк спостережуваності |

### 5. CI/CD та GitOps

| Інструмент | Що він робить |
|------------|---------------|
| **ArgoCD** | Безперервна доставка (CD) у стилі GitOps |
| **Flux** | Набір інструментів для GitOps |
| **Tekton** | Хмарно-нативні CI/CD пайплайни |

### 6. Безпека

| Інструмент | Що він робить |
|------------|---------------|
| **Falco** | Моніторинг безпеки під час виконання (runtime) |
| **Trivy** | Сканування контейнерів на вразливості |
| **OPA/Gatekeeper** | Забезпечення дотримання політик |
| **cert-manager** | Керування сертифікатами |

### 7. Сховища (Storage)

| Інструмент | Що він робить |
|------------|---------------|
| **Rook** | Оркестрація сховищ (Ceph на K8s) |
| **Longhorn** | Розподілене блокове сховище |
| **Velero** | Резервне копіювання та відновлення |

## Що вам насправді потрібно знати

Для сертифікації та більшості вакансій зосередьтеся на:

### Обов'язково знати
- **Kubernetes** — Сама платформа
- **Helm** — Керування пакетами
- **Kustomize** — Керування конфігурацією
- **kubectl** — Основний інструмент командного рядка

### Варто знати (концептуально)
- **Prometheus/Grafana** — Моніторинг
- **Концепції Service mesh** — Керування трафіком
- **Концепції CNI** — Як працює мережа подів
- **Рантайм контейнерів** — containerd, CRI

### Добре просто знати (без глибоких знань)
- **ArgoCD/Flux** — GitOps
- **Реалізації Istio/Linkerd** — Конкретні service mesh
- **OPA/Gatekeeper** — Політики
- **Falco/Trivy** — Сканування безпеки

## Чи знали ви?

- **Ландшафт CNCF налічує понад 1000 проєктів.** Жодна людина не може знати їх усі. Фокусуйтеся на тому, чого вимагає ваша робота.

- **Більшість компаній використовують ~10–20 проєктів CNCF.** Не сотні. Спеціалізація перемагає всезнайство.

- **Саме ядро Kubernetes — це близько 2 мільйонів рядків коду.** Плюс ще мільйони в проєктах екосистеми. Саме тому сертифікації фокусуються на практичному використанні, а не на внутрішній будові коду.

- **Щомісяця до CNCF приєднуються нові проєкти.** Ландшафт постійно розвивається. Навички роботи з ядром K8s залишаються стабільними, а інструментарій навколо нього змінюється.

## Рівні зрілості проєктів екосистеми

```
Стадії проєктів CNCF:

┌──────────┐
│ SANDBOX  │ ─── Рання стадія, експериментальна.
└────┬─────┘     Може бути не готовим до продакшну.
     │
     ▼
┌──────────┐
│INCUBATING│ ─── Впровадження зростає.
└────┬─────┘     Формується стабільна спільнота.
     │
     ▼
┌──────────┐
│ GRADUATED│ ─── Перевірено продакшном.
└──────────┘     Широко впроваджено, стабільно.
```

Для реальних проєктів (продакшну) надавайте перевагу **Graduated** проєктам. Для навчання та експериментів досліджуйте **Incubating** та навіть **Sandbox**.

## Тест

1. **Що таке CNCF?**
   <details>
   <summary>Відповідь</summary>
   Cloud Native Computing Foundation — вендор-нейтральна організація, яка розміщує та керує хмарно-нативними проєктами з відкритим кодом, включаючи Kubernetes. Вона є частиною Linux Foundation.
   </details>

2. **У чому різниця між Helm та Kustomize?**
   <details>
   <summary>Відповідь</summary>
   Helm — це менеджер пакетів із шаблонами (templates), де ви визначаєте змінні. Kustomize працює без шаблонів — ви пишете валідний YAML та накладаєте патчі (patches). Обидва керують конфігураціями Kubernetes, але мають різні підходи.
   </details>

3. **Що означає статус «Graduated» для проєкту CNCF?**
   <details>
   <summary>Відповідь</summary>
   Проєкти статусу Graduated перевірені у великих масштабах і широко впроваджені. Вони продемонстрували стабільність, активне використання та надійне управління спільнотою. Приклади: Kubernetes, Prometheus, Helm.
   </details>

4. **Чому Prometheus та Grafana часто використовують разом?**
   <details>
   <summary>Відповідь</summary>
   Prometheus збирає та зберігає метрики, а також обробляє сповіщення. Grafana забезпечує візуалізацію та створення дашбордів. Prometheus виступає джерелом даних, а Grafana робить їх зрозумілими для людини.
   </details>

## Практична вправа

**Завдання**: Дослідіть ландшафт CNCF.

```bash
# Це вправа на дослідження, а не робота в терміналі

# 1. Відкрийте CNCF Landscape
#    Перейдіть за адресою: https://landscape.cncf.io

# 2. Дослідіть за категоріями
#    - Натисніть "App Definition & Development" → побачите Helm, Kustomize
#    - Натисніть "Observability" → побачите Prometheus, Grafana, Jaeger
#    - Натисніть "Provisioning" → побачите Terraform, Ansible
#    Зверніть увагу, скільки проєктів існує в кожній категорії!

# 3. Відфільтруйте за зрілістю (Maturity)
#    - Натисніть фільтр "Project Maturity"
#    - Оберіть тільки "Graduated"
#    - Це найстабільніші проєкти, перевірені часом
#    - Порахуйте, скільки їх (підказка: близько 20)

# 4. Знайдіть Kubernetes
#    - Скористайтеся пошуком "Kubernetes"
#    - Зверніть увагу, що він у категорії "Orchestration & Management" → "Scheduling & Orchestration"
#    - Натисніть на нього, щоб побачити статистику використання

# 5. Дослідіть один новий проєкт
#    - Оберіть будь-який проєкт, про який ви раніше не чули
#    - Прочитайте опис
#    - Запитайте себе: Яку проблему він вирішує?

# Роздуми:
# - Скільки всього проєктів на ландшафті? (1000+)
# - Чи зможе одна людина вивчити їх усі? (Ні)
# - Що це говорить про потребу в спеціалізації?
```

**Критерії успіху**: Розуміння масштабів екосистеми cloud native та усвідомлення того, чому фокус на важливому є необхідним.

## Підсумок

Екосистема хмарних технологій включає:

**Ядро оркестрації**: Kubernetes, Helm, Kustomize
**Спостережуваність**: Prometheus, Grafana, Jaeger, Fluentd
**Мережі**: Cilium, Calico, Istio, Envoy
**Безпека**: Falco, Trivy, OPA
**CI/CD**: ArgoCD, Flux, Tekton

Ключові висновки:
- CNCF об'єднує понад 1000 проєктів
- Вам не потрібно вчити їх усі
- Фокусуйтеся на тому, чого вимагає ваша сертифікація або робота
- Проєкти статусу Graduated є найбільш стабільними
- Kubernetes — це фундамент; все інше будується поверх нього
