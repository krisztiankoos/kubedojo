## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Пояснити**, що таке контейнери і яку конкретну проблему («працює на моїй машині») вони вирішують
- **Порівняти** контейнери з віртуальними машинами та пояснити, коли використовувати кожен підхід
- **Описати**, як контейнери використовують функції ядра Linux (простори імен, cgroups) для ізоляції застосунків
- **Передбачити**, що станеться при зупинці та перезапуску контейнера (що збережеться, а що — ні)

---

## Чому цей модуль важливий

Контейнери — це будівельні блоки сучасного розгортання застосунків. Перш ніж ви зможете зрозуміти Kubernetes (оркестратор контейнерів), вам потрібно зрозуміти, що таке контейнери та які проблеми вони вирішують.

Цей модуль не про зазубрювання технічних деталей — він про розуміння «навіщо», що змусить все інше мати сенс.

## Проблема, яку вирішують контейнери

### Класична проблема розгортання

```
Розробник: "На моїй машині все працює!"
Експлуатація: "А в продакшні не працює."
Розробник: "На моїй машині Python 3.9, правильні бібліотеки, вірні шляхи..."
Експлуатація: "У продакшні Python 3.7, інші бібліотеки, інші шляхи..."
Усі: 😤
```

Це **проблема узгодженості середовищ**. Застосунки залежать від:
- Версії операційної системи
- Версії середовища виконання (Python, Node, Java)
- Версій бібліотек
- Конфігураційних файлів
- Змінних середовища
- Шляхів до файлів

Коли будь-що з цього відрізняється між розробкою та продакшном — все ламається.

### Традиційні рішення (які не масштабувалися)

**Рішення 1: Детальна документація**
```
README.md:
1. Встановіть Python 3.9.7
2. Запустіть `pip install -r requirements.txt`
3. Встановіть змінні середовища...
4. Налаштуйте шляхи...
(Ніхто це не читає. А коли читають — воно вже застаріле.)
```

**Рішення 2: Віртуальні машини**
```
Передаємо операційну систему цілком:
- Працює узгоджено
- АЛЕ 10 ГБ+ на кожен застосунок
- Хвилини на запуск
- Велике споживання ресурсів
- Важко керувати у великих масштабах
```

### Рішення через контейнери

```
Що, якби ми могли упакувати:
- Застосунок
- Його залежності
- Його конфігурацію
- Все, що йому потрібно для роботи

У легкий, портативний юніт, який працює однаково всюди?

Це і є контейнер.
```

## Контейнери проти Віртуальних Машин

```
┌─────────────────────────────────────────────────────────────┐
│              ВМ ПРОТИ КОНТЕЙНЕРІВ                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ВІРТУАЛЬНІ МАШИНИ                КОНТЕЙНЕРИ                │
│  ┌─────────────────────┐         ┌─────────────────────┐   │
│  │ Дод А │ Дод Б │ Дод В│         │ Дод А │ Дод Б │ Дод В│   │
│  ├───────┼───────┼──────┤         ├───────┼───────┼──────┤   │
│  │Гостьова│Гостьова│Гостьова│     │Рантайм контейнерів   │   │
│  │ОС     │ОС     │ОС    │         │(containerd)          │   │
│  ├───────┴───────┴──────┤         ├──────────────────────┤   │
│  │    Гіпервізор        │         │    Хостова ОС        │   │
│  ├──────────────────────┤         ├──────────────────────┤   │
│  │    Хостова ОС        │         │    Залізо            │   │
│  ├──────────────────────┤         └──────────────────────┘   │
│  │    Залізо            │                                    │
│  └──────────────────────┘                                    │
│                                                             │
│  Кожна ВМ: повна копія ОС         Контейнери: ділять ядро    │
│  Розмір: Гігабайти                хостової ОС                │
│  Запуск: Хвилини                  Розмір: Мегабайти          │
│  Ізоляція: Рівень заліза          Запуск: Секунди            │
│                                   Ізоляція: Рівень процесів  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Ключові відмінності

| Аспект | Віртуальна Машина | Контейнер |
|--------|-------------------|-----------|
| Розмір | Гігабайти | Мегабайти |
| Швидкість запуску | Хвилини | Секунди |
| ОС | Повна гостьова ОС на кожну ВМ | Спільне хостове ядро |
| Ізоляція | Апаратна віртуалізація | Ізоляція процесів |
| Портативність | Формати образів ВМ варіюються | Універсальні образи контейнерів |
| Щільність | ~10–20 ВМ на сервер | ~100-ні контейнерів на сервер |

## Як працюють контейнери

Контейнери використовують функції ядра Linux для створення ізольованих середовищ:

### 1. Простори імен (Namespaces — Ізоляція)

Namespaces змушують процес думати, що він має власну систему:

```
┌─────────────────────────────────────────────────────────────┐
│              LINUX NAMESPACES                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Namespace    Що він ізолює                                 │
│  ─────────────────────────────────────────────────────────  │
│  PID          ID процесів (контейнер бачить PID 1)          │
│  NET          Мережеві інтерфейси, IP, порти                │
│  MNT          Точки монтування файлової системи             │
│  UTS          Ім'я хоста та домен                           │
│  IPC          Міжпроцесна взаємодія                         │
│  USER         ID користувачів та груп                       │
│                                                             │
│  Результат: Процес думає, що він один у системі             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2. Контрольні групи (Control Groups / cgroups — Ліміти)

cgroups обмежують обсяг ресурсів, які може використовувати контейнер:

```
Контейнер А: макс 512МБ RAM, 0.5 CPU
Контейнер Б: макс 1ГБ RAM, 1 CPU
Контейнер В: макс 256МБ RAM, 0.25 CPU

Кожен контейнер обмежений, не може "об'їдати" інших
```

### 3. Union Filesystems (Шаруваті образи)

Образи контейнерів будуються шарами:

```
┌─────────────────────────────────────────────────────────────┐
│              ШАРИ ОБРАЗУ КОНТЕЙНЕРА                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────┐  ← Код вашого дод. │
│  │ Шар 4: COPY app.py /app             │     (крихітний)   │
│  ├─────────────────────────────────────┤                   │
│  │ Шар 3: pip install flask            │  ← Залежності     │
│  ├─────────────────────────────────────┤     (кешовані)    │
│  │ Шар 2: apt-get install python3      │  ← Рантайм        │
│  ├─────────────────────────────────────┤     (кешований)   │
│  │ Шар 1: Базова ОС Ubuntu 22.04       │  ← Базова ОС      │
│  └─────────────────────────────────────┘     (спільна)     │
│                                                             │
│  Переваги:                                                  │
│  - Шари спільні для різних образів                          │
│  - Потрібно перебудовувати лише змінені шари                │
│  - Ефективне зберігання та передача                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Образи та Реєстри

### Що таке Образ Контейнера (Container Image)?

Образ — це шаблон тільки для читання, що містить:
- Мінімальну операційну систему (часто Alpine Linux, ~5МБ)
- Код вашого застосунку
- Залежності (бібліотеки, рантайми)
- Конфігурацію

Уявіть це як **клас** у програмуванні — це креслення.

### Що таке Контейнер?

Контейнер — це **працюючий екземпляр** образу.

Уявіть це як **об'єкт** — це втілення креслення.

```
Образ → Контейнер
(Клас → Об'єкт)
(Креслення → Будівля)
(Рецепт → Страва)
```

### Реєстри контейнерів (Container Registries)

Образи зберігаються в реєстрах:

```
┌─────────────────────────────────────────────────────────────┐
│              РЕЄСТРИ КОНТЕЙНЕРІВ                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Публічні реєстри:                                          │
│  ┌────────────────────────────────────────────┐            │
│  │ Docker Hub        hub.docker.com           │            │
│  │ GitHub Container  ghcr.io                  │            │
│  │ Quay.io          quay.io                   │            │
│  └────────────────────────────────────────────┘            │
│                                                             │
│  Хмарні реєстри:                                            │
│  ┌────────────────────────────────────────────┐            │
│  │ AWS ECR          *.dkr.ecr.*.amazonaws.com │            │
│  │ Google GCR       gcr.io                    │            │
│  │ Azure ACR        *.azurecr.io              │            │
│  └────────────────────────────────────────────┘            │
│                                                             │
│  Використання:                                              │
│  docker pull nginx              # З Docker Hub             │
│  docker pull gcr.io/project/app # З Google                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Іменування образів

Образи контейнерів мають специфічний формат назви:

```
[registry/][namespace/]repository[:tag]

Приклади:
nginx                           # Docker Hub, library/nginx:latest
nginx:1.25                      # Docker Hub, конкретна версія
mycompany/myapp:v1.0.0         # Docker Hub, власний простір імен
gcr.io/myproject/myapp:latest  # Google Container Registry
ghcr.io/username/app:sha-abc123 # GitHub Container Registry
```

### Теги (Tags) — це важливо

```
nginx:latest     # Найновіше, що є (непередбачувано!)
nginx:1.25       # Конкретна версія (краще)
nginx:1.25.3     # Точна версія (найкраще для продакшну)

Правило: Ніколи не використовуйте :latest у продакшні
```

## Чи знали ви?

- **Контейнери — це не новинка.** Unix мав `chroot` ще у 1979 році. FreeBSD Jails з'явилися у 2000-му. Linux Containers (LXC) — у 2008-му. Docker просто зробив це зручним та доступним у 2013-му.

- **Більшість контейнерів використовують Alpine Linux** як основу. Він важить лише 5МБ. Порівняйте з Ubuntu (~70МБ) або повною ВМ (гігабайти).

- **Образи контейнерів іммутабельні (незмінні).** Після створення вони ніколи не змінюються. Це ключ до відтворюваності.

- **Кит Docker** має ім'я — Moby Dock. Він несе контейнери (морські вантажні контейнери) на своїй спині.

## Поширені помилки розуміння

| Міф | Реальність |
|-----|------------|
| "Контейнери — це легкі ВМ" | Контейнери ділять ядро хоста. ВМ мають власне гостьове ядро. Це принципово різні речі. |
| "Контейнери менш безпечні" | Інша модель загроз, але не гірша. Правильно налаштовані контейнери дуже безпечні. |
| "Docker = контейнери" | Docker популяризував їх, але це не єдиний варіант. containerd, CRI-O, Podman — все це працює. |
| "Контейнери повністю замінять ВМ" | ВМ все ще цінні для різних ядер ОС, сильної ізоляції та старих застосунків. |

## Аналогія: Вантажні контейнери

Назва «контейнер» походить від морських вантажних перевезень:

```
До вантажних контейнерів (1950-ті):
- Кожен товар пакувався по-різному
- Ручне завантаження/розвантаження
- Товари пошкоджувалися при транзиті
- Кораблі спеціалізувалися під типи вантажів
- Повільно, дорого, ненадійно

Після вантажних контейнерів:
- Стандартний розмір для всього
- Автоматизоване завантаження
- Вміст захищений
- Будь-який корабель може нести будь-який контейнер
- Швидко, дешево, надійно

Програмні контейнери:
- Стандартний формат для будь-якого застосунку
- Автоматизоване розгортання
- Захист від відмінностей середовищ
- Працює всюди, де є рантайм контейнерів
- Швидко, портативно, надійно
```

## Тест

1. **Яку проблему першочергово вирішують контейнери?**
   <details>
   <summary>Відповідь</summary>
   Узгодженість середовищ (environment consistency) — гарантування того, що застосунок працює однаково під час розробки, тестування та у продакшні. Фраза "на моїй машині працює" змінюється на "в контейнері працює".
   </details>

2. **Яка ключова різниця між контейнером та віртуальною машиною?**
   <details>
   <summary>Відповідь</summary>
   Контейнери ділять ядро операційної системи хоста, тоді як ВМ мають власну гостьову ОС. Це робить контейнери набагато меншими (МБ проти ГБ), швидшими у запуску (секунди проти хвилин) та ефективнішими (вища щільність на сервер).
   </details>

3. **Які дві функції ядра Linux роблять контейнери можливими?**
   <details>
   <summary>Відповідь</summary>
   Простори імен (Namespaces — для ізоляції, щоб процес думав, що він один у системі) та Контрольні групи (Control Groups / cgroups — для обмеження ресурсів: CPU, пам'яті тощо).
   </details>

4. **У чому різниця між образом контейнера та самим контейнером?**
   <details>
   <summary>Відповідь</summary>
   Образ — це шаблон тільки для читання (як клас або креслення). Контейнер — це запущений екземпляр образу (як об'єкт або побудована будівля). Можна запустити багато контейнерів з одного образу.
   </details>

## Практична вправа

**Завдання**: Дослідіть ізоляцію контейнера (якщо у вас встановлено Docker).

```bash
# 1. Запустіть контейнер та дослідіть його ізольований погляд на світ
docker run -it --rm alpine sh

# Всередині контейнера ви побачите:
# - PID 1 — це ваша оболонка (ізольований PID namespace)
# - Тільки файлову систему контейнера (ізольований mount namespace)
# - Власне ім'я хоста (ізольований UTS namespace)

# Перевірте процеси - ви бачите тільки процеси контейнера
ps aux

# Перевірте ім'я хоста
hostname

# Перевірте файлову систему
ls /

# Вийдіть з контейнера
exit

# 2. Порівняйте з вашим хостом
# На вашому комп'ютері запустіть:
ps aux | wc -l    # Сотні/тисячі процесів
hostname          # Назва вашої машини
ls /              # Повна файлова система хоста

# 3. Подивіться на контейнер ззовні
# В одному терміналі запустіть контейнер:
docker run -it --rm --name mycontainer alpine sleep 1000

# В іншому терміналі подивіться на нього з позиції хоста:
docker exec mycontainer ps aux  # Обмежений вигляд зсередини
ps aux | grep sleep             # Видно з хоста!

# Контейнер думає, що він один, але він просто ізольований.
```

**Критерії успіху**: Розуміння того, що контейнери забезпечують ізоляцію, а не віртуалізацію — процеси все ще виконуються на ядрі хоста.

## Підсумок

Контейнери вирішують проблему узгодженості середовищ шляхом пакування:
- Коду застосунку
- Його залежностей
- Конфігурації
- Всього необхідного для роботи

Це досягається завдяки:
- **Namespaces**: Ізоляція процесів
- **Control groups**: Обмеження ресурсів
- **Union filesystems**: Ефективні шаруваті образи

Контейнери є:
- **Легкими**: Мегабайти, а не гігабайти
- **Швидкими**: Секунди на запуск, а не хвилини
- **Портативними**: Працюють всюди, де є рантайм контейнерів
- **Іммутабельними**: Створюються один раз, не змінюються
