## Чому цей модуль важливий

Ви вже знаєте, що таке контейнери. Ви вмієте збирати та запускати їх за допомогою Docker. Але Docker запускає контейнери на ОДНІЙ машині. Що станеться, коли вам знадобиться:
- Сотні контейнерів?
- Висока доступність (без простоїв)?
- Автоматичне масштабування?
- Багато фізичних чи віртуальних машин?

Ось тут на сцену виходить Kubernetes. Цей модуль дає загальну картину перед тим, як зануритися в деталі.

## Проблема: Контейнери у великому масштабі

Docker чудовий для запуску кількох контейнерів на вашому ноутбуці. Але для продакшну цього замало:

```
┌─────────────────────────────────────────────────────────────┐
│              ОБМЕЖЕННЯ ОДНІЄЇ МАШИНИ                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Ваш Docker-хост:                                           │
│  ┌────────────────────────────────────────────────────┐    │
│  │  🐳 Контейнер 1  🐳 Контейнер 2  🐳 Контейнер 3    │    │
│  │  🐳 Контейнер 4  🐳 Контейнер 5  🐳 Контейнер 6    │    │
│  └────────────────────────────────────────────────────┘    │
│                                                             │
│  Проблеми:                                                  │
│  ❌ Машина "вмирає" = вмирають УСІ контейнери               │
│  ❌ Потрібно більше потужності? Купуйте більшу машину       │
│  ❌ Немає автоматичного відновлення                         │
│  ❌ Ручне балансування навантаження                         │
│  ❌ Оновлення потребують зупинки сервісу                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Що потрібно для продакшну

```
✅ Запуск контейнерів на багатьох машинах
✅ Автоматичний перезапуск при збоях контейнерів
✅ Автоматичне розміщення (на якій машині є вільне місце?)
✅ Балансування навантаження між контейнерами
✅ Оновлення без простоїв (rolling updates)
✅ Масштабування вгору/вниз залежно від попиту
✅ Самолікування (self-healing), коли щось ламається
```

Це називається **оркестрація контейнерів**, і Kubernetes є галузевим стандартом для цього.

## Що таке Kubernetes?

Kubernetes (K8s) — це платформа з відкритим кодом для оркестрації контейнерів. Він:

1. **Керує розгортанням контейнерів** на багатьох машинах.
2. **Гарантує бажаний стан** — якщо щось зламалося, K8s це виправляє.
3. **Керує мережею** — контейнери знаходять один одного та спілкуються.
4. **Керує сховищами** — зберігає дані для застосунків, які цього потребують.
5. **Автоматизує операції** — масштабування, оновлення, відновлення.

### Аналогія: Робота аеропорту

```
┌─────────────────────────────────────────────────────────────┐
│              KUBERNETES ЯК ДИСПЕТЧЕРСЬКА АЕРОПОРТУ          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Диспетчерська вежа = Kubernetes Control Plane              │
│  - Призначає літаки до гейтів (планує поди на вузли)        │
│  - Моніторить усю активність (відстежує стан кластера)      │
│  - Реагує на інциденти (перезапускає контейнери, що впали)  │
│                                                             │
│  Гейти/Злітні смуги = Worker Nodes (Робочі вузли)           │
│  - Фізична інфраструктура, де відбувається робота           │
│  - Вежа призначає завдання, гейти їх виконують              │
│                                                             │
│  Літаки = Pods (Поди / контейнери)                          │
│  - Прибувають, вилітають, обслуговуються                    │
│  - Вежа відстежує статус кожного літака                     │
│                                                             │
│  Авіалінії = Namespaces (Простори імен)                     │
│  - Авіакомпанія А використовує гейти 1-10                   │
│  - Авіакомпанія Б використовує гейти 11-20                  │
│  - Ізоляція між різними орендарями (tenants)                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Компоненти Control Plane

| Компонент | Що він робить |
|-----------|---------------|
| **API Server** | "Вхідні двері" K8s. Усі команди проходять через нього. |
| **etcd** | База даних, де зберігається стан усього кластера. |
| **Scheduler** | Вирішує, на якому вузлі запускати кожен под. |
| **Controller Manager** | Стежить, щоб бажаний стан збігався з реальним. |

### Компоненти Worker Node

| Компонент | Що він робить |
|-----------|---------------|
| **kubelet** | Агент на кожному вузлі, що керує подами. |
| **Container Runtime** | Безпосередньо запускає контейнери (наприклад, containerd). |
| **kube-proxy** | Відповідає за мережеву роботу сервісів. |

## Попередній огляд основних концепцій

### Поди (Pods)
Найменша одиниця розгортання. Зазвичай один под містить один контейнер, іноді — декілька пов'язаних.

```yaml
# Ви не запускаєте контейнери напряму — ви створюєте Поди
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.25
```

### Деплойменти (Deployments)
Керують групою однакових подів. Відповідають за оновлення та масштабування.

```yaml
# "Я хочу, щоб завжди працювало 3 поди nginx"
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25
```

### Сервіси (Services)
Забезпечують стабільну мережеву адресу для подів. Поди можуть зникати й з'являтися, а Сервіси дають постійний доступ до них.

```yaml
# "Зроби мої поди nginx доступними на порту 80"
apiVersion: v1
kind: Service
metadata:
  name: nginx
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
```

## Чому не просто Docker?

| Функція | Docker (один хост) | Kubernetes |
|---------|--------------------|------------|
| Багато вузлів | ❌ | ✅ |
| Автомасштабування | ❌ | ✅ |
| Самолікування | ❌ | ✅ |
| Оновлення без простоїв | Вручну | ✅ Автоматично |
| Балансування навантаження | Вручну | ✅ Вбудовано |
| Виявлення сервісів | Вручну | ✅ Вбудовано |
| Керування секретами | ❌ | ✅ |
| Ліміти ресурсів | На контейнер | На весь кластер |

Docker — для запуску контейнерів. Kubernetes — для експлуатації контейнерів у великому масштабі.

## Де працює Kubernetes?

### Хмарні керовані сервіси (Найпростіше)
- **AWS**: EKS (Elastic Kubernetes Service)
- **Google Cloud**: GKE (Google Kubernetes Engine)
- **Azure**: AKS (Azure Kubernetes Service)

Хмарні провайдери керують Control Plane. Ви просто запускаєте свої програми.

### Власне керування (Self-Managed)
- **kubeadm**: Офіційний інструмент для встановлення K8s.
- **k3s**: Легкий K8s (чудово для межових обчислень / IoT).
- **OpenShift**: Корпоративна версія K8s від Red Hat.

Ви керуєте всім самостійно.

### Локальна розробка
- **kind**: Kubernetes всередині Docker.
- **minikube**: Локальний K8s у віртуальній машині або контейнері.
- **Docker Desktop**: Має вбудовану опцію ввімкнення K8s.

Для навчання та розробки.

## Чи знали ви?

- **«Kubernetes» перекладається з грецької як «керманич»** (той, хто веде корабель). Логотип — це штурвал із 7 спицями, що символізують 7 оригінальних розробників.

- **K8s — це нумеронім.** K-[8 літер]-s. Як i18n (internationalization) або a11y (accessibility).

- **Google запускає все в Kubernetes.** Gmail, Пошук, YouTube — усе працює на Borg (попередник K8s) або на самому K8s.

- **Найбільший відомий кластер K8s** налічує 15 000 вузлів, на яких працює понад 300 000 подів (Alibaba).

## Поширені помилки розуміння

| Міф | Реальність |
|-----|------------|
| "K8s замінює Docker" | K8s оркеструє контейнери. Ви все ще використовуєте Docker для збірки образів. |
| "K8s лише для гігантських компаній" | Малі стартапи також використовують K8s. Керовані сервіси роблять його доступним. |
| "K8s занадто складний" | K8s складний, але керовані сервіси беруть більшість складнощів на себе. |
| "K8s вирішує всі проблеми" | K8s — це інфраструктура. Вам все ще потрібно правильно проєктувати застосунки. |

## Тест

1. **Яку проблему вирішує Kubernetes, з якою Docker не впорається самотужки?**
   <details>
   <summary>Відповідь</summary>
   Запуск контейнерів на багатьох машинах з автоматичним плануванням, самолікуванням, масштабуванням, балансуванням навантаження та оновленням без простоїв. Docker сам по собі керує контейнерами лише на одному хості.
   </details>

2. **Який зв'язок між Подом (Pod) та Контейнером?**
   <details>
   <summary>Відповідь</summary>
   Под — це найменша одиниця розгортання в Kubernetes, яка містить один або декілька контейнерів. Ви не розгортаєте контейнери напряму — ви створюєте поди. Контейнери в одному поді ділять мережу та сховище.
   </details>

3. **Що робить Kubernetes Scheduler?**
   <details>
   <summary>Відповідь</summary>
   Scheduler вирішує, на якому робочому вузлі (worker node) має працювати новий под. Він враховує наявність ресурсів, обмеження, правила спорідненості (affinity) та інші фактори для оптимального розміщення.
   </details>

4. **Навіщо використовувати керований сервіс Kubernetes (EKS, GKE, AKS)?**
   <details>
   <summary>Відповідь</summary>
   Керовані сервіси беруть на себе роботу з Control Plane (API server, etcd, scheduler, controllers). Вам не потрібно турбуватися про доступність, оновлення чи обслуговування керуючого рівня. Ви просто запускаєте своє навантаження.
   </details>

## Практична вправа

**Завдання**: Дослідіть кластер Kubernetes (попередній перегляд того, що буде далі).

```bash
# Якщо у вас уже є запущений кластер (kind, minikube або інший):

# 1. Подивіться на вузли вашого кластера
kubectl get nodes
# Вивід покаже машини у вашому кластері

# 2. Подивіться на запущені системні компоненти
kubectl get pods -n kube-system
# Це компоненти, завдяки яким K8s працює

# 3. Подивіться на всі простори імен (як теки для ресурсів)
kubectl get namespaces

# 4. Створіть щось просте
kubectl run hello --image=nginx --restart=Never
kubectl get pods
# Ви щойно створили Под!

# 5. Подивіться, що Kubernetes знає про нього
kubectl describe pod hello
# Багато інформації про планування, контейнери, події

# 6. Приберіть за собою
kubectl delete pod hello

# Не хвилюйтеся, якщо зараз це здається складним — ви всьому навчитеся.
# Мета — просто побачити K8s у дії.
```

**Ще немає кластера?** Це нормально! Трек «Основи Kubernetes» проведе вас через процес його створення. Це був лише анонс.

**Критерії успіху**: Розуміння того, що Kubernetes надає API для керування контейнерами на багатьох машинах одночасно.

## Підсумок

Kubernetes — це платформа для оркестрації контейнерів, яка:

- **Планує** (schedules) роботу контейнерів на багатьох машинах.
- **Самовідновлюється** (self-heals), перезапускаючи контейнери, що впали.
- **Масштабується** залежно від попиту.
- **Балансує навантаження** трафіку до контейнерів.
- **Оновлює** ПЗ без зупинки сервісу.
- **Керує** мережею та сховищами даних.

Ключові концепції:
- **Кластер**: Control plane + робочі вузли (worker nodes).
- **Под (Pod)**: Найменша одиниця розгортання.
- **Деплоймент (Deployment)**: Керує реплікованими подами.
- **Сервіс (Service)**: Стабільна мережева адреса для подів.
