## Що ви зможете робити після цього модуля

Після цього модуля ви зможете:
- **Порівняти** монолітну та мікросервісну архітектури з конкретними компромісами кожної
- **Пояснити**, чому функції Kubernetes (Сервіси, Деплойменти, масштабування) відповідають потребам мікросервісів
- **Визначити**, коли моноліт — насправді правильний вибір (не все має бути мікросервісами)
- **Описати** виклики, які створюють мікросервіси (мережева складність, розподілений дебагінг, узгодженість даних)

---

## Чому цей модуль важливий

Kubernetes розроблений для розподілених, контейнеризованих застосунків — зазвичай це мікросервіси. Розуміння еволюції від монолітів до мікросервісів допомагає вам:

1. Розуміти, навіщо існують певні функції Kubernetes
2. Приймати кращі архітектурні рішення
3. Розуміти, коли мікросервіси доречні (а коли ні)
4. Розмовляти мовою сучасної архітектури ПЗ

## Моноліт

### Що таке моноліт?

Монолітний застосунок — це єдиний юніт розгортання, що містить весь функціонал:

```
┌─────────────────────────────────────────────────────────────┐
│              МОНОЛІТНИЙ ЗАСТОСУНОК                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │                 MyApp.jar (або .exe)                │   │
│  │  ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐   │   │
│  │  │ Модуль  │ │ Модуль  │ │ Модуль  │ │ Модуль  │   │   │
│  │  │ Користу-│ │ Товарів │ │Замовлень│ │ Оплати  │   │   │
│  │  │ вачів   │ │         │ │         │ │         │   │   │
│  │  └────┬────┘ └────┬────┘ └────┬────┘ └────┬────┘   │   │
│  │       │           │           │           │         │   │
│  │       └───────────┴───────────┴───────────┘         │   │
│  │                       │                             │   │
│  │              ┌────────┴────────┐                    │   │
│  │              │   База Даних    │                    │   │
│  │              │    (спільна)    │                    │   │
│  │              └─────────────────┘                    │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  Розгортання: Одним цілим                                  │
│  Масштабування: Все або нічого                             │
│  База даних: Спільна для всіх модулів                      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Переваги моноліту

```
✅ Простий у розробці на початкових етапах
✅ Легко тестувати (один застосунок)
✅ Просте розгортання (один файл/артефакт)
✅ Легко налагоджувати (один процес)
✅ Відсутність мережевих затримок між компонентами
✅ ACID-транзакції реалізуються просто
```

### Виклики моноліту (у великому масштабі)

```
❌ Зміни потребують повного перерозгортання всього застосунку
❌ Тривалі цикли збірки та тестування
❌ Масштабування означає масштабування всього відразу
❌ Технологічний вибір впливає на весь застосунок
❌ Одна помилка може "покласти" все
❌ Складна координація між командами
❌ Кодова база стає некерованою
```

## Мікросервісний підхід

### Що таке мікросервіси?

Мікросервіси розділяють застосунок на маленькі, незалежні сервіси:

```
┌─────────────────────────────────────────────────────────────┐
│              МІКРОСЕРВІСНА АРХІТЕКТУРА                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌───────────┐   ┌───────────┐   ┌───────────┐            │
│  │  Сервіс   │   │  Сервіс   │   │  Сервіс   │            │
│  │ Користу-  │   │  Товарів  │   │Замовлень  │            │
│  │ вачів     │   │           │   │           │            │
│  │  ┌─────┐  │   │  ┌─────┐  │   │  ┌─────┐  │            │
│  │  │ БД  │  │   │  │ БД  │  │   │  │ БД  │  │            │
│  │  └─────┘  │   │  └─────┘  │   │  └─────┘  │            │
│  └─────┬─────┘   └─────┬─────┘   └─────┬─────┘            │
│        │               │               │                    │
│        └───────────────┼───────────────┘                    │
│                        │                                    │
│                ┌───────┴───────┐                           │
│                │  API Gateway  │                           │
│                └───────────────┘                           │
│                        │                                    │
│                     Клієнти                                 │
│                                                             │
│  Розгортання: Кожен сервіс незалежно                       │
│  Масштабування: Окремо для кожного сервісу за потребою     │
│  База даних: Кожен сервіс володіє своїми даними            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Переваги мікросервісів

```
✅ Незалежне розгортання
✅ Масштабування окремих сервісів
✅ Технологічне різноманіття (інструмент під конкретну задачу)
✅ Ізоляція збоїв (один сервіс впав ≠ все впало)
✅ Автономність команд
✅ Менші, фокусовані кодові бази
```

### Виклики мікросервісів

```
❌ Мережева складність
❌ Збої в розподілених системах
❌ Труднощі з узгодженістю даних (data consistency)
❌ Тестування стає складнішим
❌ Операційна складність
❌ Налагодження (debugging) між сервісами
❌ Потреба в потужній інфраструктурі
```

## Коли що обирати

### Моноліт часто кращий, коли:

```
✅ Мала команда (< 10 розробників)
✅ Новий продукт / фаза стартапу
✅ Проста предметна область (domain)
✅ Стислі терміни
✅ Невідомі або мінливі вимоги
✅ Одна ціль розгортання
```

### Мікросервіси мають сенс, коли:

```
✅ Велика компанія, багато команд
✅ Потреба в незалежному масштабуванні компонентів
✅ Різні технологічні вимоги для різних модулів
✅ Високі вимоги до доступності (high availability)
✅ Потреба в дуже частих релізах
✅ Чіткі межі між доменами
```

### Реальність: Типовий шлях

```
┌─────────────────────────────────────────────────────────────┐
│              ТИПОВИЙ ШЛЯХ РОЗВИТКУ                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Стадія 1: Початок з Моноліту                               │
│  ├── Швидка розробка                                        │
│  ├── Проста експлуатація                                    │
│  └── Вивчення предметної області                            │
│                                                             │
│  Стадія 2: Зростання (Все ще моноліт)                       │
│  ├── Команда росте                                          │
│  ├── Функціонал накопичується                               │
│  └── З'являються "больові точки"                            │
│                                                             │
│  Стадія 3: Стратегічне розділення                           │
│  ├── Виділення сервісів там, де це найбільше потрібно       │
│  ├── Чіткі межі відповідальності                            │
│  └── Поступова міграція                                     │
│                                                             │
│  НЕ ВАРТО: Починати з мікросервісів для нового продукту     │
│  НЕ ВАРТО: Розділяти без чітких меж доменів                 │
│  НЕ ВАРТО: Мікросервіси для маленьких команд                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Як Kubernetes допомагає мікросервісам

Kubernetes вирішує операційні виклики мікросервісів:

| Виклик | Рішення в Kubernetes |
|--------|----------------------|
| Виявлення сервісів | Services, DNS |
| Балансування навантаження | Services, Ingress |
| Масштабування | Deployments, HPA |
| Конфігурація | ConfigMaps, Secrets |
| Моніторинг стану | Probes (Liveness/Readiness) |
| Оновлення без простоїв | Deployments (RollingUpdate) |
| Відмовостійкість | ReplicaSets, самолікування |
| Керування ресурсами | Requests/Limits |

```
Без Kubernetes:
├── Ручна реєстрація сервісів
├── Кастомна конфігурація балансувальників
├── Скрипти для розгортання "на колінці"
├── Надія, що воно само перезапуститься при збої
└── Конфігураційні файли повсюди

З Kubernetes:
├── Автоматичне виявлення сервісів
├── Вбудоване балансування навантаження
├── Декларативне розгортання
├── Самолікування "з коробки"
└── Централізоване керування конфігурацією
```

## Патерни мікросервісів

### API Gateway (Шлюз API)

```
                    ┌───────────────┐
                    │  API Gateway  │
                    └───────┬───────┘
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
      ┌─────────┐     ┌─────────┐     ┌─────────┐
      │ Сервіс  │     │ Сервіс  │     │ Сервіс  │
      │    А    │     │    Б    │     │    В    │
      └─────────┘     └─────────┘     └─────────┘

Призначення:
- Єдина точка входу
- Автентифікація / Авторизація
- Обмеження частоти запитів (Rate limiting)
- Маршрутизація запитів

У K8s: Ingress Controller або спеціалізований шлюз
```

### Service Mesh (Мережа сервісів)

```
      ┌─────────────────────────────────────────┐
      │             Service Mesh                │
      │  ┌────────┐        ┌────────┐          │
      │  │Сервіс А│◄──────►│Сервіс Б│          │
      │  │┌─────┐ │        │┌─────┐ │          │
      │  ││Proxy│ │        ││Proxy│ │          │
      │  │└─────┘ │        │└─────┘ │          │
      │  └────────┘        └────────┘          │
      └─────────────────────────────────────────┘

Призначення:
- Керування трафіком
- Безпека (mTLS)
- Спостережуваність

Приклади: Istio, Linkerd
```

### Sidecar Pattern (Патерн "Коляска")

```
      ┌─────────────────────────────────────────┐
      │                 Pod                     │
      │  ┌─────────────┐  ┌─────────────┐      │
      │  │  Основний   │  │   Sidecar   │      │
      │  │ застосунок  │◄►│   (Proxy)   │      │
      │  └─────────────┘  └─────────────┘      │
      └─────────────────────────────────────────┘

Призначення:
- Додавання функціоналу без зміни коду додатку
- Логування, моніторинг, безпека

K8s: Багатоконтейнерні поди
```

## Чи знали ви?

- **Amazon був піонером мікросервісів.** Їхній наказ від 2002 року зобов'язував усі команди спілкуватися виключно через API. Це згодом стало фундаментом для створення AWS.

- **Netflix має понад 1000 мікросервісів.** Вони також створили багато інструментів, що зараз є частиною екосистеми cloud native (Zuul, Eureka, Hystrix).

- **Правило "двох піц"** стверджує, що команди мають бути достатньо маленькими, щоб їх можна було нагодувати двома піцами. Це ідеально підходить для володіння конкретним мікросервісом.

- **Моноліти теж масштабуються.** Shopify обробляє величезний трафік за допомогою монолітного застосунку на Rails. Вибір архітектури залежить від контексту.

## Поширені помилки розуміння

| Міф | Реальність |
|-----|------------|
| "Мікросервіси завжди кращі" | Вони додають складність. Часто це занадто для малих команд/продуктів. |
| "Моноліти не масштабуються" | Ще й як! Shopify та Stack Overflow це доводять. Різні компроміси. |
| "Мікросервіси = обов'язково Kubernetes" | Моноліти теж можна запускати в K8s. K8s — це інфраструктура, а не архітектура. |
| "Кожному мікросервісу потрібна окрема БД" | Це гарний патерн, але не догма. Спільні БД іноді допустимі. |

## Тест

1. **Яка основна перевага мікросервісів над монолітами?**
   <details>
   <summary>Відповідь</summary>
   Незалежне розгортання та масштабування. Кожен сервіс може розроблятися, розгортатися та масштабуватися окремою командою. Це прискорює цикл релізів та покращує використання ресурсів.
   </details>

2. **Коли моноліт буде кращим вибором, ніж мікросервіси?**
   <details>
   <summary>Відповідь</summary>
   Малі команди, нові продукти (MVP), прості предметні області, невідомі вимоги, дуже стислі терміни. Мікросервіси додають операційну складність, яка не виправдана без великого масштабу.
   </details>

3. **Як саме Kubernetes допомагає з мікросервісами?**
   <details>
   <summary>Відповідь</summary>
   K8s забезпечує виявлення сервісів (DNS), балансування навантаження, масштабування, керування конфігураціями, моніторинг стану, оновлення без простоїв та самолікування — все те, що стає критично складним при великій кількості сервісів.
   </details>

4. **Що таке service mesh?**
   <details>
   <summary>Відповідь</summary>
   Це інфраструктурний шар для взаємодії між сервісами. Він бере на себе керування трафіком, безпеку (mTLS) та спостережуваність. Реалізується зазвичай через sidecar-проксі. Приклади: Istio, Linkerd.
   </details>

## Вправа для роздумів

Цей модуль присвячений архітектурним концепціям, тому вправи в CLI немає. Натомість подумайте над цими питаннями:

**1. Згадайте застосунки, якими ви користувалися або які створювали:**
- Це були моноліти чи мікросервіси?
- Які ознаки на це вказували? (Частота релізів, структура команд, патерни масштабування)

**2. Уявіть гіпотетичний сайт електронної комерції:**
- Які сервіси ви б виділили окремо? (Користувачі, Товари, Замовлення, Оплати, Склад)
- Які з них потребували б незалежного масштабування? (Пошук товарів? Обробка оплат?)
- Якими даними володів би кожен сервіс?

**3. Оцініть ситуацію:**
> Стартап із 3 людей створює новий продукт. Вони хочуть одразу робити мікросервіси, "щоб бути сучасними".
- Чи це гарна ідея? (Скоріш за все, ні)
- Що б ви порадили? (Спочатку моноліт, розділяти на сервіси лише тоді, коли з'являться реальні проблеми)
- Чому?

**4. Дослідіть одну компанію:**
- Подивіться, як Netflix, Amazon або Spotify підходять до мікросервісів.
- З якими викликами вони зіткнулися?
- Чи підійшов би їхній підхід для меншої компанії?

Ці запитання готують вас до прийняття архітектурних рішень у кар'єрі, а не просто до складання іспитів.

## Підсумок

**Моноліти**:
- Єдиний юніт розгортання.
- Простіше розробляти й обслуговувати на початку.
- Масштабуються тільки цілком.
- Виклики з'являються разом із ростом.

**Мікросервіси**:
- Незалежні сервіси.
- Масштабуються та розгортаються окремо.
- Гнучкість у виборі технологій.
- Висока операційна складність.

**Ключові висновки**:
- Починайте просто (часто з моноліту).
- Розділяйте тоді, коли виникають реальні труднощі.
- Kubernetes робить патерни мікросервісів можливими в реальному житті.
- Архітектура має відповідати команді та потребам продукту.
