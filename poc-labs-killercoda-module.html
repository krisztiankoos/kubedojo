What is a Pod?

A Pod is the smallest deployable unit in Kubernetes. It represents a single instance of a running process in your cluster. A Pod encapsulates one or more containers, storage resources, a unique network IP, and options that govern how the container(s) should run.

Pods are designed to support co-located, co-managed helper programs such as content management systems, file and data loaders, local cache managers, and so on. The containers within a Pod share the same network namespace (they can reach each other on localhost) and can share volumes.

💡
CKA Exam Tip: You will almost never create standalone Pods in production. On the exam, however, you will frequently create pods directly with kubectl run for quick tasks. Know both approaches.
yaml 
apiVersion: v1
kind: Pod
metadata:
  name: nginx-pod
  labels:
    app: nginx
    tier: frontend
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80
    resources:
      requests:
        memory: "64Mi"
        cpu: "100m"
      limits:
        memory: "128Mi"
        cpu: "250m"

Pod Lifecycle

A Pod goes through a well-defined lifecycle from creation to termination. Understanding these phases is essential for debugging and for the CKA exam.

+-----------+ | Pending | Pod accepted, images pulling +-----+-----+ | +-----v-----+ | Running | At least one container running +-----+-----+ | +-----------+-----------+ | | +-----v-----+ +-----v-----+ | Succeeded | | Failed | | (exit 0) | | (exit !=0)| +-----------+ +-----------+ Other statuses: - CrashLoopBackOff: Container repeatedly crashing - ImagePullBackOff: Cannot pull container image - ErrImagePull: Image name/tag incorrect

Phase Details

PhaseDescriptionCommon Causes
PendingAccepted by the cluster but not yet runningImage pull, scheduling, resource constraints
RunningBound to a node, at least one container is aliveNormal operation
SucceededAll containers exited with code 0Jobs, init containers
FailedAt least one container exited non-zeroApp error, OOM kill, misconfiguration
UnknownState cannot be determinedNode communication failure

Multi-Container Pods

Multi-container Pods are a core Kubernetes pattern. The CKA exam tests your understanding of when and how to use them. All containers in a Pod share the same network namespace and can mount shared volumes.

PatternPurposeExample
SidecarExtends and enhances the main containerLog shipper, service mesh proxy (Envoy)
AmbassadorProxies network connections for the main containerLocal proxy to a remote database cluster
AdapterStandardizes output from the main containerReformatting logs for a centralized logging system
Init ContainerRuns once before the main containers startSchema migration, config download, permission setup
yaml 
# Sidecar pattern: main app + log shipper
apiVersion: v1
kind: Pod
metadata:
  name: web-with-logging
spec:
  containers:
  - name: web
    image: nginx:1.27
    volumeMounts:
    - name: shared-logs
      mountPath: /var/log/nginx
  - name: log-shipper
    image: fluent/fluent-bit:3.1
    volumeMounts:
    - name: shared-logs
      mountPath: /logs
      readOnly: true
  volumes:
  - name: shared-logs
    emptyDir: {}

Common Mistakes

MistakeWhy It FailsFix
Forgetting containerPortPort not documented (still works, but confusing)Always declare ports explicitly
No resource limitsPod can consume all node resources, get OOM killedSet both requests and limits
Wrong restartPolicyDefault is Always, Jobs need Never or OnFailureMatch policy to workload type
Image tag :latestUnpredictable, hard to debug, imagePullPolicy issuesPin specific version tags
⚠
Exam Warning: On the CKA, don't waste time writing YAML from scratch. Use kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml to generate a template, then edit it.

Hands-On Exercise

Complete these tasks to verify your understanding. Use a kind or minikube cluster, or launch the lab above for a pre-configured environment.

  • Create a pod named web running nginx:1.27 with resource limits
  • Verify the pod is running and check its IP address with kubectl get pod -o wide
  • Execute a command inside the pod: kubectl exec web -- curl localhost
  • View pod logs with kubectl logs web and kubectl logs web --previous
  • Create a multi-container pod with an nginx and a sidecar busybox container
  • Debug a failing pod: create one with image nginx:nonexistent and diagnose with describe
